# 购物模块数据模型与视图

## 0. 五 Tab 总览(顶层导航,与现状一致)

```
┌─────────────────────────────────────────────────────────────────┐
│                       购物模块                                  │
├──────┬──────────┬──────────┬──────────┬──────────────────────┤
│ 总览 │ 采购决策 │ 系统地图 │ 空间巡检 │ 阶段模板             │
└──────┴──────────┴──────────┴──────────┴──────────────────────┘
  ↓        ↓          ↓          ↓           ↓
跨模块   物品库     物品库     物品库      阶段 = 选物品 + 档位
总览     CRUD +     按系统     按空间      内部可切换"系统/空间"
(不动)   采购属性   分组视图   分组视图    两种视图
```

| Tab | 职责 | 数据源 |
|-----|------|--------|
| **总览** | 跨模块总览,不动 | — |
| **采购决策** | 物品库的主维护入口,聚焦"已有/待购"、价格、渠道等采购属性 | 物品库 |
| **系统地图** | 物品库的全量视图,按 `systemTags` group-by | 物品库 |
| **空间巡检** | 物品库的全量视图,按 `spaceTags` group-by | 物品库 |
| **阶段模板** | 把物品库的子集组合成"人生场景",每个物品配三档子级 | 阶段(引用物品) |

**核心理解**:
- **物品库是唯一数据源**,跨 Tab 共享
- 系统地图、空间巡检是物品库的两种 group-by **视图**(全量)
- 阶段模板是物品库的**组合**(子集 + 档位 + 视图切换)
- 采购决策是物品库的**维护页**(CRUD + 采购属性编辑)

```
│   │                          │                                  │

│   │  depreciation:  极快/较快/中等/慢/不折旧(可选)             │
│   │  priceRef:      入门价/甜蜜价/虚高价                        │
│   │  channels:      [京东, 山姆, 拼多多, ...]                   │
│   └─────────────────────────────────────────────                │
└─────────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────────┐
│  底层:标签字典(全局定义)                                      │
│  ───────────────────                                            │
│   SystemDefinition[]:网络系统 / 安防系统 / 清洁系统 / ...       │
│   SpaceDefinition[]: 客厅 / 厨房 / 卧室 / 卫生间 / 书房 / ...   │
└─────────────────────────────────────────────────────────────────┘
```

**关键约束**:

- 物品两层:`Item → ItemChild`,不允许更深嵌套
- 标签必填:`systemTags.length ≥ 1` 且 `spaceTags.length ≥ 1`
- 标签多选:一个物品可同时归属多个系统、多个空间
- 采购属性独立:`status`、`lifecycle`、`depreciation`、`priceRef`、`channels` 都是物品自带的元数据,**互不联动**,不影响视图分组

> - 两者分组逻辑完全一样(都用物品的 `systemTags` / `spaceTags`)

## 7. 编辑流程示例

```
场景一:在采购决策 Tab 新增物品
─────────────────────────────────
① 进入"采购决策" Tab → 点击 [+ 新增物品]
② 填写:
   - 名称:路由器
   - 子级:[千兆, 双频, WiFi6, 企业级]
   - systemTags(必填多选):[网络, 安防]
   - spaceTags(必填多选):[客厅, 书房]
   - status(必选):待购
   - lane(待购可选):等好价
   - lifecycle:耐用品
   - 价格参考:入门 ¥150 / 甜蜜 ¥300 / 虚高 ¥600
   - 渠道:[京东, 拼多多]
③ 保存
④ 自动同步:
   - 系统地图 Tab:"网络系统"和"安防系统"分组下都出现路由器
   - 空间巡检 Tab:"客厅"和"书房"分组下都出现路由器


场景二:在阶段模板里把已有物品配进阶段
─────────────────────────────────
① 进入"阶段模板" Tab → 点开"租房阶段"
② 选择视图(任选):空间巡检视图
③ 在"客厅"分组下点 [+ 添加物品] → 从物品库选"路由器"
④ 配置档位:最低=千兆 / 基础=双频 / 升级=[WiFi6, 企业级]
⑤ 保存后:
   - "书房"分组下立刻也出现路由器(同档位)
   - 切到"系统地图视图":"网络"和"安防"分组下都出现路由器(同档位)
⑥ 任何一处改档位,所有出现位置同步更新
```

## 8. 数据模型(TypeScript)

```ts
// ===== 标签字典(全局) =====
interface SystemDefinition {
  id: SystemId          // "网络" | "安防" | "清洁" | ...
  name: string          // "网络系统"
  icon?: string
}

interface SpaceDefinition {
  id: SpaceId           // "客厅" | "厨房" | ...
  name: string
  icon?: string
}

// ===== 物品库(原子,跨 Tab 共享) =====
interface Item {
  id: ItemId
  name: string                       // 如 "路由器"
  children: ItemChild[]              // 子级,允许为空(但目标场景一般 ≥1)

  // 分组标签(必填多选,决定视图分组)
  systemTags: SystemId[]             // 长度 ≥ 1
  spaceTags:  SpaceId[]              // 长度 ≥ 1

  // 采购属性(物品独立元数据,不联动,不参与视图分组)
  status:        "owned" | "wanted"      // 已有 / 待购,必选下拉
  lane?:         "now" | "wait" | "hold" // 立即买/等好价/先不买
                                         // 仅当 status=wanted 时有意义,可选
  lifecycle:     Lifecycle               // 消耗品/耐用品/工具/情感物
  depreciation?: Depreciation            // 极快/较快/中等/慢/不折旧(可选)
  priceRef?:     PriceReference          // 入门价/甜蜜价/虚高价
  channels?:     Channel[]               // 购买渠道(如京东、山姆)
}

interface ItemChild {
  id: ItemChildId
  name: string                       // 如 "WiFi6"
}

interface PriceReference {
  entryPrice:      number            // 入门价
  sweetSpotPrice:  number            // 甜蜜价
  overpayPrice:    number            // 虚高价
}

// ===== 阶段模板(组合) =====
interface StageTemplate {
  id: StageId
  name: string                       // 如 "租房阶段"
  items: StageItem[]                 // 扁平物品清单,视图层 group-by
}

interface StageItem {
  itemId: ItemId                     // 引用物品库
  tiers: {
    low:  ItemChildId[]              // 最低配置,可多选子级
    base: ItemChildId[]              // 基础配置
    up:   ItemChildId[]              // 升级配置
  }
}
```

### 视图渲染逻辑(group-by)

```ts
// 系统地图 Tab(全量视图)
function renderSystemViewGlobal(items: Item[], systems: SystemDefinition[]) {
  return systems.map(system => ({
    system,
    items: items.filter(i => i.systemTags.includes(system.id)),
  }))
}

// 阶段内 · 系统视图(只看该阶段物品)
function renderSystemViewInStage(stage: StageTemplate, items: Item[], systems: SystemDefinition[]) {
  const stageItems = stage.items.map(si => ({
    stageItem: si,
    item: items.find(i => i.id === si.itemId)!,
  }))
  return systems.map(system => ({
    system,
    entries: stageItems.filter(({ item }) => item.systemTags.includes(system.id)),
  }))
}

// 空间视图同理,把 systemTags 换成 spaceTags
```

## 9. 小结

| 概念 | 角色 | 在哪个 Tab 维护 / 查看 |
|------|------|---------------------|
| 物品库 | 原子数据,跨 Tab 共享 | **采购决策**(维护)/ 系统地图、空间巡检(查看) |
| 系统/空间标签 | 物品的分组维度 | 由独立的分类管理入口维护(系统/空间卡片) |
| 物品采购属性 | 物品自带的元数据,互不联动 | **采购决策** Tab |
| 阶段模板 | 物品库的组合 + 档位 + 视图切换 | **阶段模板** Tab |
| 视图分组 | `systemTags` / `spaceTags` 的 group-by 结果 | 系统地图 / 空间巡检 Tab + 阶段内切换 |

**最关键的设计选择**:
1. **物品两层**(类 → 子级):因为档位选的是子级,而分组是按类
2. **标签必填多选**:一物多分组是天然结果,不需要额外存储
3. **采购属性独立不联动**:status/lifecycle/depreciation/价格/渠道只是物品的元数据,不影响任何分组逻辑
