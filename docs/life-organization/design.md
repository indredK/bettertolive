# 生命整理模块设计文档
> 汇总生命整理模块的产品边界、信息架构、数据结构、Tab 设计和 V1 范围。

---

## 生命整理模块设计

### 生命整理模块设计文档

#### 1. 设计目标

生命整理模块不是一份遗书编辑器，也不是一个死亡主题页面。它是 BetterToLive 里用来安放重要交代、重要表达、人生回顾、未完成牵挂和纪念偏好的私人工作台。

它要帮助用户回答这些问题：

- 有哪些事情如果没有交代清楚，我会不安？
- 有哪些话如果一直没有说出来，我会可惜？
- 哪些内容只是写给自己整理，哪些内容将来需要交付给别人？
- 我想被怎样理解、怎样记住，也希望别人怎样继续生活？

模块气质应保持：

- 冷静
- 私密
- 克制
- 有尊严
- 不煽情
- 不催促

#### 2. 产品边界

##### 2.1 这个模块做什么

生命整理负责承接五类内容：

| 内容 | 核心目的 | 典型例子 |
|---|---|---|
| 重要交代 | 让关键事务可以被接手 | 文件线索、账号线索、保险信息、联系人、物品归属 |
| 留给某人的话 | 安放重要表达 | 感谢、道歉、解释、祝福、告别 |
| 人生回顾 | 留下对自己一生的理解 | 重要经历、珍视之物、想被记住的样子 |
| 未完成的事 | 承认仍然牵挂的事 | 想修复的关系、未完成的承诺、还没做的事 |
| 纪念偏好 | 说明希望别人如何对待离开 | 纪念方式、仪式边界、公开范围、不希望发生的安排 |

##### 2.2 这个模块不做什么

- 不提供法律遗嘱效力。
- 不替代保险、资产、医疗、法律文件管理。
- 不在 V1 做真实离世验证或自动交付。
- 不把沉重内容做成任务催办。
- 不默认把私密内容送入 AI 汇总或跨模块洞察。
- 不以“死亡想象”作为页面主叙事。

##### 2.3 命名约定

产品层统一叫“生命整理”。

当前前端已有代码使用 `legacy` / `Legacy*` 命名。为降低改造成本，开发期可以继续沿用内部命名：

| 层级 | 命名 |
|---|---|
| 产品展示 | 生命整理 |
| 前端路由 | `legacy` |
| 前端类型 | `LegacyItem`、`LegacyModuleData` |
| 后端表名 | `legacy_*` |
| 文档表达 | 生命整理，必要处注明 legacy |

后续如果要统一命名，可单独做一次重命名，不和本次模块设计混在一起。

#### 3. 顶层信息架构

生命整理模块建议采用 5 个顶层 Tab：

```text
┌─────────────────────────────────────────────────────────────────┐
│                         生命整理                                │
├────────┬──────────┬──────────┬──────────┬────────────────────┤
│ 总览   │ 条目库   │ 交付地图 │ 关系表达 │ 边界与信任         │
└────────┴──────────┴──────────┴──────────┴────────────────────┘
   ↓         ↓          ↓          ↓              ↓
完成度     唯一       按接收者    给某人        隐私、交付、
风险与     数据源     和可见      的话与        锁定、导出、
提醒       CRUD       条件分组    未完成牵挂    AI 参与边界
```

| Tab | 职责 | 数据源 |
|---|---|---|
| 总览 | 看整理覆盖度、关键缺口、重负荷内容、锁定内容和待澄清交付条件 | 条目库聚合 |
| 条目库 | 新增、编辑、查看所有生命整理条目，是唯一主维护入口 | `LegacyItem[]` |
| 交付地图 | 按接收者、可见时机、紧急度组织内容，回答“谁在什么条件下看到什么” | 条目库 group-by |
| 关系表达 | 聚焦“留给某人的话”和可转入关系模块的未完成表达 | 条目库 + 关系引用 |
| 边界与信任 | 管理隐私、锁定、导出、AI 参与、交付机制说明 | 信任边界配置 + 条目策略 |

核心约束：

- 条目库是唯一数据源。
- 交付地图、关系表达、总览都只是条目库的不同视图。
- 重要交代、信件、人生回顾、纪念偏好不要拆成多套独立数据。
- “交付条件”在 V1 只做描述和人工确认，不做自动触发承诺。

#### 5. 条目 5 维分类与 1 个评估属性

##### 5.1 五个分类维度

| 字段 | 它回答什么 | 取值 |
|---|---|---|
| `category` | 这份内容本质上是什么 | 重要交代 / 留给某人的话 / 人生回顾 / 未完成的事 / 纪念偏好 |
| `recipient` | 目标接收者是谁 | 特定的人 / 家人 / 朋友 / 公开 / 仅自己 |
| `urgency` | 有多需要优先被知道 | 关键信息 / 重要 / 锦上添花 / 可选 |
| `visibility` | 什么时候能看到 | 现在 / 某个时间后 / 我离世后 / 条件触发 / 永不交付 |
| `status` | 当前完成状态 | 草稿 / 基本完成 / 已完成 / 会持续更新 / 最终版 |

##### 5.2 一个评估属性

| 字段 | 它回答什么 | 取值 |
|---|---|---|
| `emotionalLoad` | 写或打开这一份需要多少心力 | 很重 / 中等 / 轻微 / 平静 |

`emotionalLoad` 不是第 6 个分类维度。它只评估单份内容的心理重量，不应该成为主导航或强筛选入口。

##### 5.3 维度使用原则

| 维度 | 用途 |
|---|---|
| `category` | 决定条目主要归属和默认表单提示 |
| `recipient` | 决定交付地图分组 |
| `urgency` | 决定总览里的关键缺口提示 |
| `visibility` | 决定是否需要交付条件说明 |
| `status` | 决定完成度、锁定状态和维护提醒 |
| `emotionalLoad` | 决定打开前是否温和确认、是否避免催促 |

#### 6. Tab 设计

#### 6.1 总览 Tab

总览不是情绪渲染页，而是一个冷静的整理仪表盘。

##### 需要展示

- 条目总数。
- 按 `category` 的覆盖情况。
- `urgency=关键信息` 且未完成的数量。
- `visibility=我离世后 / 条件触发` 但缺少 `deliveryCondition` 的数量。
- `status=最终版` 或 `isLocked=true` 的数量。
- `emotionalLoad=很重` 的数量。
- 最近更新的条目。
- 温和回看问题。

##### 不要展示

- 不展示全文。
- 不展示“你还有 X 条没完成”的焦虑式催促。
- 不自动聚合很私密的内容。
- 不把“很重”的内容推到显眼位置制造压力。

##### 总览线框

```text
┌───────────────────────────────────────────────────────────────┐
│ 生命整理总览                                                   │
│ 把重要的话、交代和边界慢慢安放好。                              │
├───────────────────────────────────────────────────────────────┤
│ [条目总数] [关键信息未完成] [交付条件待补] [最终版锁定]          │
├───────────────────────────────┬───────────────────────────────┤
│ 分类覆盖                       │ 需要确认的边界                 │
│ 重要交代 ███                   │ - 3 条离世后内容缺条件说明      │
│ 留给某人的话 ██                │ - 1 条最终版近期被解锁过        │
│ 人生回顾 █                     │ - 2 条排除 AI 汇总              │
├───────────────────────────────┴───────────────────────────────┤
│ 最近更新                                                        │
│ - 写给伴侣的一封信 · 已完成 · 很重                              │
│ - 重要账号和文件线索 · 会持续更新 · 轻微                         │
└───────────────────────────────────────────────────────────────┘
```

#### 6.2 条目库 Tab

条目库是唯一主维护入口，类似购物模块里的“物品库”。

##### 列表结构

```text
左侧：条目列表 + 搜索 + 轻筛选
右侧：条目详情 / 编辑入口
```

##### 列表卡片展示

- 标题。
- 分类。
- 接收者。
- 紧急度。
- 可见时机。
- 状态。
- 情感负荷。
- 更新时间。
- 是否锁定。

##### 详情展示

- 正文或内容预览。
- 5 维分类。
- 情感负荷。
- 交付条件。
- 关系引用。
- 标签。
- 回看提示。
- 保护策略。
- 更新时间和最终确认时间。

##### 编辑表单

表单建议分成双栏：

```text
左栏：正文内容
- 标题
- 摘要
- 正文
- 标签
- 回看提示

右栏：分类与保护
- category
- recipient / recipientName / relatedRelationshipId
- urgency
- visibility / deliveryCondition
- status
- emotionalLoad
- isLocked
- requiresSecondConfirm
- excludeFromAi
```

#### 6.3 交付地图 Tab

交付地图回答的是：

> 谁，会在什么条件下，看到哪些内容？

它不是自动交付系统，而是帮助用户检查交付意图是否清楚。

##### 分组方式

第一层按 `recipient` 分组：

- 特定的人。
- 家人。
- 朋友。
- 公开。
- 仅自己。

第二层按 `visibility` 分组：

- 现在。
- 某个时间后。
- 我离世后。
- 条件触发。
- 永不交付。

第三层按 `urgency` 排序：

- 关键信息优先。
- 重要其次。
- 锦上添花和可选靠后。

##### 交付地图线框

```text
┌───────────────────────────────────────────────────────────────┐
│ 交付地图                                                       │
├───────────────────────┬───────────────────────────────────────┤
│ 接收者                 │ 内容                                  │
│ - 指定联系人            │ 我离世后                              │
│ - 伴侣                 │ - 重要账号和文件线索 · 关键信息        │
│ - 家人                 │ - 纪念方式偏好 · 重要                  │
│ - 公开                 │                                       │
│ - 仅自己               │ 条件触发                              │
│                       │ - 给未来家人的一段说明 · 可选          │
└───────────────────────┴───────────────────────────────────────┘
```

##### 必须提示的风险

- `visibility=我离世后 / 条件触发` 但没有 `deliveryCondition`。
- `recipient=特定的人` 但没有 `recipientName` 或 `relatedRelationshipId`。
- `urgency=关键信息` 但 `status=草稿`。
- `status=最终版` 但未锁定。
- `visibility=现在` 且内容已完成，可以提示“这也许可以转入关系模块或直接表达”，但不能催促。

#### 6.4 关系表达 Tab

关系表达 Tab 聚焦两类内容：

- `category=留给某人的话`。
- `category=未完成的事` 且和某段关系有关。

它和关系模块的边界是：

| 内容 | 生命整理 | 关系模块 |
|---|---|---|
| 时间视角 | 面向未来、终点或长期安放 | 面向当下关系状态 |
| 交付意图 | 通常有明确可见时机 | 不一定要交付 |
| 文本气质 | 总结性、交代性、告别性 | 过程性、即时性、沟通草稿 |
| 典型动作 | 写好、锁定、说明交付条件 | 继续打磨、决定是否说出口 |

##### 页面职责

- 列出所有写给人的内容。
- 显示是否关联关系模块。
- 显示哪些内容可以趁现在表达。
- 显示哪些内容只是私人整理，不应被推送。
- 支持把关系模块的“想说的话”升级为生命整理条目。
- 支持把生命整理中 `visibility=现在` 的表达复制或转入关系模块。

V1 可以先只展示关联信息，不做真实跨模块写入。

#### 6.5 边界与信任 Tab

这个 Tab 负责让用户知道系统不会偷偷越界。

##### 需要承接的边界

| 边界 | 说明 |
|---|---|
| 私密确认 | 很重、锁定、仅自己内容打开前可二次确认 |
| AI 参与 | 条目可选择不参与 AI 汇总、跨模块洞察或自动摘要 |
| 导出范围 | 区分可交付内容、私人内容、仅本机内容 |
| 交付说明 | V1 只记录条件描述，不承诺自动交付 |
| 法律边界 | 明确不具备法律遗嘱效力 |
| 修改审计 | 最终版解锁、修改、重新锁定应保留记录 |

##### 默认策略

- `recipient=仅自己` 默认 `excludeFromAi=true`。
- `emotionalLoad=很重` 默认 `requiresSecondConfirm=true`。
- `status=最终版` 默认 `isLocked=true`。
- `visibility=我离世后 / 条件触发` 必须填写 `deliveryCondition` 才能标记为最终版。
- 任何内容都不默认公开。

#### 8. 状态与行为规则

##### 8.1 最终版锁定

当 `status=最终版`：

- 自动设置 `isLocked=true`。
- 记录 `finalizedAt`。
- 编辑前必须主动解锁。
- 解锁后修改，状态回退为 `基本完成` 或保持 `最终版待确认`。V1 可先回退为 `基本完成`。

##### 8.2 交付条件校验

当 `visibility=我离世后 / 条件触发 / 某个时间后`：

- 显示 `deliveryCondition` 字段。
- 如果状态要变成 `已完成 / 最终版`，建议填写 `deliveryCondition`。
- 如果 `urgency=关键信息`，必须填写 `deliveryCondition` 才能进入 `最终版`。

##### 8.3 情感负荷保护

当 `emotionalLoad=很重`：

- 打开全文前可二次确认。
- 不出现在催促型提醒。
- 总览只显示数量和温和说明。
- 不默认进入 AI 汇总。

##### 8.4 AI 参与边界

每条内容可有 `excludeFromAi`。

默认规则：

- `recipient=仅自己` 默认排除。
- `emotionalLoad=很重` 默认排除。
- `visibility=我离世后` 默认排除。
- 用户可以手动允许，但需要清楚提示。

#### 9. 与其他模块的关系

| 模块 | 关系 |
|---|---|
| 关系深化 | 生命整理可引用关系条目；关系模块里的“想说的话”可升级为生命整理条目 |
| 成长记忆 | 人生回顾可引用关键记忆，但生命整理关注“想留下什么理解” |
| 观念 | 人生回顾可能引用重要观念，说明自己相信什么 |
| 原则 | 纪念偏好和最后边界本质上是原则的延伸 |
| 购物 / 生活物品 | 重要交代可引用情感物、重要物品和物品归属 |
| 未来蓝图 | 有些“未完成的事”可能其实是未来方向，不一定属于生命整理 |

#### 10. 典型问题与视图回答

| 真实问题 | 用哪个视图回答 | 查询逻辑 |
|---|---|---|
| 我有哪些关键事还没交代？ | 总览 / 条目库 | `urgency=关键信息` 且 `status` 未完成 |
| 谁将来会看到哪些内容？ | 交付地图 | 按 `recipient + visibility` 分组 |
| 哪些话其实可以现在说？ | 关系表达 | `category=留给某人的话` 且 `visibility=现在` |
| 哪些内容还没有写清交付条件？ | 总览 / 边界与信任 | `visibility` 需要条件但 `deliveryCondition` 为空 |
| 哪些内容对我来说很重？ | 总览 / 条目库 | `emotionalLoad=很重`，只显示摘要 |
| 哪些内容已经不能随便改？ | 边界与信任 | `status=最终版` 或 `isLocked=true` |
| 我这一生最想留下什么理解？ | 条目库 | `category=人生回顾` |

#### 11. V1 范围

##### 必做

- 生命整理页面拆成 5 个 Tab。
- 条目库支持新增、编辑、删除、查看。
- 总览展示分类覆盖、关键缺口和边界风险。
- 交付地图按接收者和可见时机 group-by。
- 关系表达聚焦写给人的内容。
- 边界与信任展示隐私、AI、导出、法律边界。
- 保留当前 mock 数据，逐步升级字段。

##### 可选

- 最终版解锁审计。
- 从关系模块复制“想说的话”。
- 从购物模块引用情感物或物品归属。
- 导出某个接收者的交付清单。

##### 不做

- 不做真实死亡验证。
- 不做自动交付。
- 不做云同步。
- 不做多人授权访问。
- 不做法律文件生成。
- 不做复杂加密方案设计。

#### 12. 验收标准

当以下条件成立时，生命整理模块设计可以认为清晰：

- 实施者知道条目库是唯一数据源。
- 实施者知道 5 个 Tab 分别解决什么问题。
- 实施者不会把情感负荷误当成第 6 个分类维度。
- 实施者知道 V1 不承诺自动交付，只记录交付意图。
- 用户能看清“写了什么、给谁、什么时候可见、完成到哪、是否受保护”。
- 页面气质克制，不把沉重内容做成催促任务。
