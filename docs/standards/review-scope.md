# 代码审查收敛边界（review-scope）

> 用途：被要求"审查某模块有没有问题"时，用本文档划范围，避免无限挖出"理论上可以更好"的点。
>
> 与 `conventions.md` 分工：那份立"该怎么写"的规约；本份定"什么算 bug、何时停"。判"硬约定违反"时引用 `conventions.md` 对应条目，不重复其细节。
>
> 三个维度（后端/前端/动画）共用同一套三档分级，审查时按改动涉及的维度各取所需。

---

## 核心原则

AI 审查永远能找到"可以更好"的点，没有自然终点。**收敛靠提前定好"什么才算 bug"**。判断一个发现是否值得报，先问：

1. **会不会真的发生**？用户能否通过现有 UI/操作真的触发，还是纯假设场景。
2. **发生后影响多大**？数据丢失/错乱 > 决策误导 > 操作受阻 > 体验别扭 > 风格不一致。

只有"会发生 + 影响达到下方门槛"的才算需要修。

---

## 三档分级（三维度共用）

| 档位 | 含义 | 处理 |
|---|---|---|
| **第一档** | 数据丢失/损坏、功能阻断、崩溃白屏、契约漂移 | 无下限，必须找、必须修 |
| **第二档** | 影响真实决策/可见结果、违反硬约定 | 有阈值，达门槛才报 |
| **第三档** | 风格/偏好/微优化/假设性场景 | 划红线，**不报、不重复提**，归技术债清单 |

**结束条件**：当某轮审查在第一档 + 第二档下结果为 0（或全部已修），本轮即完成——**不为"显得认真"去凑第三档**。第三档红线的作用是抑制噪音，不是待办。

---

## 维度一：后端 / 数据逻辑（`src-tauri/`）

**第一档（必修）**
1. **数据丢失/损坏/孤立**——已保存数据被静默删除、覆盖、或变成 UI 访问不到的孤儿数据，无论概率多低都修。
2. **多步写缺事务**——违反 `conventions.md` B1（`write_tx` / `let mut conn`）。机械可查：搜连续 `conn.execute`、`let conn`（非 mut）后跟写操作。
3. **操作应成功却失败 / 应失败却成功**——合法操作被误拒，或非法数据被误放行进库。
4. **崩溃 / panic**——`unwrap`/`expect`/越界/除零等正常输入下可 panic 的代码（违反 B2）。
5. **类型契约漂移**——改了 command 没跑 `generate:bindings`、或 typecheck 报错。**跑命令兜底，不靠肉眼。**

**第二档（按是否影响真实决策）**
1. **统计/聚合数字算错**——仅当用户会拿它当决策依据时算（总览计数、汇总金额）；内部中间量口径差异但展示正确，不算。
2. **校验缺失致输入被静默丢弃**——填了却消失且无提示算；若保存后立即可见没存上（用户自己会发现），降级。
3. **状态/枚举语义不一致**——同一数据在不同位置判定逻辑不同得出不同结论算；仅文案不同不算。

**第三档（红线，不报）**
并发/多写者场景（单用户本地应用，乐观锁是同用户多窗口特例）· 死代码/废弃字段（归 cleanup 清单）· 纯 UI/文案细节 · 假设性扩展场景（现有 UI 产生不了的数据）· 架构级"理想化"重构建议。

## 维度二：前端 UI / 状态 / 数据流（`src/features/`）

**第一档（必修）**
1. **用户输入被静默丢弃**——提交后数据消失且无提示（过滤空值/非法值不告知，应至少 `toast.warning`）。
2. **提交了无效/不完整数据**——表单缺校验致非法数据写出（违反 `conventions.md` F3）。口径：**所有手写校验（含存量）一律照报**，要求迁移到 zod，不豁免。
3. **Query 缓存失效缺失/错误**——mutation 成功后没 `invalidateQueries` 致显示旧数据；或 mutation 失败 `onError` 无反馈；或半吊子乐观更新不回滚（违反 F4）。等同数据错乱。
4. **异步竞态 / 过期闭包**——`useEffect` 缺依赖或缺 cleanup 致泄漏/卸载后 setState；旧请求覆盖新结果；`setTimeout`/订阅没在 unmount 清理。`useEffect` 依赖警告**先靠 lint 兜底**，人眼只查异步有无 isMounted 保护。
5. **崩溃 / 白屏**——访问可能不存在的 `obj.a.b`；解构与类型不符；列表用 index 当 key 且会重排。路由/页面级缺 ErrorBoundary 致整应用白屏也属此档（违反 F8）。
6. **类型契约漂移**——引用了 `bindings.ts` 里不存在的字段（永远 undefined）。跑 typecheck 兜底。

**第二档（按是否影响真实使用）**
1. **状态/枚举语义不一致**——同一数据不同位置判定不同（如"已购/想要"列表与总览不符）；仅文案不同不算。
2. **i18n 缺失**——用户可见新文案没走 `t(...)`。算 bug，但"补一行"级、修复成本低，批量处理优先。
3. **可见展示错误**——日期/金额/数字格式化错、排序不符预期、空状态缺失致页面像坏了。仅"用户据此判断"或"看着像故障"时算。
4. **加载/错误态缺失**——Query 的 `isLoading`/`isError` 没处理致闪烁/白屏/卡旧数据。中型页面算，纯展示小组件降级。
5. **a11y 关键项**——交互元素缺 label、键盘不可达、对比度严重不足。开发期只追"完全无法键盘/读屏操作"的硬阻断。
6. **弹窗交互违反 F9**——页脚按钮构成/顺序错、删除按钮非 destructive 或在新建态出现、乐观关闭（不等结果就关）、保存按钮没绑 `isPending`+`canSubmit`、失败不保持打开。口径：**新写/改动的弹窗一律照报**；存量"删除确认/回车/自动聚焦/脏数据提示"等零实现项归 `conventions.md`「弹窗改造清单」统一跟踪，**不逐个当新 bug 反复提**（专项收口前）。

**第三档（红线，不报）**
组件拆分/抽象偏好 · 无实测依据的性能微优化（`useMemo`/`memo`）· 样式/视觉细节 · 假设性极端输入 · "TS 类型可以更严"（typecheck 过即可）· `isMock()` 分支完整性。

## 维度三：动画（`motion/react`）

**判断标准**：动画 bug 必须是"用户能察觉的故障感"，不是"主观不够精致"。

**第一档（必修）**
1. **动画致功能不可用/卡死**——元素停在中间态（`exit` 被打断残留 `opacity:0`/`transform`，看不到也点不到）；进场动画挡住/推走可交互元素；依赖的 state 不更新致 loading 永久旋转。
2. **闪烁/跳变/布局抖动**——key 不稳致 `AnimatePresence` 误判疯狂重进场；`layout` 动画与高度变化打架致列表跳动；初次挂载未设 `initial={false}` 致首屏全部"飞入"闪屏。
3. **未尊重 `useReducedMotion`**——新增位移/缩放/旋转无降级（违反 `conventions.md` F2）。纯 opacity 淡入淡出例外（`AnimatedVisibility` 已兜底）。
4. **内存泄漏/卸载后动画**——手写 `animate()`/`requestAnimationFrame` 没在 unmount 清理；动画回调对已卸载组件 setState。
5. **性能硬伤（可感知掉帧）**——动画驱动 `width`/`top`/`margin` 等触发 layout 的属性而非 `transform`/`opacity`，**仅在确实能感知卡顿时算**，否则降第三档。

**第二档（按是否违反约定）**
1. **硬编码动画参数**——就地写魔法 `duration`/`ease` 而项目有对应语义常量（违反 F2）；新增/改动要求复用，纯历史遗留降第三档。
2. **用了 `motion.div` 而非 `m`**——违反 F2，批量可改。
3. **时长明显不合理**——入退场 > 0.6s 显迟钝或 < 0.08s 等于没动画却仍重排。仅"明显偏离项目其它动画量级"时报。

**第三档（红线，不报）**
缓动曲线/时长主观微调 · "这里可以加个动画更生动"（缺动画不是 bug）· 动画编排/分层架构偏好 · 假设性极端场景。

## 审查流程与产出

1. **工具先行**——先跑 typecheck → lint（`--max-warnings=0`）→ 后端加 `check:rust`，能机械穷尽的（类型漂移、未用变量、hook 依赖警告）不用 AI 重复找。
2. **优先 diff 审查**——只审本次改动的文件 + 直接调用链，不每次整模块重扫（否则反复挖同一批第三档历史遗留）。
3. **前端按数据流追问**——顺着`用户操作 → 表单/state → mutation → 后端 → invalidate → query 重拉 → 重渲染`找断点，比逐文件读更易命中真 bug。
4. **全模块审计只在里程碑做一次**——某模块开发告一段落时全量审一遍，第三档统一归"已知限制"清单。
5. **结束条件**——见上方三档分级（第一+第二档为 0 即停）。

### 按模块产出 Bug 报告

完整审查产出到 `docs/bugs/<模块>/`，按维度分文件（仅审了哪些维度就产出哪些）：

```
docs/bugs/<模块>/
  ├─ bug-backend.md     # 维度一
  ├─ bug-frontend.md    # 维度二
  └─ bug-animation.md   # 维度三
```

每条记录：`### BUG-XX 标题` + `文件:行号` + `严重程度（🔴第一档/🟡第二档）` + `描述` + `复现步骤` + `修复建议`；文件末尾放汇总表 `ID | 严重程度 | 简述 | 文件`。第三档"已知限制"单独归档，不混进 bug 报告。

### 给 AI 的下达话术（可复制）

> 审查 `<模块>` 的 **<后端/前端/动画>**。读 `docs/standards/review-scope.md`（对应维度）+ `conventions.md`，严格按三档分级，只报第一档+第二档，第三档归技术债。产出到 `docs/bugs/<模块>/bug-<维度>.md`，含复现步骤、修复建议、汇总表。第一+第二档为 0 即停，不要凑数。
