# 记账模块设计文档
> 汇总记账模块的产品定位、数据维度、页面结构、编辑流程和样本数据边界。

---

## 1. 模块定位

记账模块不是完整财务管理系统，也不是报税、投资或企业账务工具。

V1 的目标是让用户看见：

- 最近收入和支出如何流动
- 哪些消费类别正在塑造真实生活
- 钱是否流向了稳定生活、身体恢复、关系、成长或冲动消耗
- 哪些账目需要复盘，而不是只看总额

因此本模块的核心表达是：

> 金钱记录是一种生活证据。它帮助用户看清“我实际上正在怎样生活”。

---

## 2. V1 数据边界

### 2.1 必填字段

| 字段 | 说明 | 示例 |
|---|---|---|
| `id` | 账目唯一 ID | `finance-entry-001` |
| `date` | 发生日期，使用 `YYYY-MM-DD` | `2026-06-06` |
| `label` | 账目标题 | `午饭` |
| `category` | 类别 | `餐饮` |
| `amount` | 金额，正数存储 | `28` |
| `direction` | 收入或支出 | `income` / `expense` |
| `note` | 生活语境说明 | `赶工日的基础补给。` |

### 2.2 可选评估字段

| 字段 | 说明 | 示例 |
|---|---|---|
| `account` | 钱从哪个账户进出 | `银行卡` |
| `lifeSystem` | 这笔钱主要支撑哪个生活面 | `基本生活` |
| `necessity` | 必要性/价值判断 | `稳定维护` |
| `reviewStatus` | 后续是否需要复盘 | `待复盘` |
| `linkedModule` | 来源或关联模块 | `购物` / `饮食` / `手动录入` |
| `tags` | 辅助检索标签 | `["通勤", "恢复"]` |

这些字段不把模块推成复杂财务系统，而是给“这笔钱说明了什么”提供更清楚的上下文。

## 3. 分类建议

### 3.1 类别 category

V1 标准类别：

| 类别 | 含义 |
|---|---|
| 餐饮 | 三餐、咖啡、饮料、外卖 |
| 居住 | 房租、水电、家政、空间维护 |
| 交通 | 通勤、打车、公共交通 |
| 购物 | 生活物品、衣物、设备 |
| 学习成长 | 书、课程、工具订阅 |
| 健康 | 药品、运动、体检、护理 |
| 社交 | 约见、礼物、人情往来 |
| 娱乐 | 游戏、影音、演出、休闲 |
| 收入 | 工资、项目结算、退款等正向流入 |
| 储蓄 | 转入储蓄或长期账户 |
| 其他 | 暂时无法归类的账目 |

### 3.2 lifeSystem

`lifeSystem` 回答“这笔钱支撑了哪个生活系统”：

| 生活系统 | 说明 |
|---|---|
| 基本生活 | 吃饭、居住、通勤等基础运转 |
| 身体健康 | 身体维护、运动、医疗与恢复 |
| 关系社交 | 维系关系、见面、礼物 |
| 成长学习 | 学习、工具、长期能力 |
| 居住环境 | 空间舒适度、物品维护 |
| 自由安全 | 储蓄、保险、现金流安全感 |
| 娱乐恢复 | 娱乐、放松和情绪恢复 |

### 3.3 necessity

`necessity` 回答“这笔钱值不值得、属于哪种必要性”：

| 必要性 | 判断 |
|---|---|
| 生存必需 | 不花会影响基本生活 |
| 稳定维护 | 让生活维持稳定 |
| 体验改善 | 改善舒适度或效率 |
| 长期投资 | 对能力、健康或未来有长期价值 |
| 冲动/待复盘 | 当下可能合理，但需要回看 |

### 3.4 reviewStatus

`reviewStatus` 是单条账目的评估属性，不进入主分类：

| 状态 | 含义 |
|---|---|
| 已确认 | 这笔钱合理，不需要继续纠结 |
| 待复盘 | 需要回看背后的触发因素 |
| 可优化 | 不是错误，但可以换更好的方式 |
| 值得保留 | 这笔钱确实改善了生活 |

## 4. 页面结构

记账页保持一页响应式工作台，不拆成多页。

### 4.1 桌面固定布局

- 顶部：收入、支出、结余、日均支出摘要
- 中部：账目列表，支持管理模式下新增、编辑、删除
- 侧栏：类别分布、月度目标、复盘提示

### 4.2 移动/窄屏布局

- 摘要卡片先出现
- 账目列表居中展开
- 类别分布、目标和复盘提示依次堆叠

布局原则参考购物模块：

- 浏览模式只看数据和判断
- 管理模式才显示新增、编辑、删除入口
- 桌面尽量一屏显示主要信息
- 窄屏保持同一信息结构，只改排列方式

---

## 5. 后端与模拟数据边界

V1 不在前端页面内硬编码模拟账目。

数据来源统一走 `BetterToLiveApi`，Tauri 环境下由 Rust 后端提供：

- `get_finance`
- `save_finance`

样本数据放在：

- `src-tauri/src/finance/seed.json`

当前 seed 至少覆盖：

- 收入与支出
- 本周与本月账目
- 多个类别
- 多个 `lifeSystem`
- 多种 `reviewStatus`

这样既能让新安装时页面有足够内容验证结构，也不会让前端继续依赖独立 mock 数据。
