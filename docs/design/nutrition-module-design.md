# 饮食模块设计：糖类、蛋白质与常见食物语义建模

## 1. 文档目标

这份设计稿用于把 `nutrition` 模块从“能记热量”推进到“能解释日常吃法”。

目标不是做临床营养系统，而是把真正影响生活体验的结构化信号沉淀下来：

- 同样是碳水，白米饭、燕麦片、红薯、全糖奶茶、100% 果汁不能被当成一类。
- 同样写着蛋白质，鸡蛋、鸡胸肉、豆腐、酸奶、午餐肉的产品语义完全不同。
- 真正影响执行感受的，往往不是单一热量，而是游离糖、纤维、饱和脂肪、加工度、钠负担和进食场景。

本稿优先服务当前数据链路：

- `foods`
- `nutrientProfiles`
- `recipes`
- `dailyPlans`
- `mealLogs`

研究基线校验日期：`2026-06-23`

## 2. 范围与非目标

### 2.1 本次要解决什么

1. 为糖类、蛋白质、脂肪和关键微量营养建立更细的产品语义。
2. 用家常、外卖、便利店三类高频场景整理常见食物模板。
3. 让 `Foods / Nutrients / Daily Plan / Logs` 四个界面共享同一套底层判断。

### 2.2 本次不做什么

- 不做疾病诊断。
- 不输出医学处方或减肥处方。
- 不假装所有食物都有精确实验室数据。
- 不把用户吃饭行为道德化。

## 3. 官方研究基线与产品含义

以下基线来自官方公开营养指南。这里的作用不是把产品变成论文，而是防止底层分类做错方向。

### 3.1 碳水、膳食纤维与完整食物

WHO 当前公开建议可直接转成产品规则：

- 成人总能量中，碳水化合物通常可占 `45% - 75%`。
- 更推荐来自未精制谷物、蔬菜、水果和豆类的碳水。
- 10 岁以上人群建议膳食纤维至少 `25g/天`。
- 水果和蔬菜应足量，但其营养体验与果汁不同。

对产品的直接含义：

1. `carbG` 只能描述“量”，不能描述“质量”。
2. `fiberG` 不是附属字段，而是碳水质量、饱腹感和进食稳定性的关键解释字段。
3. 主食型碳水和甜饮型碳水必须在数据层拆开。
4. `foods` 层要让白米饭、杂粮饭、红薯、燕麦片这类食物显性化，而不是只留下“米面糖”。

### 3.2 糖的层级：总糖、天然糖、游离糖、添加糖

WHO 当前公开建议：

- 游离糖应少于总能量的 `10%`。
- 若进一步降到 `5%` 或以下，可能更有益。
- 游离糖不仅包括直接添加的糖，也包括蜂蜜、糖浆、果汁和果汁浓缩液中的糖。

这意味着产品不能只记录 `sugarG`，至少要分出下面三层：

1. `总糖 sugarG`：营养表里的总糖。
2. `添加糖 addedSugarG`：加工时额外加入的糖。
3. `糖的语义 sugarKind`：天然存在 / 游离糖或添加糖 / 混合 / 未知。

对常见食物的产品判断：

- 香蕉有糖，但通常属于“完整水果 + 钾来源”，不应被判成甜饮风险。
- 纯牛奶有乳糖，但它的主要产品语义更接近“蛋白 + 钙”。
- 100% 果汁虽然未必额外加糖，但在产品上更接近“游离糖风险”，不能和完整水果等价。
- 全糖奶茶、甜咖啡、含糖酸奶、风味乳饮则更接近“游离糖驱动的饮品或甜点场景”。

### 3.3 蛋白质：克数不等于角色

WHO 当前公开建议中，成人蛋白质通常占总能量 `10% - 15%` 已可满足基本需要；但对于产品设计，更重要的是“来源语义”和“伴随成本”。

只记录 `proteinG` 会产生两个问题：

1. 用户会把“有蛋白质”误解为“是优先蛋白来源”。
2. 推荐系统会把午餐肉、火腿肠、炸鸡、甜味酸奶和豆腐放得太近。

因此蛋白质至少要拆成四个判断维度：

1. `数量`：是否足以成为一餐中的蛋白主轴。
2. `来源`：蛋类、奶类、豆制品、鱼虾海鲜、禽肉、畜肉、加工肉。
3. `伴随风险`：高钠、高游离糖、高饱和脂肪。
4. `加工度`：原食材、轻加工、中度加工、高度加工。

### 3.4 脂肪：不是越低越好，而是质量要分开

WHO 当前公开建议：

- 总脂肪一般不超过总能量 `30%`。
- 饱和脂肪不超过总能量 `10%`。
- 反式脂肪不超过总能量 `1%`。
- 更推荐不饱和脂肪来源，如鱼、坚果和植物油。

对产品的直接含义：

1. `fatG` 单独存在会误导用户。
2. 至少应补 `saturatedFatG`，并为 `transFatG` 预留字段。
3. 三文鱼、坚果和猪五花都可能“有脂肪”，但生活意义完全不同。
4. 油炸、酥皮、奶精、加工肉的风险不该被“总脂肪”冲淡。

### 3.5 钠、钾、钙、铁：真正会影响日常判断的微量维度

WHO 当前公开建议：

- 成人钠摄入应少于 `2g/天`，约等于食盐少于 `5g/天`。
- 钾达到 `3510mg/天` 可能有益。
- 大量钠来自加工食品和高钠调味品。

美国官方膳食指南公开资料中，`calcium`、`potassium`、`dietary fiber` 被视为普遍值得关注的膳食成分，铁也有明确的食物来源指导。

对产品的直接含义：

1. `sodiumMg` 不应只存在于营养表，它必须进入风险提示。
2. `potassiumMg` 能解释为什么土豆、香蕉、菠菜、豆类值得出现。
3. `calciumMg` 能让牛奶、无糖酸奶、强化豆奶、钙盐豆腐不被系统埋掉。
4. `ironMg` 能帮助区分牛肉、贝类、豆类、菠菜等常见补铁食物，但不能把任何绿叶菜直接神化为“高铁解决方案”。

## 4. 设计原则

### 4.1 不把食物道德化

模块应该提供结构化解释，而不是给食物贴“好 / 坏”。

正确表达：

- “这杯饮品游离糖偏高”
- “这个蛋白来源伴随较高钠负担”
- “这餐纤维偏弱，可能不耐饿”

错误表达：

- “这是垃圾食品”
- “这顿吃坏了”

### 4.2 数量、质量、场景分开

同一种食物在不同场景下意义不同：

- 香蕉可以是办公室加餐，也可以是运动后补给。
- 酱油不是热量主角，却可能是整餐钠负担的关键来源。
- 奶茶主要不是“饮品”，而是“高游离糖场景”。

因此：

- 食品层负责营养事实与语义标签。
- 食谱层负责组合方式。
- 计划层负责一天结构。
- 日志层负责场景、诱因与身体反馈。

### 4.3 高频生活食物优先于冷门超级食物

第一批必须围绕真实生活：

- 家常主食
- 常见蛋白
- 高频蔬果
- 便利店和外卖风险食品
- 常见调味料

先把 20 - 40 个高频食品做对，比做 500 个长尾食品更有价值。

### 4.4 缺失值不能伪装成完整值

继续沿用当前正确原则：

- 缺失不按 `0` 算
- 缺失要可见
- 推荐逻辑不能对缺失字段过度自信
- 数据置信度要可追踪

## 5. 细致营养调查：结合常见食物的产品总结

以下分层是产品归纳，不是临床定级。目的只有一个：让系统更接近日常吃饭决策。

### 5.1 主食与碳水：真正要分的是“主食型”还是“甜饮型”

| 食物 | 高频场景 | 产品语义 | 需要重点提示 | 建议标签 |
| --- | --- | --- | --- | --- |
| 白米饭 | 家常、盖饭、便当 | 稳定主食型碳水 | 纤维通常偏低，容易和高油高钠配菜绑定 | `主食` `精制谷物` |
| 杂粮饭 | 家常、备餐 | 主食 + 更高纤维 | 不是“减肥符号”，而是更稳的主食结构 | `主食` `全谷/杂粮` |
| 燕麦片 | 早餐、加餐 | 主食 + 纤维候选 | 原味和调味即食燕麦必须分开 | `主食` `高纤维候选` |
| 红薯 | 早餐、替代主食 | 碳水 + 纤维 + 钾 | 不是“零碳水”，但通常比甜饮更稳 | `薯类` `高钾候选` |
| 挂面 / 汤面 | 快手正餐 | 主食型碳水 | 汤底和调味常带来额外钠负担 | `主食` `快手` |
| 玉米 | 早餐、家常 | 主食 + 纤维 | 适合作为稳态主食的一部分 | `主食` `高纤维候选` |
| 面包 / 甜面包 | 早餐、便利店 | 主食或甜点，差异极大 | 原味全麦和高糖夹馅不能放一起 | `主食` `甜点风险` |

核心结论：

1. 用户需要的是“这是不是一份像样的主食”，不是抽象的 `carbG`。
2. 主食型碳水更适合和蛋白、蔬菜一起看。
3. 甜饮和甜点里的糖，虽然也属于碳水，但在产品上不该和米饭面条归为一类。

### 5.2 糖类：真正需要分清的不是“有没有糖”，而是“糖来自哪里”

| 食物 | 总糖语义 | 产品语义 | 为什么不能混为一谈 |
| --- | --- | --- | --- |
| 香蕉 / 苹果 | 天然糖 | 完整水果 | 有糖，但伴随完整咀嚼、体积感和部分纤维 |
| 纯牛奶 | 天然糖 | 蛋白 + 钙来源 | 乳糖存在，但主要角色不是甜饮 |
| 无糖酸奶 | 低游离糖 | 蛋白 + 钙来源 | 要和风味高糖酸奶分开 |
| 100% 果汁 | 游离糖风险 | 甜饮型糖 | 即便未额外加糖，也不能和完整水果等价 |
| 全糖奶茶 | 游离糖 / 添加糖 | 高糖饮品 | 常同时伴随奶精、脂肪和体积大 |
| 甜咖啡 / 乳饮 | 游离糖 / 添加糖 | 高糖饮品 | 甜味来自额外添加，不应伪装成“提神饮品” |
| 蜂蜜水 / 糖浆饮品 | 游离糖 | 高糖饮品 | 虽然常被包装成“自然”，但产品意义仍是高糖来源 |

核心结论：

1. `sugarG` 不足以支撑产品判断。
2. `sugarKind` 是必要字段，否则水果、果汁、奶茶会被误判过近。
3. `addedSugarG` 让系统能快速识别“加工时额外堆进去的甜味”。

### 5.3 蛋白质：真正有用的是“蛋白主轴”而不是“蛋白痕迹”

| 食物 | 高频场景 | 蛋白语义 | 伴随成本 | 建议标签 |
| --- | --- | --- | --- | --- |
| 鸡蛋 | 早餐、加菜 | 稳定蛋类蛋白 | 份量建模要稳定，以“个”为核心 | `蛋类蛋白` |
| 鸡胸肉 | 备餐、轻食 | 高密度禽肉蛋白 | 做法差异大，炸鸡不能混入 | `禽肉蛋白` |
| 鱼 / 虾 | 家常、外食 | 海鲜蛋白 | 调味和烹调法影响钠脂肪 | `海鲜蛋白` |
| 北豆腐 / 老豆腐 | 家常、素食 | 豆制品蛋白 | 是否高钙取决于凝固剂与来源 | `豆制品蛋白` |
| 毛豆 / 豆类 | 家常、加餐 | 植物蛋白 + 纤维 | 更接近“豆类食物”，不是代餐粉 | `豆类蛋白` `高纤维候选` |
| 纯牛奶 / 无糖酸奶 | 早餐、加餐 | 奶类蛋白 + 钙 | 风味高糖版本要单独建模 | `奶类蛋白` `高钙候选` |
| 无糖豆奶 | 早餐、便利店 | 豆制品蛋白 | 强化钙与否影响很大 | `饮品蛋白` `高钙候选` |
| 午餐肉 / 火腿肠 | 火锅、应急餐 | 加工肉蛋白 | 高钠、加工度高，不能当优先蛋白来源 | `加工肉` `高钠风险` |

核心结论：

1. “有蛋白”不等于“值得系统优先推荐”。
2. `proteinSource` 是必要字段，否则推荐系统会鼓励错误替代。
3. 蛋白来源必须和 `processingLevel`、`sodiumRiskLevel` 联动。

### 5.4 脂肪与加工度：总脂肪会掩盖真正的问题

| 食物 | 常见误判 | 实际应如何理解 |
| --- | --- | --- |
| 三文鱼 / 鲭鱼 | “脂肪高” | 更应理解为海鲜蛋白 + 不错的脂肪质量 |
| 坚果 | “热量高” | 少量使用时更像脂肪质量和饱腹辅助 |
| 猪五花 | “和鱼一样只是脂肪高” | 饱和脂肪占比与吃法语义不同 |
| 炸鸡 | “只是蛋白质来源” | 蛋白存在，但加工与油炸会改变产品定位 |
| 方便面 | “只是主食” | 主食 + 高钠 + 高加工度，风险主要不止来自面饼 |

核心结论：

1. 系统不该用低脂/高脂做粗暴标签。
2. 更应该用 `saturatedFatG`、`processingLevel` 和 `sodiumRiskLevel` 解释。

### 5.5 钠风险：真正危险的常常不是“咸口食物”，而是调味体系

| 食物 / 场景 | 为什么要单独建模 | 建议标签 |
| --- | --- | --- |
| 酱油 | 热量不高，但可能显著抬高整餐钠负担 | `调味料` `隐性高钠` |
| 咸菜 / 榨菜 | 体积小但钠密度高 | `腌制品` `高钠风险` |
| 方便面调味包 | 高钠通常来自调味包，不只是面体 | `加工主食` `高钠风险` |
| 午餐肉 / 火腿肠 | “蛋白”外壳下常有更高钠负担 | `加工肉` `高钠风险` |
| 外卖勾芡汁 / 卤汁 / 拌饭汁 | 用户常忽略，但能决定整餐压力 | `调味汁` `隐性高钠` |

核心结论：

1. 如果没有 `sodiumRiskLevel`，系统会低估酱油和复合调味汁的重要性。
2. 日计划和日志都需要能指出“高钠不是只来自肉，也来自调味”。

### 5.6 钙、铁、钾：常见食物里最值得显性的微量维度

| 营养维度 | 值得显性的常见食物 | 产品作用 |
| --- | --- | --- |
| 钙 | 牛奶、无糖酸奶、强化豆奶、部分豆腐 | 让早餐和加餐推荐更有依据 |
| 铁 | 牛肉、贝类、部分豆类、部分绿叶菜 | 为“补铁食物”提供更稳的对比，而不是口号 |
| 钾 | 红薯、土豆、香蕉、番茄、菠菜、豆类 | 帮助系统解释“为什么这些食物值得补进来” |
| 纤维 | 杂粮饭、燕麦、豆类、绿叶菜、西兰花、水果 | 直接影响饱腹和结构稳定性 |

核心结论：

1. 这几项不是为了堆字段，而是为了让推荐像真实生活。
2. 对大多数用户来说，钙、钾、纤维比“更精确的热量”更能解释吃后的体验差异。

### 5.7 容易被错误归类的高频对比

| 容易被写成一类 | 实际应拆成什么 | 原因 |
| --- | --- | --- |
| 香蕉 vs 100% 果汁 | 完整水果 vs 游离糖饮品 | 糖的结构和进食体验不同 |
| 无糖酸奶 vs 风味酸奶 | 蛋白/钙来源 vs 高糖乳制甜点 | 游离糖压力完全不同 |
| 挂面 vs 方便面 | 主食型碳水 vs 高加工高钠快餐 | 风险不在“面”本身，而在加工和调味 |
| 豆腐 vs 午餐肉 | 豆制品蛋白 vs 加工肉蛋白 | 推荐语义完全不同 |
| 酱油 vs 食用油 | 钠风险调味料 vs 脂肪来源 | 影响维度不同，不能都塞进“调味品” |

## 6. 数据模型扩展建议

### 6.1 保持现有链路，做增量扩展

不推翻 `FoodNutrientProfile`，只补能显著提高解释力的字段：

```ts
export type SugarKind = "天然存在" | "游离糖/添加糖" | "混合" | "未知"

export type ProteinSourceKind =
  | "蛋类"
  | "奶类"
  | "豆制品"
  | "豆类"
  | "鱼虾海鲜"
  | "禽肉"
  | "畜肉"
  | "加工肉"

export type ProcessingLevel =
  | "原食材"
  | "轻加工"
  | "中度加工"
  | "高度加工"

export type SodiumRiskLevel = "低" | "中" | "高"

export type FoodPortionProfile = {
  unit: "个" | "份"
  estimatedGrams: number
  note?: string
}

export type FoodSourceRef = {
  id: string
  label: string
  url?: string
  sourceType: "包装" | "成分表" | "官方数据库" | "手动估算"
  checkedAt: string
}

export type FoodNutrientProfile = {
  id: string
  foodId: string
  basisAmount: number
  basisUnit: "g" | "ml"
  energyKcal?: number
  proteinG?: number
  fatG?: number
  saturatedFatG?: number
  transFatG?: number
  carbG?: number
  fiberG?: number
  sugarG?: number
  addedSugarG?: number
  sodiumMg?: number
  calciumMg?: number
  ironMg?: number
  potassiumMg?: number
  sugarKind?: SugarKind
  proteinSource?: ProteinSourceKind
  processingLevel?: ProcessingLevel
  sodiumRiskLevel?: SodiumRiskLevel
  portionProfiles?: FoodPortionProfile[]
  sourceRefs?: FoodSourceRef[]
  source: "手动" | "包装" | "食物成分表" | "外部导入"
  confidence: "高" | "中" | "低"
}
```

### 6.2 优先级排序

最值得优先落地的字段：

1. `addedSugarG`
2. `sugarKind`
3. `proteinSource`
4. `processingLevel`
5. `sodiumRiskLevel`
6. `saturatedFatG`
7. `calciumMg`
8. `ironMg`
9. `potassiumMg`
10. `portionProfiles`

原因：

- 这组字段能立刻拉开水果、果汁、奶茶、豆腐、午餐肉、酱油、方便面之间的差异。
- 它们对推荐逻辑和 UI 标签的解释力远高于继续追加边缘字段。

### 6.3 显示层启发式阈值

这些阈值只用于产品语义，不是医学诊断：

| 标签 | 启发式阈值（以基础单位为准） | 说明 |
| --- | --- | --- |
| 高蛋白 | `proteinG >= 12` | 适合作为一份显著蛋白来源 |
| 高纤维 | `fiberG >= 3` | 用于快速识别更稳的主食/蔬果 |
| 高钙 | `calciumMg >= 100` | 足以让奶类/强化豆奶/豆腐显性 |
| 高铁 | `ironMg >= 2` | 帮助常见补铁食物浮现 |
| 高钾 | `potassiumMg >= 300` | 让红薯、香蕉、菠菜、豆类被看见 |
| 高糖风险 | `sugarKind=游离糖/添加糖` 或 `addedSugarG >= 8` | 识别甜饮和高糖加工品 |
| 高钠风险 | `sodiumRiskLevel=高` 或 `sodiumMg >= 600` | 识别方便面、午餐肉、酱料等 |
| 高饱和脂肪风险 | `saturatedFatG >= 4` | 避免只看总脂肪 |

## 7. 常见食物 seed 策略

### 7.1 第一批必须覆盖的高频食物

优先做 20 - 30 个食物模板，覆盖三类真实生活场景：

1. 家常主食
   - 白米饭、杂粮饭、燕麦片、挂面、红薯、玉米
2. 稳定蛋白
   - 鸡蛋、鸡胸肉、鱼、虾、牛奶、无糖酸奶、无糖豆奶、北豆腐、毛豆
3. 高频蔬果
   - 番茄、青菜、西兰花、菠菜/小白菜、胡萝卜、香蕉、苹果
4. 风险食物与调味
   - 全糖奶茶、100% 果汁、方便面、午餐肉、火腿肠、酱油、咸菜

### 7.2 数据来源优先级

建议 `sourceRefs` 按以下顺序记录：

1. 包装营养成分表
2. 官方食物数据库
3. 国家或地区食物成分表
4. 手动估算

对于用户最常录入的便利店和包装食品，包装数据通常比通用数据库更可靠。

### 7.3 份量建模策略

高频食品不能只依赖 `g/ml`：

- 鸡蛋：`1 个 ≈ 50g`
- 香蕉：`1 根 ≈ 100g - 120g（去皮可另记）`
- 酸奶：`1 杯 / 1 盒`
- 牛奶 / 豆奶：`1 盒 / 1 瓶`
- 便利店饭团：`1 份`

`portionProfiles` 的价值不在于绝对精确，而在于减少录入漂移。

## 8. 界面与交互落地

### 8.1 Foods Tab

新增四组语义筛选与徽标：

- 营养主轴：`高蛋白` `高纤维` `高钙` `高铁` `高钾`
- 风险提示：`高糖风险` `高钠风险` `高饱和脂肪风险`
- 来源语义：`蛋类` `奶类` `豆制品` `鱼虾` `禽肉` `畜肉`
- 加工度：`原食材` `轻加工` `高度加工`

这样用户可以直接回答生活问题：

- “我家里有哪些真正的蛋白来源？”
- “哪些东西是高钠风险？”
- “我不是没有食物，我是缺高纤维和高钾的选择。”

### 8.2 Nutrients Tab

从单纯数字表升级成双层视图：

1. 第一层：宏量营养总览  
   `能量 / 蛋白 / 脂肪 / 碳水 / 纤维 / 糖 / 钠`

2. 第二层：质量与微量维度  
   `添加糖 / 饱和脂肪 / 钙 / 铁 / 钾 / 蛋白来源 / 加工度`

### 8.3 Daily Plan Tab

每日计划不应只显示 `kcal` 和 `protein`。至少要能回答：

1. 今天有没有真正的蛋白主轴？
2. 今天有没有蔬菜和纤维来源？
3. 今天是否被甜饮或高钠调味主导？

示例：

- 早餐：燕麦片 + 无糖酸奶 + 香蕉  
  可提示“纤维和钾较完整，蛋白中等。”

- 午餐：白米饭 + 外卖红烧肉 + 全糖奶茶  
  可提示“游离糖和钠压力偏高，纤维与蔬菜偏弱。”

### 8.4 Meal Logs

日志已记录 `scene / trigger / bodyFeedback`，下一步应做关联解释：

- 高糖饮品是否更常出现在 `情绪驱动`
- 高钠外卖后是否更常出现 `偏重偏胀`
- 晚间甜饮或咖啡因是否与次日反馈重复出现

`mealLogs` 的价值不是回忆录，而是行为-营养-体感的关联线索。

## 10. 分阶段实施建议

### Phase 1：补数据结构

- 扩 `FoodNutrientProfile`
- 引入 `portionProfiles`
- 为每条关键营养值保留来源与置信度

### Phase 2：补高频食物模板

- 先录 20 - 30 个高频食品
- 先保证字段质量与语义一致
- 不急着追求海量

### Phase 3：补界面语义

- Foods Tab 语义筛选和徽标
- Nutrients Tab 双层视图
- Daily Plan 的蛋白 / 纤维 / 高糖 / 高钠提示

### Phase 4：补行为关联

- 将 `mealLogs` 与营养信号关联
- 在周回顾中输出更像生活反馈的洞察

## 11. 结论

饮食模块下一步不该只做“多几个字段”，而应完成三件事：

1. 把糖类拆细，至少分清总糖、添加糖、游离糖风险与完整水果语义。
2. 把蛋白质做实，至少分清来源、加工度和伴随的钠/脂肪成本。
3. 把常见食物放回真实生活，围绕家常、外卖、便利店三种高频场景建模。

只要这三件事做对，`nutrition` 模块就会从“能记”升级到“能解释”，后面的计划、日志和周回顾才有坚实底层。
