# BetterToLive 产品需求文档（V1）

## 1. 文档目的

本文档用于定义 BetterToLive V1 的产品方向与需求边界，重点回答以下四个问题：

1. 这个产品是什么
2. 它为什么存在
3. V1 要做什么
4. V1 暂时不做什么

本文档不讨论技术实现细节，不展开数据库设计，也不作为对外宣传材料使用。它服务于当前产品思考与后续实现落地。

## 2. 产品定义

### 2.1 产品愿景

BetterToLive 是一个帮助人看见自己、整理人生、靠近日常理想状态的私人空间。

它不是一个单纯的工具集合，也不是一个只用来写情绪日记的产品。它的目标是把记录、整理、回看、描画未来放在同一条主线上，让用户逐渐回答这些问题：

- 我过去是怎么走到这里的
- 我现在到底在过什么样的生活
- 我想去哪里，想成为什么样的人

### 2.2 产品定位

BetterToLive 的统一定位是：

**一个本地优先、单人使用、以反思记录为主要入口的自我认知系统桌面 app。**

它通过 `过去 / 现在 / 未来` 的框架，帮助用户理解自己的人生轨迹、当下状态和未来方向。

### 2.3 为什么现在做

很多生活内容虽然被记录了下来，但往往是分散的、断裂的、难以回看的。消费记录、待办事项、随手记下的想法、某天晚上的反思，通常各自存在，却很少被放在同一个生命视角里理解。

BetterToLive 想做的事，是把这些零散的内容重新组织起来，让记录不只是堆积，而是逐渐帮助用户认识自己。

## 3. 要解决的问题

当前产品需要解决的问题主要有以下几类：

- 生活记录分散，难以形成连续的自我认知
- 人往往只在情绪高点或低点时才思考自己，缺少稳定的回看机制
- 现实行为和理想生活之间缺少连接，难以判断自己是否正在靠近真正想要的状态
- 日常工具很多，但很少有工具能把“我是谁”“我在追求什么”“我实际如何生活”放在一起看

## 4. 目标用户

V1 只定义一个目标用户：产品创建者本人。

当前目标用户画像如下：

- 愿意长期记录自己的生活和想法
- 愿意回看过去，观察自己的变化
- 关心自我成长、人生方向和生活方式
- 不需要社交表达，更需要一个私密、稳定、可持续使用的空间

虽然 V1 只服务于个人，但产品表达应保留一定通用性，使其未来可以自然扩展到类似人群。

## 5. 产品原则

BetterToLive V1 采用以下产品原则：

- **本地优先**：默认以本地使用为前提，不依赖在线服务才能成立
- **隐私优先**：记录内容默认为私人内容，不围绕分享或公开传播设计
- **低负担记录**：输入门槛要低，用户不需要每次都进行完整复盘
- **自由切换入口**：不强制规定唯一主流程，用户可以按当下需要进入任意模块
- **不锁定使用模式**：用户可以在不同入口、不同记录方式之间来回切换
- **先收集整理，再做洞察**：V1 先解决记录、归档、回看问题，不以自动分析能力为中心
- **所有模块服务于同一主线**：每个模块都应帮助用户更了解自己，而不是单独存在

## 6. 核心理解框架：过去 / 现在 / 未来

BetterToLive 不是按工具类别组织自己，而是按用户对人生的理解过程组织自己。

### 6.1 过去

过去关注的是：

- 发生过什么
- 我曾经怎样想
- 我曾经怎样生活
- 某些变化是如何逐渐发生的

对应的主要内容来源包括：反思、记事、记账、已完成或已发生的清单记录。

### 6.2 现在

现在关注的是：

- 我此刻正在经历什么
- 我把时间、精力和金钱投向了什么
- 我现在的生活状态是否接近我真正想要的样子

对应的主要内容来源包括：近期反思、近期事件、近期消费、当前待办或待买事项。

### 6.3 未来

未来关注的是：

- 我想成为谁
- 我想过怎样的生活
- 什么对我真正重要
- 接下来我想朝哪些方向靠近

这里的“未来”不是预测，而是对理想自我、理想生活和行动方向的定义。

## 7. 使用方式与整体体验

V1 的整体体验应体现为一个可以自由进入和切换的私人空间，而不是固定流程工具。

核心约定如下：

- 首页展示全部核心入口
- 用户进入 app 后，不需要先完成某个步骤，才能使用其他功能
- 用户可以先写反思，也可以先记账、记事、写购物清单或打开未来蓝图
- 用户在选择某种使用模式后，仍然可以随时切换到其他模块

这意味着首页是一个总览入口和切换中枢，而不是一个要求用户按顺序操作的任务面板。

## 8. V1 信息架构

V1 的结构分为 `1 个总览首页 + 5 个核心功能模块`。

### 8.1 总览首页

总览首页不是独立业务模块，而是承接全局体验的入口层。它负责：

- 展示全部核心功能入口
- 帮助用户快速决定此刻想做什么
- 承接“过去 / 现在 / 未来”的整体视角
- 提供模块之间的切换能力

### 8.2 五个核心功能模块

1. 反思
2. 记事
3. 记账
4. 购物清单
5. 未来蓝图

## 9. V1 模块需求

### 9.1 反思

**定位**

反思是 V1 最核心的主入口，用来承接情绪、想法、困惑、总结和阶段性复盘。

**V1 必备能力**

- 创建一条反思记录
- 支持自由书写
- 能看到记录时间
- 能被后续回看

**它解决的问题**

- 帮助用户表达当下状态
- 把模糊感受转成可被保留的文字
- 为后续理解自己留下连续轨迹

### 9.2 记事

**定位**

记事用于记录事件、经历、片段和念头，帮助形成个人时间线。

**V1 必备能力**

- 创建一条事件记录
- 记录发生时间或记录时间
- 用简洁方式保存内容
- 后续可以按时间回看

**它解决的问题**

- 让生活不只剩下情绪和消费记录
- 帮助用户记住“发生了什么”
- 为回顾人生阶段提供客观片段

### 9.3 记账

**定位**

记账不是为了做复杂财务管理，而是为了看见现实生活如何被消费行为塑造。

**V1 必备能力**

- 记录收入或支出
- 记录金额
- 记录类别
- 记录时间

**它解决的问题**

- 让金钱流向成为理解当下生活的重要线索
- 帮助用户观察消费习惯和价值取向
- 让“我实际上如何生活”变得更可见

### 9.4 购物清单

**定位**

购物清单用于记录想买、待买、已买事项，是日常欲望、计划和决策的轻量载体。

**V1 必备能力**

- 新建一条待买事项
- 标记状态，例如待买或已买
- 记录简单说明
- 后续可回看

**它解决的问题**

- 补足日常生活管理的现实场景
- 帮助用户看到自己的需求和欲望是如何变化的
- 为理解“我为什么想要这些东西”提供素材

### 9.5 未来蓝图

**定位**

未来蓝图用于记录理想自我、理想生活、重要价值和阶段方向，是产品中承接“未来”的核心模块。

**V1 必备能力**

- 记录想成为的人
- 记录想过的生活状态
- 记录重要价值和方向
- 记录阶段性的行动路径或靠近方式

**它解决的问题**

- 避免所有记录都只停留在过去和当下
- 为用户提供一个持续校准人生方向的地方
- 帮助用户把“想法”转成更清楚的自我定义

## 10. 模块关系

V1 的五个核心功能模块不是并列小工具，它们共同服务于三个核心问题：

1. 我过去是怎么走到这里的？
2. 我现在到底在过什么样的生活？
3. 我想去哪里，想成为什么样的人？

各模块之间的关系可以概括为：

- `反思` 负责表达主观感受和内在思考
- `记事` 负责保留客观经历和事件片段
- `记账` 负责呈现现实资源流向
- `购物清单` 负责呈现日常需求、欲望和决策
- `未来蓝图` 负责定义理想方向与想靠近的生活

这些内容在长期使用中应逐渐形成一个更完整的个人画像。

## 11. 术语与共享对象约定

本次文档不引入代码 API 变更，但需要统一产品层术语。

### 11.1 术语约定

- `记录`：泛指用户输入到系统中的一切内容
- `回看`：对过去记录进行浏览、翻阅和复盘
- `未来蓝图`：对理想自我和理想生活的定义，不代表预测结果
- `模式切换`：用户可在首页和各模块之间自由进入、自由切换，不受固定流程限制

### 11.2 共享对象建议

为了让不同模块未来可以被放回统一视角中理解，V1 建议各类记录默认具备以下基础属性：

- 时间
- 内容
- 标签或分类
- 状态（如适用）

后续实现时，应尽量让不同模块的记录能够被放回同一条时间线或同一组人生主题中理解，但 V1 不要求在产品文档中细化到数据库 schema。

## 12. MVP 范围

V1 采用“全部轻量纳入”的策略。

这意味着：

- 五个核心功能模块都会出现
- 每个模块都只要求形成最基础的记录与回看闭环
- 重点是建立统一产品心智，而不是先把某一个模块做深

V1 成立的标准，不是功能数量多，而是用户已经可以借由这些模块逐渐回答“我是谁、我正在怎样生活、我想成为什么样的人”。

## 13. 后续版本方向

以下内容进入后续版本路线，而不作为 V1 必做项：

- `性格测试`：作为辅助画像模块，用于补充自我认知，而不是主线入口
- `自动洞察`：例如趋势提炼、主题归纳、变化总结
- `周期报告`：例如周报、月报、阶段性过去/现在/未来对比
- `教练式提问`：通过连续问题帮助用户进一步思考
- `同步/导出`：例如多端同步、内容导出、阶段总结分享

## 14. 非目标

V1 明确不做以下方向：

- 不做社交产品
- 不做多人协作产品
- 不做公开社区
- 不把 V1 定义为 AI 对话产品
- 不把性格测试作为第一主线
- 不优先解决云同步、账号体系、分享机制等能力

## 15. 验收标准

当以下条件都成立时，可以认为这份 V1 产品定义是清晰的：

- 实施者读完文档后，不需要再判断它到底是日记 app、工具箱还是人生系统
- 实施者能明确总览首页只是入口层，五个核心功能模块才是业务主体
- 实施者能明确首页不应强制单一路径，而应支持自由进入与切换
- 实施者能明确 V1 的重点是记录与整理优先，而不是 AI 洞察优先
- 实施者能明确性格测试不属于首版核心范围
- 实施者能理解每个模块为什么存在，以及它们如何共同服务于“更了解自己”这条主线

## 16. 默认假设

- 文档默认服务于产品创建者本人和后续实现者
- 文风采用冷静、真诚、偏产品定义的写法
- 平台默认是当前仓库对应的 Tauri 桌面 app
- 默认使用场景是纯本地、私人使用
- 除非后续单独提出，本版本不补充云同步、账号体系、分享机制等额外约束
