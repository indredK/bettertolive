## 从设计文档提取的开发相关内容

### 9. 实施步骤（草稿）

1. 7 个领域 → `EconDomain` 枚举
2. 3 个层次 → `EconLayer` 枚举
3. 4 级掌握程度 → `EconConfidence` 枚举
4. 5 种来源 → `EconSource` 枚举
5. 4 级决策距离 → `EconRelevance` 枚举
6. `EconKnowledge` 类型上 5 个字段必填
7. 支持关联：相关概念、对立观点、上下游因果
8. 每条保留 confidence 的修订历史（"我什么时候从'听过名词'升到'知道逻辑'"，本身是重要的学习轨迹）

### 10. 还没想清楚 / 需要你定的问题

1. **领域切法**：7 个领域合不合适？特别是"房地产"要不要单独成域？
2. **模块命名**：用"社会经济"、"经济常识"、"经济认知"哪个更合适？文件名暂用 `socioeconomics-module.md`。
3. **知识 vs 笔记**：一条记录是只存"知识点 + 我的标签"，还是要额外有个"我对它的理解笔记"字段？后者更接近 Anki/Obsidian 的形态。
4. **跟新闻 / 阅读输入的关系**：要不要支持"从某篇文章/书摘录"的来源溯源？这会让 source 维度变成多对多。
5. **学习节奏**：要不要内建"复习提醒"（confidence=听过名词 的知识隔一段时间提醒补课）？这超出了分类设计，属于功能层。

---

> 这份骨架延续了观念/原则模块的 "N 维 + 1 评估属性" 结构，但用 `confidence + relevance` 替代了 `impact / cost`——因为这是外部知识，不是自我状态。你看完之后，如果方向 OK，我们再展开 §4 的领域细分、§3 的更多例子、以及 §10 的几个待定问题。

## 当前实现约定

- 社会经济模块已接入 Tauri 本地后端，当前采用 `app_data_dir/socioeconomics.json` 做第一阶段持久化。
- 前端不再使用独立的社会经济 mock 数据文件；样例数据由 Rust 侧 `seed.json` 提供，并在首次读取时落到本地后端链路。
- 页面分为浏览模式与控制模式，控制模式下支持对 `entries / gaps / reviewPrompts` 的新增、编辑与删除。
- 搜索结果只负责展示过滤；保存时始终使用未过滤的原始模块数据，避免因为当前搜索条件导致隐藏数据被误删。
- 国际化采用“界面文案 + 枚举值翻译”方式，实际记录值仍保留设计文档中的中文枚举，避免存储层迁移。
