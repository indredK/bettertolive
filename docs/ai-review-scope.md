# AI 代码审查 / Bug 排查收敛边界

> 用途：当你（或其他 AI agent）被要求"审查某个模块/功能有没有逻辑问题"时，用本文档划定范围，避免无限循环式地一直挖出"理论上可以更好"的点。
>
> 适用对象：bettertolive 全部功能模块（购物、记账、饮食、关系、反思等），无论后端 Rust 还是前端 TS/React。

---

## 核心原则

AI 审查代码理论上永远能找到"可以更好"的点——命名、边界情况、未来可能的需求变化等，这类审查没有自然终点。**收敛的关键不是降低 AI 能力，而是提前定好"什么才算 bug"**，并且区分"这次审查该不该报"。

判断一个发现是否值得报告，先问两个问题：

1. **会不会真的发生**？（用户能否通过现有 UI/操作真的走到这个分支，还是纯假设场景）
2. **发生后影响多大**？（数据丢失 > 决策误导 > 体验别扭 > 风格不一致）

只有"会发生 + 影响达到下方分级门槛"的才算需要修的 bug。

---

## 第一档：无下限，必须找、必须修

这类问题不存在"以后再说"，发现就要报，报了就该修。

1. **数据丢失 / 数据损坏 / 数据孤立**
   本地应用没有云端兜底，用户的记录就是唯一副本。任何会导致已保存数据被静默删除、覆盖、或变成无法再被 UI 访问到的"孤儿数据"的逻辑，无论概率多低都要修。

2. **多步写操作缺事务保护**
   凡是"先 DELETE 再 INSERT/UPDATE"或多条相互依赖的 SQL，且中途失败会留下不一致状态的，必须用 `write_tx` 包裹。这是机械规则——审查时直接搜 `conn.execute` 连续出现、`let conn`（非 `mut`）后面跟写操作的 command，基本可以穷尽，不需要"灵感"。

3. **操作应该成功却失败 / 应该失败却成功**
   用户做了一个看起来合法的操作（编辑、保存、删除），但因为某个分支逻辑被意外拒绝或意外放行，导致功能阻断或数据进入非法状态。

4. **崩溃 / panic**
   `unwrap()` / `expect()` / 数组越界 / 除零 / 索引访问等可能在正常输入下 panic 的代码。

5. **前后端类型契约漂移**
   Rust command 的参数/返回结构改了但没跑 `bun run generate:bindings`，或 `bun run typecheck` 报错。**这条不需要 AI 凭经验找**——跑一遍命令就能穷尽，AI 只需要确认命令已跑且通过。

---

## 第二档：有阈值，按"是否影响真实决策/可见结果"取舍

1. **统计口径 / 聚合数字不一致**
   只有当**用户会看到一个会被当作决策依据的数字**且这个数字算错了，才算 bug（例如总览页的计数、汇总金额）。如果只是两处内部中间变量命名/口径略有差异、但最终展示给用户的数字是对的，不算。

2. **校验缺失导致用户输入被静默丢弃**
   用户填了数据、点了保存，数据却消失且没有任何提示——算 bug。但如果用户填错后**保存即可见结果不对**（比如刷新页面立刻能看到没保存上），用户自己会发现，优先级降低，不必主动报。

3. **状态/枚举类语义不一致**
   两个函数对同一个业务状态（比如"已购买/想要"）的判定逻辑不一致，导致同一份数据在不同 UI 位置显示不同结论——算 bug。如果只是文案措辞不同但底层判定逻辑一致，不算。

---

## 第三档：明确划红线，不再追、不要报

遇到以下情况，直接归档为"已知限制"或忽略，**不要在新一轮审查中重复提出**：

1. **多写者/并发场景**
   这是单用户本地桌面应用，几乎不存在多进程/多窗口同时写同一份数据的场景。已有的乐观锁（如属性定义的 `version` 字段）是为同一用户在多个标签页/弹窗间编辑设计的特例，不要以此为由去设计"如果两个用户同时编辑"之类的假设性问题。

2. **死代码 / 未使用字段**
   遗留字段（如废弃的价格字段）、永远返回 null 的函数等——这是技术债，归入单独的 cleanup 清单批量处理一次性解决，不要每次审查都重复提出同一处死代码。

3. **纯 UI/文案细节**
   按钮样式、disabled 态、图标、措辞——除非这个细节**直接导致用户能提交无效数据或丢失数据**（这种情况升级到第一/二档），否则归入"打磨阶段"任务，开发期不追。

4. **假设性扩展场景**
   "如果用户通过某种方式创建了两个语义键相同的属性会怎样" —— 如果现有 UI/校验根本不允许产生这种数据，就不算 bug。除非能证明通过现有界面真的可以走到这个状态。

5. **架构级"理想化"重构建议**
   "这里应该抽象一层 / 这个模式应该统一成另一种模式" —— 除非现状已经造成第一/二档问题，否则不是 bug，是设计偏好讨论，不要混进 bug 列表。

---

## 审查流程建议

1. **优先做 diff 审查，而不是整模块重新扫描**
   日常开发中，新一轮审查应聚焦"这次改动涉及的文件 + 直接调用链"，而不是每次都把整个模块从头看一遍——否则会反复挖出同一批历史遗留的第三档问题。

2. **全模块审计只在里程碑节点做一次**
   比如某个模块开发阶段告一段落、准备进入打磨/测试前，做一次全量审查，按本文档分级输出，第三档的问题统一记入"已知限制"清单，不再单独跟踪。

3. **结束条件**
   当某一轮审查在第一档 + 第二档下结果为 0（或全部已修复）时，本轮审查即为完成，不需要继续深挖第三档问题去凑数。

4. **记录格式**
   发现的 bug 按文件分级记录（参考 `docs/bug.md` 历史格式）：文件路径 + 行号 + 描述 + 复现步骤 + 修复建议。第三档的"已知限制"可以单独记一份清单，避免和真 bug 混在一起。
