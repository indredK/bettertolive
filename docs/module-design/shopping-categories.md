# 购物模块 - 生活物品分类建议

## 1. 30 秒看懂这份文档

这份文档建议给每件生活物品打 **5 个分类标签** + **1 个评估属性**：

> **当前产品边界**：这个购物模块在 V1 里主要是一个**只读内容工作台**。重点是整理生活物品分类、阶段模板和采购判断，不是做一个以“已买 / 待买”流转为核心的操作系统。

### 5 个分类标签（用来归类、过滤、看集合）

| 标签 | 它回答什么 | 取值示例 |
|---|---|---|
| **system**（生活系统） | 它在撑哪个生活功能？ | 睡眠 / 饮食 / 清洁 / 照明 / ……（共 14 个，见 §4） |
| **space**（空间） | 它放在哪？ | 卧室 / 客厅 / 书桌 / 玄关 |
| **stage**（阶段） | 当前生活阶段需要它吗？ | 搬家最低配 / 租房 / 长期居住 / 自有住房 |
| **necessity**（必要性） | 它有多重要？ | 最低配置 / 必要 / 改善体验 / 提升幸福感 |
| **lifecycle**（生命周期） | 它该多久换一次？ | 消耗品 / 耐用品 / 工具 / 情感物 |

### 1 个评估属性（跟着每件物品走，用来判断"这一件"）

| 属性 | 它回答什么 | 取值示例 |
|---|---|---|
| **depreciation**（折旧率） | 它的价值衰减多快？ | 极快 / 较快 / 中等 / 慢 / 不折旧或升值 |

> **5 维和折旧率的区别**：5 维用来**归类**（"睡眠系统里都有什么"）；折旧率用来**评估**（"这一台手机该不该买、能不能保值"）。详见 §7。

打满这 5 个标签 + 1 个属性后，原来问不出的问题就能问出来了：

- "我家是不是还差什么？"
- "下个月该补什么？"
- "搬家先打包什么？"
- "这件买不买？买新还是二手？"

下面所有内容都在解释**为什么是这套标签、14 个系统怎么定、遇到边界情况怎么办**。如果只看一节，跳到 §10（典型问题 → 怎么用回答）。

## 2. 为什么不按"它是什么东西"分类

如果按"它是什么"分，会出现这样的列表：

> 杯子 / 椅子 / 灯 / 床 / 衣服 / 鞋 / 锅 / 刷子 / 手机 / …

这种分类的问题：

1. **看不出缺口**——你知道自己有 5 个杯子，但不知道"早晨喝咖啡的杯子是不是没有"
2. **驱动不了下一步**——知道"灯有 3 盏"，不知道"床头是不是缺一个能调亮度的"
3. **重复购买不可见**——两个空净放在不同房间，按品类看不出"是不是重复了"
4. **跟人生阶段对不上**——你正在搬家，但"我有几把椅子"对决策无用，"客厅起居系统够不够撑 6 个月"才有用

### 换一个问法

不要问"这是什么"，问 "**它在撑哪个生活系统？**"

> **什么叫"系统"** —— 一组物品共同撑起某个生活功能。少一个，整个功能就会"卡顿"。
>
> 比如「睡眠系统」由床垫、被子、枕头、遮光帘、白噪音、温控这些一起组成。少了遮光帘，整个睡眠质量都会下降——不是"少了一件东西"那么简单。

### 按系统分类的好处

- **看缺口比看数量有用**："睡眠系统里没有遮光" 比 "床品有 5 件" 更能驱动下一步
- **跟生活叙事一致**：spotlight 现在写的"睡眠系统还没真正稳定"、"潮湿季节要准备" 本来就是按系统讲的
- **避免重复购买**：同系统下的东西放一起对比，能看出"两个空净是不是冗余了"
- **跟阶段清单对得上**：每个阶段清单都可以按系统列"最低配 / 必要 / 之后再升级"

## 3. 用两个具体物品走一遍

抽象地说没感觉，先看两件具体物品被打满的样子。后面遇到不理解的地方，回到这里。

### 例 1：一床冬天用的厚被

**5 个分类标签**：

| 维度 | 取值 | 为什么 |
|---|---|---|
| system | **睡眠** | 它撑起的是睡眠功能，不是装饰 |
| space | **卧室** | 一般就放在床上 |
| stage | **租房起就需要** | 不论房屋阶段都得有，搬家最低配里也得带一床 |
| necessity | **必要** | 冬天没有它睡不好；不是"提升体验"级 |
| lifecycle | **耐用品** | 用 3-5 年，磨损或太旧才换 |

**评估属性**：

| 属性 | 取值 | 含义 |
|---|---|---|
| depreciation | **慢折旧** | 一床好被子用 5 年价值变化不大，值得一次买好的 |

> 现在你可以问：**我的睡眠系统里**（system）属于**必要级别**（necessity）的**耐用品**（lifecycle），**寝具这一组**有没有缺口？买的时候**因为慢折旧**（depreciation）所以选耐久款而不是赶新款。

### 例 2：一盏书桌阅读用的护眼台灯

**5 个分类标签**：

| 维度 | 取值 | 为什么 |
|---|---|---|
| system | **照明** | 它撑起的是"光环境"。即便放在书桌上，它也不算"工作学习"系统 |
| space | **书桌** | 它就在书桌上 |
| stage | **长期居住起** | 短期凑合用顶灯也行，但长期工作必须有护眼光源 |
| necessity | **必要** | 在「照明系统」里这是骨架级 |
| lifecycle | **耐用品**（但灯泡是消耗品） | 灯本身用很多年；灯泡定期换 |

**评估属性**：

| 属性 | 取值 | 含义 |
|---|---|---|
| depreciation | **中等折旧** | 中端品牌新品几年贬一半，但二手保值率一般，建议买二线品牌成熟款 |

> 两个物品都是"必要 + 耐用品"，但分属不同 system，且折旧率不同——这就是 **5 维互不替代** 加上 **评估属性穿透 5 维** 的体现。

## 4. 第 1 维：14 个生活系统

### 为什么是 14 个

不是按现成的家居品类划，而是**反推**——一个普通成年人的家庭日常要支撑 11 件基础功能 + 3 件生活方式 + 按需扩展。

### 基础系统（11 个）

| 系统 | 它撑起的功能 | 典型物品 | 看完整性的关键问题 |
|---|---|---|---|
| 睡眠 | 撑起 1/3 的人生 | 床垫、床架、枕、寝具、遮光、白噪音 | "我睡得着、睡得好吗" |
| 饮食 | 食物从买到吃完的全链路 | 锅具、刀具、餐具、餐桌、储存、净水 | "我能舒服地做+吃三餐吗" |
| 清洁 | 让家维持"干净状态"的循环 | 扫除、洗衣机、垃圾、消毒 | "家在变干净还是变脏" |
| 收纳 | 让物品有"家"，找得到放得回 | 衣柜、收纳盒、挂钩、储物架 | "找东西卡顿吗" |
| 照明 | 一天不同时间的光环境 | 主灯、台灯、氛围灯、夜灯、遮光帘 | "白天 / 夜晚 / 睡前的光都对吗" |
| 环境 | 温湿度、空气、噪音 | 空调、加湿、除湿、空净、风扇 | "家里待着舒服吗" |
| 电力网络 | 电与信号的供给 | 路由、插座、延长线、UPS、充电 | "用电用网会卡吗" |
| 工作学习 | 一个能坐住的"生产空间" | 办公桌、人体工学椅、显示器、文具 | "我能坐住吗" |
| 应急健康 | 平时用不上但关键时刻救命 | 药箱、备品、保险单、文件备份 | "突发情况能撑过去吗" |
| 个人护理 | 身体的日常维护 | 洗护、护肤、口腔、剪发 | "身体被照顾着吗" |
| 穿着 | 一年四季能体面出门 | 日常衣、内衣袜、鞋、配饰、外套、正装 | "下楼 / 见人 / 出差都有合适衣服吗" |

### 家居与生活方式（3 个）

| 系统 | 它撑起的功能 | 典型物品 |
|---|---|---|
| 家具陈设 | 空间感、氛围、起居 | 沙发、茶几、地毯、装饰窗帘、装饰画、植物、摆件 |
| 出行 | 把人和东西带出门 | 自行车、汽车、行李箱、通勤包、雨伞、户外装备 |
| 娱乐爱好 | 沉浸 / 兴趣 / 放松 | 书、乐器、相机、游戏机、运动器材、藏品 |

### 按需扩展（不是人人都需要，按生活状态激活）

只在用户处于这种生活状态时打开，避免给单身上班族显示"育儿系统空"。

- **宠物**：食盆、猫砂盆、笼具、护理用品、出行装备
- **育儿**：婴儿用品、奶粉/辅食、玩具、儿童家具、学习用品
- **车辆维护**：保养工具、车载应急包、洗车用品、行车记录仪
- **植物养护**：花盆、营养土、肥料、喷壶、修剪工具、植物灯（几盆并入家具陈设；认真养独立）
- **创作生产**：绘画 / 手作 / 木工 / 烘焙 / 摄影专业器材（区别于工作学习的脑力工种装备）
- **居家维修**：螺丝刀、扳手、电钻、五金件、胶带、备件（不自己修就并入应急健康；动手就独立）
- **节庆与季节**：圣诞树、春节装饰、电热毯、电风扇、防潮垫（一年用一两次但每年重复用）
- **健康追踪**：血压计、体重秤、血糖仪、健身环、睡眠监测（少量并入应急健康；老人/慢病管理时独立）

## 5. 第 5 维：lifecycle（生命周期）

`system + space + stage + necessity` 这 4 维回答的是 "**这件东西该不该在我生活里？放哪？有多重要？**"

加上 `lifecycle`，回答的是 "**它在我生活里走的是什么节奏？**"

### 4 种 lifecycle

| 类型 | 节奏 | 触发什么动作 | 典型物品 |
|---|---|---|---|
| **消耗品** | 会被用完、需要补 | 用完前补 | 米油、洗洁精、卫生纸、灯泡、电池、护肤品、宠物粮 |
| **耐用品** | 长期使用、偶尔升级 | 磨损或失效才换 | 床垫、锅具、椅子、衣柜、洗衣机、相机 |
| **工具** | 偶尔用一次但必须有 | 坏了再补，少备一个就行 | 螺丝刀、扳手、应急灯、剪刀、量尺、急救包 |
| **情感物** | 不靠功能存在 | 不淘汰、不补充 | 纪念品、礼物、收藏品、相册、童年物 |

### lifecycle 不能被 necessity 替代

很多人会问："最低配置 / 必要 / 改善体验 / 提升幸福感" 不是已经告诉我重不重要了吗？为什么还要 lifecycle？

因为它们回答不同问题：

- `necessity` 回答："**没有它，生活会不会塌？**" —— 这是优先级
- `lifecycle` 回答："**它在我生活里以什么节奏出现？**" —— 这是动作节奏

同样是"必要"，米油（消耗品，每月看库存）和床垫（耐用品，5 年才换一次）的动作完全不同：

| 物品 | necessity | lifecycle | 实际动作 |
|---|---|---|---|
| 米油 | 必要 | 消耗品 | **每月补** |
| 床垫 | 必要 | 耐用品 | **5 年换一次，看好价** |
| 螺丝刀 | 必要 | 工具 | **平时不动它，坏了才补** |

只有 necessity 的话，这三件物品看起来"同样紧急"，但实际行动完全不同。**lifecycle 决定了 necessity 后的下一步动作。**

## 6. 五维速览

完整的 5 维分类结构：

| 维度 | 答什么 | 取值 | 关键作用 |
|---|---|---|---|
| `system` | 撑哪个生活功能 | 14 个生活系统（见 §4） | 看缺口 |
| `space` | 放在哪 | 房间 / 区位 | 按空间检视 |
| `stage` | 当前阶段需要吗 | 搬家最低配 / 租房 / 长期居住 / 自有住房 | 决定深度 |
| `necessity` | 它有多重要 | 最低配置 / 必要 / 改善体验 / 提升幸福感 | 排优先级 |
| `lifecycle` | 它的节奏 | 消耗品 / 耐用品 / 工具 / 情感物 | 决定动作 |

5 维互不替代，每件物品都应打全 5 个标签。

## 7. 物品的评估属性：折旧率

> **重要前提**：折旧率**不是第 6 个分类维度**，它是穿透所有 5 维的**物品内禀属性**。
> 5 维是用来**归类**的（按 system 聚合、按 stage 过滤、按 lifecycle 看动作节奏），折旧率是用来**评估单件物品**的（这一件该不该买、买新还是二手、修还是换）。

### 5 维 vs 折旧率的区分

| 5 个分类标签 | 折旧率（评估属性） |
|---|---|
| **组织物品集合**：让你能问"睡眠系统里都有什么" | **判断单件物品**：让你能问"这一台手机该不该买" |
| 出现在过滤器、TabsList、分组、统计里 | 跟着每件物品走，在物品详情/购买决策时显示 |
| 用来回答"我有什么、缺什么、放哪了" | 用来回答"这件值不值、能保值吗、什么时候出手最划算" |

### 5 级取值

| 等级 | 年贬值参考 | 典型物品 | 决策含义 |
|---|---|---|---|
| **极快折旧** | >30%/年 | 手机、笔记本、相机、新车前 3 年 | 别堆顶配；旗舰半年就降一大截；考虑二手 |
| **较快折旧** | 15-30%/年 | 大家电、小电子 | 选成熟款胜过赶新款 |
| **中等折旧** | 5-15%/年 | 家具、大件耐用 | 值得买好的、用 8-10 年 |
| **慢折旧** | <5%/年 | 铁锅、经典款服装、好工具、瑞士军刀 | 一次到位最优 |
| **不折旧 / 升值** | ≥0%/年 | 老木家具、收藏品、贵金属、艺术品 | 视为资产 |

### 适用边界（字段是可选的）

不是每件物品都有折旧率。在数据结构上**设为 optional**：

- ✅ `lifecycle = 耐用品 / 工具` —— 有市场价、可二手流转，折旧率有意义
- ❌ `lifecycle = 消耗品` —— 拆封即归零，没有折旧曲线
- ❌ `lifecycle = 情感物` —— 不按市场价值衡量

### 它能回答的问题

5 维都答不出来，但很常见：

| 问题 | 折旧率怎么帮 |
|---|---|
| 该买新还是二手？ | 极快折旧 → 买二手；慢折旧 → 二手溢价小，买新更省心 |
| 该升级还是再用一年？ | 极快折旧 → 再用一年没多大代价（旧的本来就没多少残值）；慢折旧 → 升级太亏 |
| 该保养还是直接换？ | 极快折旧 → 修不如换；慢折旧 → 值得保养 |
| 租还是买？ | 极快折旧 + 短期使用 → 租；慢折旧 + 长期使用 → 买 |
| 这件能保值吗？ | 直接看 |
| 我是不是在被消费掏空？ | 警惕"提升幸福感 + 极快折旧"组合（VR、最新款电子产品） |

### UI 呈现建议

- ✅ 物品详情页**始终显示**折旧率（伴随物品）
- ✅ 购买决策视图（采购决策 tab 的某件物品卡片）显示折旧率作为决策辅助
- ✅ 可以做"折旧率 × necessity"的交叉提示（红色警告：你的"提升幸福感 + 极快折旧"占比过高）
- ❌ **不要**把折旧率放进 TabsList 或主筛选器（那是 5 维的位置）
- ❌ **不要**做"按折旧率分类的物品列表"（那会让它变成第 6 维，违背定位）

## 8. 边界约定（查询表，不必通读）

> 这一节是**查询参考**——平时不必通读，遇到一件不知道归哪个 system 的物品时回来查这里。

### 决策规则

> 先问"**它存在的目的是什么**"，再问"**这个目的最强归哪个 system**"。
> 如果两个 system 都强，选**用户用它的那个场景**（不是"它本身是什么"）。

### 常见混淆物品的归属

| 物品 | 归属 | 理由 |
|---|---|---|
| 床架 | 睡眠 | 床架是睡眠系统的骨架，不是"一件家具" |
| 办公桌椅 | 工作学习 | 它存在的目的就是工作 |
| 餐桌 | 饮食 | 吃饭场景的核心 |
| 茶几、边几 | 家具陈设 | 没有强单一功能，是客厅起居的陈设 |
| 衣柜 | 收纳 | 衣柜是装衣服的容器，不是穿着系统本身 |
| 床头灯、阅读灯 | 照明 | 照明系统的一部分，不必拆到睡眠 |
| 遮光帘 | 睡眠 | 它存在是为了睡眠质量 |
| 装饰窗帘 | 家具陈设 | 它存在是为了空间感 |
| 通勤包 | 出行 | 用于把东西带出门 |
| 时尚手提包 | 穿着（配饰） | 它是搭配的一部分 |
| 跑步机、瑜伽垫 | 娱乐爱好（运动器材） | 运动是个人活动 |
| 哑铃 | 娱乐爱好 | 同上 |
| 户外帐篷 | 出行 | 出门才用 |
| 香薰、蜡烛 | 家具陈设 | 起氛围作用 |
| 加湿器、除湿机 | 环境 | 调节空气，不是装饰 |
| 文件柜 | 收纳 | 装文件的容器 |
| 保险柜 | 应急健康 | 存放应急文件、证件、备用钥匙 |
| 纸质相册 | 娱乐爱好（或独立"记忆"模块） | 不靠功能存在，承载情感 |
| 钢琴、电子琴 | 娱乐爱好 | 是兴趣器材，占空间在 space 体现 |
| 厨房消耗品（米油盐） | 饮食 | 不要拆到"消耗品系统"，消耗是 lifecycle 维度 |
| 工具螺丝刀套装 | 应急健康 或 居家维修（扩展） | 没维修习惯就并入应急健康 |
| 电热毯 | 节庆与季节（扩展）或 睡眠 | 一年用 3 个月就独立成季节；天天用就归睡眠 |
| 蓝牙音箱 | 家具陈设 或 娱乐爱好 | 放氛围音乐就家具陈设；认真听音乐就娱乐爱好 |
| 投影仪 | 娱乐爱好 | 区别于工作显示器 |
| 体重秤、血压计 | 应急健康 | 健康监测的基础件 |
| 行车记录仪 | 车辆维护（扩展）或 出行 | 没扩展车辆系统就并入出行 |
| 化妆品 | 个人护理 | 身体维护的一部分 |
| 美妆工具（眉夹、卷发棒） | 个人护理 | 同上 |

## 9. 系统内的二级分组

### 什么时候用二级分组

| 条件 | 用不用二级 |
|---|---|
| 该系统物品数 > 15 件 | ✅ 用 |
| 系统内物品有明显不同的"使用场景"或"动作节奏" | ✅ 用 |
| 系统物品 < 10 件，且都是同一类东西 | ❌ 不用 |

### 各系统的二级分组

> 二级分组用 `category` 内的子字段或 tags 实现，**不另立 system**。

#### 睡眠
- **骨架**：床架、床垫、床头板
- **寝具**：被子、枕头、床单、枕套、被套、保护垫
- **温控**：薄被、厚被、电热毯、凉席、空调被
- **环境**：遮光帘、白噪音机、香薰、夜灯
- **睡眠辅助**：眼罩、耳塞、助眠喷雾、记录设备

#### 穿着
- **基础件**：内衣、袜子、家居服（消耗品逻辑，磨损就换）
- **日常衣**：T 恤、衬衫、裤、裙（搭配逻辑，看搭配组合是否齐全）
- **外套**：春秋外套、羽绒服、雨衣（季节逻辑，过季就退场）
- **鞋**：日常鞋、运动鞋、正装鞋、室内鞋（场景逻辑）
- **配饰**：包、帽子、围巾、首饰、表（点缀逻辑）
- **特殊场合**：正装、礼服、面试装、户外/运动装

#### 饮食
- **烹饪工具**：锅、刀、案板、小家电
- **餐具**：碗、盘、杯、筷勺
- **储存**：保鲜盒、密封罐、冰箱整理
- **食材消耗**：米油盐糖、调料、常备干货
- **家具**：餐桌、餐椅

#### 清洁
- **日常打扫**：扫帚、拖把、扫地机器人、抹布
- **洗衣**：洗衣机、洗衣液、烘干机、衣架、洗衣袋
- **厨卫深度清洁**：去油剂、马桶刷、水垢清洁剂
- **垃圾处理**：垃圾桶、垃圾袋、厨余处理器
- **消毒杀菌**：消毒液、紫外灯、洗手液

#### 收纳
- **大件容器**：衣柜、储物柜、置物架、文件柜
- **小件容器**：收纳盒、抽屉分隔、密封袋、瓶罐
- **挂收纳**：挂钩、挂袋、墙面收纳
- **标签与索引**：标签机、便签、分类索引

#### 家具陈设
- **大件家具**：沙发、茶几、地毯、餐边柜
- **软装**：窗帘、抱枕、毯子
- **装饰物**：装饰画、摆件、植物、镜子、香薰
- **改善型小件**：储物凳、托盘、置物架（兼具收纳但视觉占位强）

#### 工作学习
- **桌椅与支撑**：办公桌、人体工学椅、显示器支架
- **显示与输入**：显示器、键盘、鼠标、绘图板
- **文具与纸面**：本子、笔、文件夹、打印机
- **辅助**：耳机、台灯（仅工作场景）、降噪/隔音

#### 出行
- **代步**：自行车、电动车、汽车
- **承载**：行李箱、通勤包、保温壶、外出杯
- **应对天气**：雨伞、雨衣、防晒用品
- **户外**：帐篷、睡袋、登山装备

#### 娱乐爱好
- **阅读**：书、kindle、阅读灯、书架
- **影音**：投影、音箱、耳机、游戏机
- **运动器材**：瑜伽垫、哑铃、跑步机、健身环
- **乐器与艺术**：吉他、钢琴、画具、手工材料
- **收藏与情感**：纪念品、模型、相册、藏品

#### 个人护理
- **洗护**：洗发、沐浴、护发、剃须
- **护肤化妆**：护肤品、化妆品、美妆工具
- **口腔**：电动牙刷、牙膏、牙线、漱口水
- **形象修整**：剪发器、指甲剪、眉夹、卷发棒

体量小的系统（应急健康、电力网络、环境、照明）暂不需要二级分组，等条目堆到 15 件以上再分。

## 10. 典型问题 → 怎么用回答

读完前面所有，最关键的是这一节——它告诉你**这套分类能帮你做哪些日常决定**。

| 你的真实困惑 | 用哪几个维度 / 属性 |
|---|---|
| "我家是不是还差什么？" | 看每个 `system` 里 `necessity=必要` 的项有没有缺口 |
| "下个月该补什么？" | 按 `lifecycle=消耗品` 看库存 |
| "搬家先打包什么？" | 按 `stage=搬家最低配` 过滤 |
| "我家睡眠 / 工作 / 饮食 …… 系统稳了吗？" | 看单个 `system` 内所有 `necessity=必要` 是否齐全 |
| "这件买不买？" | 看 `lifecycle`——情感物的判断逻辑和功能物完全不同 |
| "现在应该等好价的是哪些？" | `necessity=改善体验` + `lifecycle=耐用品` |
| "卧室东西够不够？" | 按 `space=卧室` 过滤，看 `system` 分布有没有缺口 |
| "做断舍离应该先扔哪些？" | `lifecycle=情感物` 单独处理；其他按 `necessity` 倒序 |
| "圣诞快到了该提前备什么？" | 扩展系统 `节庆与季节`，看 `lifecycle=消耗品` |
| "下次发工资买点什么改善生活？" | `necessity=改善体验` 或 `提升幸福感`，按 `system` 看哪个最弱 |
| "保险单、证件这种重要文件够齐全吗？" | `system=应急健康` 里的文件类清单 |
| "看着东西很多但总觉得缺，到底缺什么？" | 按 `system` 列每一项的 `necessity=必要` 清单，对比已有 → 缺口就显现了 |
| "这台手机该买新还是二手？" | 看这件物品的 `depreciation`（评估属性，不是过滤）—— 极快折旧→建议二手 |
| "我是不是在被消费掏空？" | 扫描所有物品的 `depreciation × necessity` 组合 —— "提升幸福感 + 极快折旧" 占比高就要警惕 |
| "我现在要换手机，旧的能卖多少？要不要修？" | 看旧手机的 `depreciation`（评估属性）—— 极快折旧的东西修不如换 |

## 11. 实施步骤（给开发者）

把这套结构落到 mock 数据 / 类型层：

1. 把 14 个系统作为 `system` 枚举写出来
2. 把 4 个 lifecycle 类型一起写出来（消耗品 / 耐用品 / 工具 / 情感物）
3. 把 5 个 depreciation 等级写出来（极快 / 较快 / 中等 / 慢 / 不折旧）
4. 在 `ShoppingOwnedItem` / `ShoppingPlanItem` 类型上：
   - 新增 `system: SystemEnum` （必填）
   - 新增 `lifecycle: LifecycleEnum` （必填）
   - 新增 `depreciation?: DepreciationEnum` （**可选**，仅耐用品和工具填）
5. 把 `ownedItems` 和 `purchaseLanes` 里的现有条目逐个归位
6. 体量大的系统（穿着、饮食、家具陈设、工作学习、出行、睡眠、清洁、收纳、娱乐爱好、个人护理）再分二级
7. 阶段清单（`stageChecklists`）的 `minimum / essentials / upgrades` 按系统重新分组
8. 价格参考（`priceReferences`）按系统挂一遍
9. 按需扩展系统（宠物、育儿等）等用户激活了再开，不要预先全开
10. UI 层注意：**5 维**进入 TabsList / 过滤器 / 分组；**depreciation** 只进物品详情和单件决策卡片，不做集合过滤

## 12. 不在本文档讨论的范围

- 高密度录入、编辑、增删的交互
- 跨模块联动（购物 ↔ 记账 ↔ 反思）

这些等分类稳定之后再单独讨论。

## 13. 当前实现与存储建议

### 13.1 当前实现重点

当前购物模块更适合先做成：

- 一个展示生活系统的只读工作台
- 一组按 `system / space / stage / necessity / lifecycle` 组织起来的内容快照
- 一套帮助判断“缺什么、先补什么、哪些值得慢慢买”的参考视图

这意味着 V1 不需要先把重点放在：

- 已买 / 待买状态流转
- 高频新增条目
- 复杂表单录入
- 大量事务型后端接口

### 13.2 当前数据库方向

如果后端要开始落地，当前最合适的起点是：

- `SQLite`
- 单人、本地优先
- 先按**内容快照**存，而不是一开始就拆很多关系表

推荐用一张类似下面的表承接：

- `shopping_module_content`
  - `id`
  - `module_key`
  - `content_json`
  - `version`
  - `updated_at`

其中 `content_json` 直接对应前端当前消费的 `ShoppingModuleData`，也就是：

- `systemDefinitions`
- `spotlights`
- `ownedItems`
- `purchaseLanes`
- `stageChecklists`
- `priceReferences`
- `boundaryEntries`
- `lifestyleCollections`

等未来真的出现稳定的编辑需求，再决定哪些内容要拆成独立表。
