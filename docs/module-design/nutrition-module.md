# 饮食模块 - 食谱、食品、营养与每日计划

## 1. 模块重新定位

旧版饮食模块的中心是“记录一次进食并观察场景、触发、身体反馈”。这个方向保留为回顾能力，但不再是主轴。

新版饮食模块的目标是：把“我能吃什么、怎么做、营养大概怎样、今天怎么安排、实际吃成什么样”放进同一个生活工作台。

一句话：

> 饮食模块不是打卡减肥 app，也不是公共菜谱站，而是个人饮食系统的“食谱库 + 食品库 + 营养表 + 每日计划 + 进食回顾”。

## 2. 与购物模块的对应关系

购物模块的核心是“一个物件库，多种视图”。饮食模块也采用同样的结构：

| 购物模块 | 饮食模块 |
|---|---|
| 物件库是唯一数据源 | 食品库是营养与食谱的基础数据源 |
| 系统地图 / 空间巡检是不同 group-by 视图 | 食品分类 / 营养成分表是食品库的不同视图 |
| 阶段模板引用物件并配置档位 | 每日计划引用食谱 / 食品并配置餐次 |
| 采购决策维护物件价格与渠道 | 食谱库维护做法、份量、食材和可执行性 |
| 总览体现设计初衷 | 总览体现今天吃什么、缺什么、怎么补 |

核心约束：

- 食品库是基础数据源。
- 食谱引用食品库，不复制营养数据。
- 每日计划引用食谱或食品，不直接复制食谱内容。
- 实际进食记录可以从计划生成，也可以独立记录。
- 营养成分表是食品库和食谱的派生视图，不作为独立录入孤岛。

## 3. 顶层 Tab 设计

新版饮食模块建议 6 个 Tab：

| Tab | 主要职责 | 类比购物模块 |
|---|---|---|
| 总览 | 一页看今天计划、营养缺口、推荐动作、近期结构 | 总览 |
| 每日计划 | 按日期安排早餐 / 午餐 / 晚餐 / 加餐 / 饮品 | 阶段模板 + 采购决策 |
| 食谱库 | 管理可复用做法、食材、步骤、份量、标签 | 物件库 |
| 食品分类 | 按食物类别、使用场景、储存方式、饮食立场分组 | 系统地图 |
| 营养成分表 | 按食品 / 食谱查看每 100g、每份、每日合计营养 | 空间巡检 |
| 进食记录 | 记录实际吃了什么、身体反馈和偏差原因 | 计划校准与回顾区 |

### 为什么不是只做四个 Tab

用户提出的“食谱、营养成分表、食品分类、每日计划”是四个核心对象，但实际使用时还需要两个闭环：

- 总览负责把四个对象收敛成今天的判断。
- 进食记录负责把计划和现实连接起来。

如果没有总览，模块会变成资料库；如果没有记录，计划无法校准。

## 4. 总览设计

总览是一页式工作台，默认不整体滚动；被高度或宽度挤压时允许模块内部出现滚动，遵守 [布局规范](../layout-spec.md)。

总览分为三块：

| 区块 | 内容 | 目的 |
|---|---|---|
| 今日餐盘 | 今日餐次、已安排 / 未安排、待补食材 | 让用户知道今天怎么吃 |
| 营养脉冲 | 蛋白、蔬果、主食、油盐糖、饮水 / 饮品提示 | 让用户知道今天缺什么 |
| 可执行建议 | 可做食谱、可替换食品、剩余食材优先使用 | 让用户知道下一步做什么 |

右侧还需要一个紧凑的“饮食意图”卡片，用来展示当前阶段目标、硬约束和软立场。它不是新的独立 Tab，而是对今日餐盘和可执行建议的解释层：先知道哪些东西不能吃、哪些只是阶段性关注，再决定今天怎么安排。

总览要体现“设计初衷”：

- 不是批评“吃得好不好”。
- 不是要求每日精确克数。
- 是把饮食从零散选择变成可回看的生活系统。

推荐布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ 今日饮食总览：今天不是要完美，而是要能被照顾到              │
├───────────────────────────────┬──────────────────────────────┤
│ 今日餐盘                       │ 营养脉冲                     │
│ 早餐 / 午餐 / 晚餐 / 加餐       │ 蛋白 / 蔬果 / 主食 / 油盐糖   │
├───────────────────────────────┴──────────────────────────────┤
│ 饮食意图 / 可执行建议：当前目标、边界、缺什么、哪个食谱合适   │
└──────────────────────────────────────────────────────────────┘
```

视觉风格应沿用购物总览的“主题色 + 卡片分区 + 图表点缀”方向，但颜色全部走全局主题 token。

## 5. 每日计划

每日计划是饮食模块的行动入口。

### 计划粒度

一个 `DailyPlan` 对应一天，内部包含多个 `MealSlot`：

| 餐次 | 是否默认展示 | 说明 |
|---|---|---|
| 早餐 | 是 | 可以为空，但要能看出是否跳过 |
| 午餐 | 是 | 主要计划位 |
| 晚餐 | 是 | 主要计划位 |
| 加餐 | 可选 | 零食、水果、补剂类食物 |
| 饮品 | 可选 | 咖啡、酒、奶茶等有摄入意义的饮品 |

### 每个餐次能放什么

餐次条目支持三种来源：

| 类型 | 例子 | 说明 |
|---|---|---|
| 食谱 | 番茄鸡蛋面、清蒸鱼套餐 | 推荐主入口 |
| 单个食品 | 香蕉、牛奶、鸡蛋 | 适合早餐、加餐、补充项 |
| 临时文本 | 公司食堂、朋友聚餐 | 不阻断计划流程 |

### 计划状态

| 状态 | 含义 |
|---|---|
| planned | 已计划 |
| prepared | 已准备食材 |
| eaten | 已吃 |
| skipped | 跳过 |
| replaced | 被替换 |

计划不是打卡压力，而是“预案”。允许用户当天替换、跳过、补记。

## 6. 食谱库

食谱库不是公共菜谱收藏，而是“我真的会做 / 愿意做 / 想复刻”的个人食谱库。

### 食谱结构

| 字段 | 含义 |
|---|---|
| name | 食谱名 |
| summary | 简短说明 |
| servings | 份数 |
| mealRoles | 适合早餐 / 午餐 / 晚餐 / 加餐 |
| ingredients | 引用食品库的食材和用量 |
| steps | 步骤 |
| prepMinutes / cookMinutes | 准备与烹饪时间 |
| difficulty | 简单 / 中等 / 麻烦 |
| repeatability | 常做 / 偶尔 / 只想记录 |
| tags | 家常、清淡、高蛋白、快手等 |
| linkedFoodMemoryId | 可选，关联食物记忆 |

### 食谱卡片应展示

- 名称和一句话说明。
- 适合餐次。
- 总时间。
- 主要食材。
- 每份营养摘要。
- 是否缺食材。
- 最近一次实际吃后的身体反馈。

### 食谱和食品库的关系

食谱的营养来自食材：

```text
Recipe
  ├─ ingredient: foodId + grams
  ├─ ingredient: foodId + grams
  └─ derived nutrition per serving
```

食谱不直接维护“蛋白质 30g”这类结果，除非进入手动覆盖模式。

## 7. 食品分类

食品分类用于组织“我可以买、会吃、会用来做饭的食物”。

### 食品库是基础层

`FoodItem` 是饮食模块最底层的可复用对象。

| 字段 | 含义 |
|---|---|
| name | 食品名 |
| categoryIds | 食品分类，多选 |
| defaultUnit | g / ml / 个 / 份 |
| edibleRatio | 可食部比例，可选 |
| storage | 常温 / 冷藏 / 冷冻 / 即食 |
| lifecycle | 新鲜短期 / 常备 / 干货 / 调味 / 饮品 |
| allergenTags | 过敏原 |
| dietaryTags | 素食、低碳、清真等 |
| nutrientProfileId | 关联营养成分 |

### 分类维度

食品分类不是单一树，而是多维标签：

| 维度 | 示例 | 用途 |
|---|---|---|
| 食物大类 | 主食、蛋白、蔬果、脂肪、调味、饮品 | 看结构 |
| 食材形态 | 原食材、半成品、成品、调味品 | 看加工度 |
| 储存方式 | 常温、冷藏、冷冻、即食 | 看备餐可行性 |
| 使用频率 | 常备、高频、偶尔、节令 | 看采购和计划 |
| 饮食立场 | 过敏、医学限制、软立场适配 | 看能不能吃 |

这和购物模块的 `systemTags / spaceTags` 类似：食品可以同时属于多个分类。

## 8. 营养成分表

营养成分表负责把食品库、食谱库和每日计划连接起来。

### 三种查看粒度

| 粒度 | 内容 |
|---|---|
| 每 100g / 100ml | 食品基础营养 |
| 每份 | 食谱按份量推导 |
| 每日合计 | DailyPlan 或实际进食合计 |

### V1 必须支持的营养字段

| 字段 | 说明 |
|---|---|
| energyKcal | 能量 |
| proteinG | 蛋白质 |
| fatG | 脂肪 |
| carbG | 碳水 |
| fiberG | 膳食纤维 |
| sugarG | 糖 |
| sodiumMg | 钠 |

### V1 可选字段

- 钙、铁、钾。
- 饱和脂肪。
- 咖啡因。
- 酒精。
- 维生素相关字段。

### 设计边界

- 不做医疗级营养建议。
- 不强制每个食品都有完整营养。
- 不因为营养缺失阻断食谱和计划。
- 营养缺失要在 UI 中明确显示为“待补数据”，而不是算成 0。

## 9. 进食记录与观察能力迁移

原有的 `scene / structure / composition / origin / trigger / value_density / body_feedback` 不再作为模块主设计，而是迁移为“进食记录和回顾”的字段。

### 新版进食记录要回答

| 问题 | 字段 |
|---|---|
| 原计划吃什么 | plannedSlotId |
| 实际吃了什么 | recipeId / foodId / freeText |
| 为什么变了 | changeReason |
| 吃完身体如何 | bodyFeedback |
| 这一顿值不值 | valueDensity |
| 这顿处在什么生活场景 | scene / trigger |
| 这顿唤起了什么味道记忆 | relatedFoodMemoryId |

原有观察能力中最有价值的是“为什么吃”和“吃完怎样”，它应该服务计划校准，而不是阻止用户建立食谱与营养系统。

        ▼

        │

               ▼
          MealLog[]

### FoodCategoryDefinition

| 字段 | 说明 |

| description | 说明 |

### FoodItem

| 字段 | 说明 |
|---|---|
| id | 食品 ID |

### FoodNutrientProfile

| 字段 | 说明 |
|---|---|
| id | 营养档案 ID |
| foodId | 食品 ID |

| basisUnit | g / ml |
| nutrients | 营养键值 |
| source | 手动 / 包装 / 食物成分表 / 外部导入 |

### Recipe

| 字段 | 说明 |

| 字段 | 说明 |
|---|---|

### MealLog

| 字段 | 说明 |

## 11. 页面交互原则

### 固定壳布局

饮食模块应遵守项目 [布局规范](../layout-spec.md)：

- 桌面端模块根容器固定壳，不整体滚动。
- 内部列表、详情、表格独立滚动。
- 窄屏回退为单列整体滚动。
- 所有滚动链路设置 `min-h-0`。

### 颜色和主题

- 不在饮食模块写固定颜色。
- 使用全局主题 token。
- 图表使用 `--color-chart-*`。
- 卡片、徽章、筛选态使用 `bg-card / bg-muted / bg-accent / border-foreground/10 / border-ring`。

### 信息密度

饮食模块会比旧版内容更多，因此要避免“长页面堆卡片”：

- 总览必须一屏优先。
- 食谱库、食品分类、营养成分表使用左列表 + 右详情。
- 每日计划使用日期栏 + 餐次列 + 右侧营养摘要。
- 表格视图只放在营养成分表，不把所有 Tab 都做成表格。

## 12. 与其他模块的关系

| 模块 | 关系 |
|---|---|
| 购物 | 食材采购可关联购物 system=饮食；每日计划可生成待买食材 |
| 记账 | MealLog 可关联餐饮支出，Recipe 可计算每份成本 |
| 情绪 | 情绪性进食、身体反馈、睡眠影响进入回顾 |
| 关系 | 聚餐、家庭味道、食物记忆可关联人 |
| 成长记忆 | 食物记忆可挂成长节点 |
| 总览 | 今日计划、营养缺口、饮食节奏进入全局总览 |

## 13. V1 范围

### V1 必须做

- 食品库与食品分类。
- 食谱库。
- 轻量营养成分表。
- 每日计划。
- 从每日计划生成实际进食记录。
- 总览一屏展示今天的饮食状态。

### V1 暂不做

- 外部食品数据库联网查询。
- 图片识别食物。
- 自动导入外卖订单。
- 医疗级营养建议。
- 精确热量目标和减肥曲线。

### 旧设计废弃说明

旧文档中“饮食模块不做食谱库、不做营养计算器”的定位废弃。新的边界是：

- 不做公共食谱平台，但做个人食谱库。
- 不做医疗级精算，但做轻量营养成分表。
- 不做强迫打卡，但做每日计划。
- 不取消进食观察，而是把它作为计划校准和回顾的一部分。
