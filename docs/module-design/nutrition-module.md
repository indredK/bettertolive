# 饮食模块 - 进食分类与营养建议

## 1. 30 秒看懂这份文档

这份文档建议给每一次进食打 **5 个分类标签** + **2 个评估属性**：

### 5 个分类标签（用来归类、过滤、看饮食世界）

| 标签 | 它回答什么 | 取值示例 |
|---|---|---|
| **scene**（场景） | 这顿饭出现在什么场景里？ | 在家做 / 外卖 / 堂食 / 路边 / 应酬 / 旅行 / 加餐零食（共 7 个，见 §4） |
| **structure**（结构） | 这顿饭在一天里是什么角色？ | 早餐 / 午餐 / 晚餐 / 加餐 / 夜宵 / 节庆餐 / 饮品 |
| **composition**（构成主轴） | 这顿饭的食物以什么为主？ | 主食为主 / 蛋白为主 / 蔬果为主 / 综合搭配 / 几乎只有碳水 / 几乎只有油盐糖 |
| **origin**（食材来源/风味） | 这顿饭的食材或做法属于什么? | 家常 / 地方菜系 / 异国料理 / 工业速食 / 自给自种 / 节令食材 |
| **trigger**（进食触发） | 为什么是这一刻吃、吃这一份？ | 准时按点 / 真饿了 / 社交场合 / 情绪驱动 / 习惯反射 / 不想浪费 / 看到就想吃 |

### 2 个评估属性（跟着每一次进食走，用来衡量"这一次"）

| 属性 | 它回答什么 | 取值示例 |
|---|---|---|
| **value_density**（性价比/营养密度比价格） | 这顿饭付出的钱换来了多少身体能用的东西？ | 高 / 中 / 低 / 不划算 |
| **body_feedback**（身体反馈） | 吃完之后身体有什么感觉？ | 满足舒服 / 普通 / 偏重偏胀 / 不适 / 想再吃 |

> **5 维和两个评估属性的区别**：5 维用来**归类**（"我外卖里高频出现哪类组合"）；评估属性用来**判断这一顿**（划不划算、身体喜不喜欢）。详见 §9。

打满这 5 个标签 + 2 个属性后,原来问不出的问题就能问出来了:

- "我把饮食预算花在了哪里？哪一类性价比最低？"
- "我哪些进食根本不是因为饿？"
- "我最近的饮食结构是失衡的吗？哪一类几乎缺席？"
- "什么样的饭吃完身体最舒服？什么样的不舒服？"

> **此外**:饮食立场(过敏/宗教/医学/选择不吃)和饮食意图是**用户档案级**设置,影响整个模块的解释方式；食物记忆是**独立内容类型**,不属于单次进食。两者都在 §11。

## 2. 为什么不按"食谱/营养素清单"分类

如果按"它是什么菜""含多少蛋白质碳水脂肪"来组织,会出现这样的列表：

> 番茄炒蛋 / 麻婆豆腐 / 沙拉 / 蛋白质 25g / 脂肪 18g / 碳水 60g / ……

这种分类的问题：

1. **算了营养却看不出生活**——蛋白够不够是一回事，"为什么我这周外卖里几乎没有蔬菜" 是另一回事
2. **回避了真正的入口**——大多数人并不是"不知道菜里有什么"，而是 "明明知道，但还是这样吃了"
3. **看不见钱的去向**——同样补蛋白，一份 38 元的鸡胸便当 和 一份 12 元的鸡蛋豆腐，性价比天差地别，但按"菜名"看不出
4. **抓不住情绪**——情绪性进食、习惯性进食、应酬性进食在动机上完全不同，但按食物名都长一样

### 换一个问法

不要问"这顿吃了什么菜",问 "**它是在什么场景里出现的？为什么在这一刻吃？身体后来怎么说？这笔钱花得值不值？**"

> **饮食模块的气质定位**：不做食谱库,不做卡路里精算,不做"标准三餐"教条。它是一个**让自己看清楚"吃"在生活里到底扮演什么角色**的入口——既照顾身体的客观需要,也照顾"为什么会想这样吃"的主观线索。

### 按这套维度分类的好处

- **看结构而不是看单顿**：看出"这周外卖占比 70%、蔬果几乎没出现"，比"今天少吃 200 卡"更能改变下一步
- **把钱和身体连起来**：value_density 让"花了多少 vs 拿到多少营养"可比,而不是只看价格或只看口味
- **看见情绪/习惯的入口**：trigger 让 "我为什么吃" 浮上水面,不必每次都靠自我反省
- **不跟下厨房比赛**：不储存食谱本身,而是让食谱/外卖/堂食/路边小吃都能进入同一张视图

## 3. 用三顿具体进食走一遍

### 例 1：加班晚上 9 点点的麻辣烫外卖

**5 个分类标签**:

| 维度 | 取值 | 为什么 |
|---|---|---|
| scene | **外卖** | 平台下单送到家 |
| structure | **晚餐**（偏夜宵） | 时间已经晚了,但本质是当天主餐 |
| composition | **综合搭配**（偏碳水油盐） | 有菜有肉有粉,但汤底重油重盐 |
| origin | **地方菜系**（川味改良） | 商家口味,但已经标准化 |
| trigger | **真饿了 + 情绪驱动** | 既是饿,也是"今天太累想吃点重口的" |

**评估属性**：

| 属性 | 取值 | 含义 |
|---|---|---|
| value_density | **中** | 38 元有蔬菜蛋白质,但油盐过重,身体能用的部分被抵消一部分 |
| body_feedback | **偏重偏胀** | 当晚睡眠变差,第二天上午口渴 |

> 现在你可以问:我所有 **trigger=情绪驱动** + **scene=外卖** 的进食里, **body_feedback** 大多落在哪一档?如果绝大多数都是"偏重偏胀/不适",这就是一个值得看见的模式。

### 例 2：周日中午自己煮的一碗杂粮饭 + 清蒸鱼 + 焯青菜

**5 个分类标签**：

| 维度 | 取值 | 为什么 |
|---|---|---|
| scene | **在家做** | 自己采购自己做 |
| structure | **午餐** | 正餐 |
| composition | **综合搭配** | 主食、蛋白、蔬菜齐 |
| origin | **家常** | 没有走任何菜系套路 |
| trigger | **准时按点** | 不是饿到极点,也不是情绪驱动 |

**评估属性**：

| 属性 | 取值 | 含义 |
|---|---|---|
| value_density | **高** | 食材成本约 20 元,蛋白、纤维、微量元素都有 |
| body_feedback | **满足舒服** | 下午精神稳定、不困、不馋零食 |

> 上面两顿进食 scene、trigger、value_density、body_feedback 几乎处处不同——这就是 **5 维互不替代**加上**两个评估属性穿透 5 维**的体现。从两条进食出发,可以推出:**当 scene=在家做 + trigger=准时按点 时,我的 body_feedback 普遍更好**——这是一个比"我应该多做饭"更具体的发现。

### 例 3：下午 3 点街边买的一杯全糖奶茶

**5 个分类标签**:

| 维度 | 取值 | 为什么 |
|---|---|---|
| scene | **路边/便利店** | 街边连锁奶茶店 |
| structure | **饮品** | 不构成一顿,单独消费 |
| composition | （饮品场景下相关性低,可不填） | 见 §5 关于"饮品"的特殊处理 |
| origin | **工业速食** | 高度标准化的工业制品 |
| trigger | **习惯反射** | 下午到点就想喝,不一定渴或饿 |

**饮品补充标签（structure=饮品 时启用）**:

| 标签 | 取值 | 含义 |
|---|---|---|
| beverage_kind | **含糖饮品** | 主要影响是糖 |

**评估属性**:

| 属性 | 取值 | 含义 |
|---|---|---|
| value_density | **不划算** | 28 元,几乎全是糖、奶精、茶味 |
| body_feedback | **想再吃** | 短暂满足,1 小时后又馋零食 |

> 例 3 体现的是**饮品作为独立 structure** 的处理:并不是所有"吃进去的东西"都是一顿饭,但带糖、酒、咖啡因的饮品依然要进入饮食视图——通过 `beverage_kind` 来分类,而不是硬塞进 `composition`。

## 4. 第 1 维：7 种饮食场景

### 为什么是 7 种

不是按"在哪个餐厅吃",而是按 **食物从哪里来 + 你处于什么参与度** 来切。同一道菜在不同场景里,带来的影响和钱的去向完全不同。

| 场景 | 含义 | 典型例子 | 看完整性的关键问题 |
|---|---|---|---|
| **在家做** | 自己采购、自己烹饪 | 早餐煮粥、晚上煮面 | "我有多少天真正在为自己做饭？" |
| **外卖** | 平台下单、配送到家/办公室 | 美团饿了么订餐 | "我对外卖的依赖在变深还是变浅？" |
| **堂食** | 走进餐厅坐下来吃 | 餐厅、咖啡馆正餐 | "我一周/月有几次坐下来好好吃饭？" |
| **路边/便利店** | 简餐、街边小吃、便利店速食 | 包子煎饼、便利店饭团 | "我有多少顿饭是顺手买的、不算数的？" |
| **应酬/聚餐** | 因为人而吃、不是因为饿 | 工作聚餐、家庭聚餐、朋友聚会 | "应酬正在多大程度上影响我的饮食结构？" |
| **旅行** | 在路上、在外地吃 | 出差、度假、探亲 | "我旅行时的饮食状态是放松还是失控？" |
| **加餐零食** | 不构成"一顿"的零食、饮料、水果 | 下午茶、办公室零食 | "零食在我一天的能量里占多大？我注意到了吗？" |

### 场景的交叉

- 在公司吃外卖 → scene=**外卖**（虽然在公司,但食物来源是外卖）
- 在家点外卖请朋友吃 → scene=**外卖**,但可在备注里标注社交性
- 旅行时在当地餐厅吃 → scene=**旅行**（旅行作为场景优先于堂食）
- 应酬中点的奶茶 → scene=**应酬**,不再单独算加餐
- 一个原则：**应酬 / 旅行 是覆盖性场景**,它们优先于其他场景。

## 5. 第 2 维：structure（结构）

`scene` 回答 "饭从哪来",`structure` 回答 "这顿在一天里是什么角色"。

### 7 种结构

| 结构 | 含义 | 时间常见区间 |
|---|---|---|
| **早餐** | 起床后的第一顿 | 6:00 - 10:00 |
| **午餐** | 一天里的主餐之一 | 11:00 - 14:00 |
| **晚餐** | 一天里的主餐之一 | 17:00 - 20:00 |
| **加餐** | 主餐之间的补充进食 | 任意,体量小 |
| **夜宵** | 晚餐之后、临睡前 | 21:00 之后 |
| **节庆餐** | 春节、生日、过年过节、纪念日的特殊饭 | 不固定 |
| **饮品** | 单独消费的饮品(咖啡、茶、酒、奶茶、果汁等),不含纯水和无糖无咖啡因的茶水 | 任意 |

### 饮品的特殊处理

并非所有饮品都进入饮食条目:**纯水、无糖无咖啡因的清茶**通常不需要单独记录(可在周/月概览设可选的"饮水"统计)。需要进入饮食条目的是带来 **能量摄入、咖啡因、酒精、糖** 这四类影响的饮品。

当 `structure=饮品` 时,补一个 **`beverage_kind`** 子标签:

| beverage_kind | 含义 | 典型 |
|---|---|---|
| **含咖啡因** | 主要影响是咖啡因 | 黑咖啡、美式、浓茶 |
| **含酒精** | 主要影响是酒精 | 啤酒、葡萄酒、白酒、鸡尾酒 |
| **含糖饮品** | 主要影响是糖 | 奶茶、可乐、果汁饮料、含糖咖啡 |
| **鲜榨果汁** | 整体接近食物 | 现榨果汁、蔬果汁 |
| **茶水** | 影响轻、但不算无 | 淡茶、花草茶 |
| **奶/豆奶** | 偏蛋白补充 | 牛奶、豆浆、燕麦奶 |
| **其他** | 不属于以上 | 运动饮料、电解质水 |

饮品场景下,`composition` 的相关性变弱(可不填),`beverage_kind` 成为主分类信号。**佐餐饮料**(配饭的可乐、配菜的酒)不另立条目,可在该餐 notes 里记录,或在详细模式中追加。

### 为什么需要这一维

- "我经常跳过早餐" 和 "我经常吃夜宵" 是两类完全不同的饮食模式,structure 是看出这类模式的最直接维度
- 同样是"夜宵",一个是周末偶尔放纵,一个是几乎每天 23 点必吃——后者才是要关注的
- structure 也是"节奏" 的体现:固定吃三餐 vs 一天 5 次进食 vs 跳餐 + 暴食,这些规律本身就是饮食状态

## 6. 第 3 维：composition（构成主轴）

`scene + structure` 回答了 "在哪、什么时候",`composition` 回答 "这顿饭以什么为骨架"。

### 6 种构成

| 构成 | 含义 | 典型例子 |
|---|---|---|
| **主食为主** | 主食占主导,蛋白/蔬菜较少 | 一碗面、一份盖浇饭主要吃饭 |
| **蛋白为主** | 蛋白质占主导 | 鸡胸沙拉、清蒸鱼配少量米 |
| **蔬果为主** | 蔬菜或水果占主导 | 大份沙拉、水果餐 |
| **综合搭配** | 主食、蛋白、蔬菜大致齐全 | 经典三菜一汤、便当盒 |
| **几乎只有碳水** | 一顿基本只有面食/米/糕点 | 一杯奶茶 + 一份蛋糕、纯方便面 |
| **几乎只有油盐糖** | 一顿基本是高油高盐高糖食物 | 烧烤、油炸、奶油甜点 |

### composition 为什么不是"营养素百分比"

营养素百分比需要每顿都精确录入克数,绝大多数人坚持不了。"构成主轴" 是粗粒度的视觉判断,门槛低,但已经足以发现"我大多数午餐都是几乎只有碳水"这类模式。

### composition 不能被 origin 替代

家常菜可以是"几乎只有碳水"（妈妈做的炒面）,异国料理可以是"蛋白为主"（牛排）。**做法/风味**和**结构骨架**是两件事。

## 7. 第 4 维：origin（食材来源/风味）

### 6 类来源

| 来源 | 含义 | 典型例子 |
|---|---|---|
| **家常** | 没有特别风味套路的日常做法 | 番茄炒蛋、青菜豆腐 |
| **地方菜系** | 川/粤/鲁/湘/江浙/北方面食/西北等 | 麻婆豆腐、白切鸡、馕、油泼面 |
| **异国料理** | 日韩、东南亚、欧美、中东等 | 寿司、拌饭、意面、汉堡、咖喱 |
| **工业速食** | 高度加工、即食的标准化产品 | 方便面、自热饭、能量棒、罐头 |
| **自给自种** | 自己种或自己采的食材为主 | 自家阳台菜、亲戚送的农产品 |
| **节令食材** | 当季当令、有时间窗口的食材 | 春笋、秋蟹、冬至饺子 |

### 为什么需要这一维

- 回顾"我对地方/异国饮食的接触在变窄还是变宽"——这其实是生活丰富度的一个侧面
- 工业速食占比是一个值得长期跟踪的健康信号
- 节令食材会自然带入季节感和地方感,是饮食和"活在哪里" 的联结
- 自给自种是少数人的入口,但对这一部分人,它就是生活方式的体现

## 8. 第 5 维：trigger（进食触发）

### 7 种触发

| 触发 | 含义 | 典型表现 |
|---|---|---|
| **准时按点** | 到时间了就吃,不一定饿 | 12 点准时午餐 |
| **真饿了** | 身体明确发出饥饿信号 | 饿到注意力下降 |
| **社交场合** | 有人在,就一起吃 | 聚会、家庭聚餐 |
| **情绪驱动** | 累了/烦了/开心/无聊 → 想吃 | 加班点宵夜、心情差吃甜的 |
| **习惯反射** | 到了某个场景自动就吃 | 看剧必配零食、咖啡必配点心 |
| **不想浪费** | 剩饭/快过期食物 | 冰箱里的边角料、家人剩下的 |
| **看到就想吃** | 视觉/嗅觉触发的偶发进食 | 路过面包店、外卖广告 |

### trigger 为什么是这个模块的灵魂之一

工具型饮食 app 几乎都不问 "为什么吃"。但很多人真正想改变的,不是 "吃什么",而是 "为什么一到这个时刻就要吃"。trigger 让进食的动机第一次浮上水面——这是这个模块和普通饮食 app 最大的不同。

### trigger 不需要也不应该带道德判断

"情绪驱动" 不是错的,"习惯反射" 也不是病。这一维只负责记录,不负责评判。判断在用户自己心里。

## 9. 两个评估属性：value_density 与 body_feedback

> **重要前提**：两个评估属性都**不是分类维度**,它们是穿透 5 维的进食内禀属性。
> 5 维用来**归类**(按场景看分布、按构成看结构、按触发看动机),两个评估属性用来**判断单次进食**(划不划算、身体喜不喜欢)。

### 5 维 vs 评估属性的区分

| 5 个分类标签 | 两个评估属性 |
|---|---|
| **组织饮食世界**：让你能问"我外卖里高频出现哪类组合" | **判断单次进食**:让你能问"这一顿钱花得值不值、身体舒不舒服" |
| 出现在过滤器、分组、统计里 | 跟着每条进食走,在进食详情/回顾里显示 |
| 用来回答"吃了什么、在哪吃、为什么吃" | 用来回答"这一顿对我意味着什么" |

### value_density（性价比/营养密度比价格）

这不是"卡路里 ÷ 价格",而是 **"身体能用的东西 ÷ 付出的钱"** 的主观判断。一顿主要是水和油盐糖的奶茶 + 蛋糕,即便便宜,value_density 也不高。

| 等级 | 含义 | 典型例子 |
|---|---|---|
| **高** | 钱花得少 + 身体拿到的多;或钱不算少但拿到的足够多 | 自己做的杂粮饭蔬菜蛋白俱全;38 元便当蛋白蔬菜齐 |
| **中** | 一分钱一分货,不亏 | 家常外卖、商务餐 |
| **低** | 钱花了,但身体能用的不多 | 高价但主要是油盐糖的精致甜品 |
| **不划算** | 花了钱,身体几乎拿不到什么,甚至带来负担 | 网红奶茶 + 蛋糕组合;高价低质食物 |

为什么不强制要求录入克数和价格:绝大多数人不会坚持精确录入。**给一个 4 档主观选择,门槛低,样本量大,模式才能浮现。**对希望更精确的用户,留出"详细模式"作为可选,可填具体价格和大致蛋白/碳水/蔬菜量。

### body_feedback（身体反馈）

吃完一段时间后身体的整体感觉。要等饭后一段时间再填,而不是吃完立刻填。

| 等级 | 含义 | 典型表现 |
|---|---|---|
| **满足舒服** | 饭后精神稳定,不困、不馋、状态平稳 | 经典"刚刚好"的状态 |
| **普通** | 没什么特别感觉,不好也不差 | 大多数日常餐 |
| **偏重偏胀** | 饭后困重、胃胀、想躺、口渴 | 油盐过重或吃太多 |
| **不适** | 反酸、胀气、腹泻、过敏反应、心跳加速 | 身体明确抗议 |
| **想再吃** | 吃完没多久又饿、馋零食 | 几乎只有碳水的一餐之后常见 |

### 两个评估属性能回答的问题

| 问题 | 怎么用 |
|---|---|
| 我哪类组合性价比最差? | 按 composition + scene 分组,看 value_density 平均档 |
| 哪些场景下我身体反馈最好? | 按 scene 分组,看 body_feedback 分布 |
| 我有没有"看起来很贵但不划算"的高频选择? | 过滤 value_density=不划算,按 origin 看是不是某类异国料理/网红甜品在反复出现 |
| 哪类 trigger 之后我身体反馈最差? | 交叉 trigger × body_feedback,看 trigger=情绪驱动/习惯反射 是不是和"偏重/不适"高度相关 |

### UI 呈现建议

- ✅ 进食详情页**始终显示** value_density 和 body_feedback
- ✅ 周/月回顾里做 "**scene × value_density**" 和 "**trigger × body_feedback**" 两张交叉视图
- ✅ value_density 可补一个可选的 "实际花费(元)" 字段,有数据时自动汇总到记账模块（见 §14）
- ❌ **不要**把两个评估属性放进主筛选器(那是 5 维的位置)
- ❌ **不要**对 body_feedback=不适 的进食弹"你不应该再吃 XX"的提示(没有谁是被这种提示改变的)

## 10. 营养维度：进食之上的"周/月看板",而不是逐顿精算

这个模块**不要求每顿录入营养素克数**,但保留一个**轻量周/月营养概览**作为可选视图,从已录入的 composition + scene + origin 推算粗粒度的营养结构。

### 概览要回答的核心问题

| 问题 | 怎么算 |
|---|---|
| 这周我的蛋白质摄入大致够吗? | 统计 composition 含 "蛋白为主/综合搭配" 的进食占比 |
| 蔬果有没有缺席? | 统计 composition 含 "蔬果为主/综合搭配" 的进食天数 |
| 工业速食占比是多少? | origin=工业速食 的进食占比 |
| 油盐糖的高负担餐次占比 | composition=几乎只有油盐糖 的进食次数 |
| 我的进食节奏是规律还是失衡? | 看 structure 在一周内的分布是否稳定 |

### 为什么不做精确营养计算

- 精确计算需要每顿录入克数 → 几乎没人能坚持
- 即便算准,日常生活也不靠 "今天蛋白 67g、明天 73g" 来调整
- 真正能改变行为的是**结构性发现**："我这周几乎没吃蔬菜" 比 "你今天纤维少 3g" 有用得多

### 给愿意更精确的用户：详细模式

可以提供 **可选的"详细模式"**:对单顿进食补充录入大致克数 / 主要食材 / 实际花费,系统据此给出更精细的 value_density 和营养结构。**默认轻量、可选精细**——不强迫,但留口子。

## 11. 进食之外：饮食立场与食物记忆

5 维 + 2 个评估属性覆盖了"每一次进食"。但有两类内容不属于单次进食,却深深影响饮食模块的运作:**用户层面的立场/意图**和**作为独立内容的食物记忆**。

### 11.1 饮食立场（用户档案级,一次设置长期生效）

不是每顿要打的标签,而是**用户层面的设置**,影响周/月概览的解释、提示语和性价比基线。分两层:

**硬约束（不能吃）**

| 类型 | 含义 | 例子 |
|---|---|---|
| **过敏** | 食用会引发不良反应 | 海鲜过敏、坚果过敏、麸质不耐受、乳糖不耐 |
| **宗教/文化** | 信仰或文化约束 | 清真、犹太洁食、印度教不食牛肉 |
| **医学限制** | 医嘱限制 | 糖尿病限糖、痛风限嘌呤、高血压限钠、肾病限蛋白 |

**软立场（选择不吃）**

| 类型 | 含义 | 例子 |
|---|---|---|
| **饮食方式** | 长期选择 | 素食、纯素、低碳水、生酮、地中海饮食 |
| **进食节律** | 时间维度选择 | 间歇性断食(16:8 / 5:2)、过午不食 |
| **临时关注** | 当前阶段在意 | 减糖、增加蛋白、戒酒 |

### 11.2 饮食意图（可时间窗口化）

短期内"我希望吃成什么样子",影响 value_density 的解释和周回顾的提示语。

| 意图 | 含义 |
|---|---|
| **维持（默认）** | 不刻意调整,保持现状 |
| **稍微注意** | 软性关注,不严格 |
| **在调整具体问题** | 针对肠胃/睡眠/皮肤/能量/体重等具体目标在调整,可填一段说明（如"看胃酸反流是不是跟咖啡有关"） |
| **阶段性需求** | 减脂期、增肌期、孕期、哺乳期、术后恢复期、考试冲刺期 |

意图可以为空（等同于"维持"），也可以带时间窗口（"接下来 3 个月"）。

### 立场与意图如何影响整个模块

- **value_density 基线变化**：对纯素者,蛋白主要看豆制品/坚果而不是按肉类基线判断;对低碳者,主食为主的午餐 value_density 自动下调
- **周回顾的语言**：对断食者,"早餐覆盖率低"不是缺口而是设计;对低碳者,"几乎只有碳水"是异常信号而非常态
- **过敏/医学约束的红线提示**：若记录里出现明确触发立场的内容（海鲜过敏者吃了含虾的菜）,详情页可给一个温和的事实标注,不催促不指责
- **不替用户做判断**：这些立场是用户告诉系统的事实,系统不质问"为什么要素食"、不推荐"放弃低碳"

### 11.3 食物记忆（独立内容类型,不属于单次进食）

有些食物存在的方式不是"某一次吃过",而是 **"一类东西在我心里有特殊位置"**。这类内容不进 MealEntry,而是单独的 `FoodMemory` 条目。

**5 种类型**

| 类型 | 含义 | 例子 |
|---|---|---|
| **家庭味道** | 与家人/家庭氛围绑定 | 妈妈做的红烧肉、奶奶煮的粥、爸爸的拿手菜 |
| **地方味道** | 与某个地方/家乡绑定 | 家乡的某道菜、读书时学校楼下的小店 |
| **关系味道** | 与某段关系/某个人绑定 | 和恋人第一次约会的餐厅、和老友常去的烧烤摊 |
| **人生节点味道** | 与某个时期/节点绑定 | 高考前反复吃的、产后第一顿饭、出国前最后一顿家乡菜 |
| **已经吃不到的** | 因任何原因再也无法吃到 | 已经搬走的店、已不在的人做的菜、配方失传的 |

**每条食物记忆承接**

| 内容 | 说明 |
|---|---|
| 食物本身 | 名称、关键味道描述、配料/做法（如能复刻） |
| 背后的故事 | 第一次吃在什么场景、和谁、那时自己什么状态 |
| 关联条目 | 关系条目（谁做的/和谁一起吃）、成长记忆条目（哪个人生节点） |
| 当前状态 | 仍能吃到 / 偶尔能吃到 / 已经吃不到 / 自己尝试复刻 |
| 情感重量 | emotional_load：很重 / 中等 / 轻微 / 平静（参考[生命整理模块](./life-organization-module.md) §7 的取值） |

**与进食条目的关系**

- 进食条目记录"这一次吃了什么"
- 食物记忆记录"这一类食物在我心里的位置"
- 一次具体进食可以**关联**到一条食物记忆（如"今天复刻了妈妈的红烧肉"）
- 食物记忆不必依赖任何一次进食条目存在——它独立成立

**为什么单独成类**

食物记忆有两个特性让它无法塞进进食条目:

1. 它跨越多次进食,不属于任何一次
2. 它的核心是关系、记忆、地方,而不是 scene/structure/composition

这和[生命整理模块](./life-organization-module.md)中"留给某人的话"独立成类的逻辑一致——有些内容需要自己的位置,而不是塞进别的容器。

## 12. 边界约定

### 决策规则

> 先问"**这顿饭最主要的场景是什么**"(scene),再问"**它在一天里是什么角色**"(structure),再看"**它是怎么搭起来的**"(composition + origin),最后问"**我为什么在这一刻吃**"(trigger)。
> 如果一顿饭同时涉及多个场景(比如旅行中的应酬),按 §4 的覆盖性规则处理。

### 饮食 vs 记账模块中的"吃" 分类的区分

| | 饮食模块 | 记账模块的"餐饮" |
|---|---|---|
| **关注点** | 吃了什么、为什么、身体怎么说 | 这笔钱花在了哪一类支出 |
| **粒度** | 每一次进食 | 每一笔支付 |
| **覆盖范围** | 包括在家做、自己带饭、零食等不一定花钱的进食 | 只覆盖真正花钱的那部分 |
| **联动方向** | 提供"吃了什么"的语义 | 提供"花了多少"的金额 |

> 同一笔外卖,在饮食模块是一次进食条目,在记账模块是一笔餐饮支出。两者可以双向关联(进食条目挂账单 ID,账单条目挂进食 ID)。

### 饮食 vs 情绪模块中"情绪性进食" 的分工

- **饮食模块**：记录 trigger=情绪驱动 的进食事件本身（吃了什么、什么场景、身体反馈如何）
- **情绪模块**：记录情绪本身(波动、触发因素、恢复方式)
- 当一次进食 trigger=情绪驱动 时,该进食条目可关联当时的情绪条目;周回顾时可以看到"我的情绪低谷期 × 进食模式" 的对应关系

### 常见归属疑问

| 情景 | scene 归属 | 理由 |
|---|---|---|
| 公司请客户在餐厅吃饭 | 应酬 | 因人而吃,不是因为饿 |
| 出差期间公司订的外卖 | 旅行 | 旅行优先于外卖 |
| 自己买菜让家人做 | 在家做 | 食物来源在家庭 |
| 公司食堂 | 堂食 | 等同于走进餐厅 |
| 自己带便当到公司 | 在家做 | 食物是在家做的,带到哪吃不重要 |
| 在便利店买饭团坐下吃 | 路边/便利店 | 便利店餐归这一类 |
| 朋友家做饭叫去吃 | 应酬/聚餐 | 因人而吃 |
| 一杯水 | 不进入饮食模块 | 仅水不构成"进食条目",但可在周回顾里看到"饮水"另一类(可选) |
| 一杯黑咖啡 / 一杯啤酒 | 进入,structure=饮品 | 带咖啡因/酒精,有显著影响 |
| 服用维生素/药物 | 不进入饮食模块 | 不是进食,可走"健康"或"用药"另开模块 |

### 不属于本模块的内容

- 减肥计划、目标体重设定、运动消耗(那是健康/运动模块的事)
- 食谱本身的存储(它不是这个模块要解决的问题,见 §13)
- 食材库存管理 / 冰箱有什么(可与购物模块联动,但不在本模块内做)
- 餐厅评分、菜单收藏(走"地点/记事"模块更合适)

## 13. 关于"食谱/地方美食/营养搭配"这三块,怎么取舍

这是用户最初提的三个工具型方向。本模块**不直接做食谱库**,但用以下方式承接相关需求:

### 食谱

- ❌ 不内置大规模食谱库（下厨房、小红书已经做得很好）
- ✅ 但支持给一次"在家做"的进食**挂一段自由文本作为"做法记录"**——下次想复刻能找到
- ✅ 这些做法记录会自然形成"我自己的家常本",而不是一个公共食谱库

### 地方美食

- ❌ 不做百科式介绍
- ✅ 通过 origin=地方菜系/节令食材 维度,让"我吃过哪些地方/季节的东西" 自然浮现
- ✅ 旅行场景下的进食单独成集,自然形成"旅行饮食地图"
- ✅ 真正特别的地方味道,可升级为一条"地方味道"类的食物记忆(见 §11.3)

### 营养搭配

- ❌ 不做"每日推荐摄入量 vs 你今天的"这种计算
- ✅ 通过 composition 的轻量分类 + 周/月概览,告诉用户**结构性的缺口**(蔬果几乎缺席、蛋白连续几天不足等)
- ✅ 营养建议会**根据 §11.1 的立场和 §11.2 的意图调整基线**——纯素者和低碳者看到的不会是同一套语言
- ✅ 不给"你应该吃 XX" 的建议,而给"你最近几乎没出现的是 XX" 的事实描述

这种取舍的原则:**不替代下厨房和营养计算器,提供它们不提供的"自我观察"维度。**

## 14. 与其他模块的关系

| 关联模块 | 联系方式 |
|---|---|
| **记账** | 进食条目可关联一笔餐饮支出;周回顾可看 "饮食支出/value_density 平均档" 帮助判断钱花得值不值 |
| **购物** | 在家做的食材采购可在购物模块标记 system=饮食;两边互相佐证"我在家做的频率是否真的提高了" |
| **情绪** | trigger=情绪驱动 的进食可关联当时的情绪条目;周回顾交叉看"情绪低谷 × 进食模式" |
| **关系** | scene=应酬/聚餐 的进食可记录 **companions**(和谁一起吃),关联具体的关系条目;§11.3 食物记忆中的"关系味道""家庭味道"直接挂靠关系条目;长期看可以发现"和谁一起吃身体反馈普遍更好" |
| **成长记忆** | §11.3 食物记忆中的"人生节点味道""已经吃不到的"通常对应一条成长记忆;反过来,一条成长记忆里提到的食物可升级为独立的食物记忆 |
| **情绪** | 饮食可作为"是否在自我照顾"的客观信号之一——连续多天 跳餐 / 几乎只有碳水 / 大量工业速食 可能与状态恶化相关 |
| **节奏总览** | 饮食的节奏(规律/外卖依赖/暴食)进入总览页,和睡眠、情绪、运动等并列 |

## 15. 典型问题 → 怎么用回答

| 你的真实困惑 | 用哪几个维度 / 属性 |
|---|---|
| "我把饮食预算花在了哪里？" | 按 scene × value_density 分组,看高金额低密度的集中在哪 |
| "我哪些进食根本不是因为饿？" | 过滤 trigger=情绪驱动/习惯反射/不想浪费/看到就想吃 |
| "我的饮食结构是失衡的吗？" | 周/月看 composition 分布,看蔬果/蛋白是否长期缺席（按 §11.1 立场调整基线） |
| "什么样的饭吃完身体最舒服？" | 按 scene + composition + origin 交叉,看 body_feedback=满足舒服 的集中模式 |
| "我外卖依赖在加深吗？" | 按月看 scene=外卖 的占比变化趋势 |
| "情绪低谷时我吃了什么？" | 关联情绪模块,过滤情绪低谷时段对应的进食条目 |
| "我对地方/异国食物的接触在变窄吗？" | 按月看 origin 的多样性变化 |
| "我跳早餐多少次了？" | 按 structure=早餐 的覆盖率统计 |
| "哪些组合花了钱但身体不感谢？" | 过滤 value_density=低/不划算 + body_feedback=偏重偏胀/不适 |
| "应酬餐对我状态的影响有多大？" | 过滤 scene=应酬,看 body_feedback 分布;与非应酬日对比第二天状态 |
| "在家做和外卖,哪个让我状态更好？" | 按 scene 分组,看 body_feedback + 性价比对比 |
| "我有没有'看起来贵但其实没用'的高频饮食？" | 过滤 value_density=不划算 高频出现的 origin/composition |
| "我咖啡因/酒精摄入多吗？" | 过滤 structure=饮品 + beverage_kind=含咖啡因/含酒精,按周/月看次数 |
| "和谁一起吃身体反馈最好？" | 按 companions 分组,看 body_feedback 分布 |
| "我有没有违反自己饮食立场的进食？" | 对照 §11.1 立场设置,筛选触发硬约束/软立场的记录 |
| "我心里最重的几个食物记忆是什么？" | 看 §11.3 FoodMemory 中 emotional_load=很重 的条目 |
| "有没有我想复刻但还没尝试的食物记忆？" | 过滤 FoodMemory.当前状态=已经吃不到 + 备注里有"想复刻"的 |

## 16. 实施步骤（给开发者）

### 进食条目（MealEntry）

1. 把 7 种场景作为 `MealScene` 枚举
2. 把 7 种结构作为 `MealStructure` 枚举（包含 饮品）
3. 把 7 种饮品类型作为 `BeverageKind` 枚举（仅当 structure=饮品 时启用）
4. 把 6 种构成作为 `MealComposition` 枚举
5. 把 6 类来源作为 `MealOrigin` 枚举
6. 把 7 种触发作为 `MealTrigger` 枚举
7. 把 4 级 value_density 作为 `ValueDensity` 枚举
8. 把 5 级 body_feedback 作为 `BodyFeedback` 枚举
9. 在 `MealEntry` 类型上：
   - `scene: MealScene`（必填）
   - `structure: MealStructure`（必填）
   - `beverageKind?: BeverageKind`（structure=饮品 时必填,其他场景留空）
   - `composition?: MealComposition`（必填,但 structure=饮品 时可空）
   - `origin: MealOrigin`（必填）
   - `trigger: MealTrigger`（必填）
   - `valueDensity?: ValueDensity`（可选,可吃完后或第二天补填）
   - `bodyFeedback?: BodyFeedback`（可选,需要饭后一段时间再填）
   - `cost?: number`（可选,作为 value_density 的客观参考）
   - `companions?: PersonRef[]`（可选,引用关系模块条目）
   - `relatedFoodMemoryId?: string`（可选,关联到一条食物记忆）
   - `notes?: string`（自由文本,做法记录/食材/感受）

### 用户档案（DietaryProfile）

10. 新增用户档案级结构 `DietaryProfile`:
    - `hardConstraints: HardConstraint[]`（过敏/宗教文化/医学）
    - `softStances: SoftStance[]`（饮食方式/进食节律/临时关注）
    - `currentIntent: DietaryIntent`（维持/稍微注意/在调整具体问题/阶段性需求）
    - `intentNote?: string`（意图的自由说明）
    - `intentWindow?: { start: Date; end?: Date }`（意图时间窗口,可选）
11. 档案变化保留历史,可看"我从什么时候开始素食"这类轨迹

### 食物记忆（FoodMemory）

12. 独立类型 `FoodMemory`:
    - `type: FoodMemoryType`（家庭味道/地方味道/关系味道/人生节点味道/已经吃不到的）
    - `name: string`（食物名）
    - `flavorDescription?: string`（味道描述）
    - `recipe?: string`（配料/做法,如能复刻）
    - `story: string`（背后的故事）
    - `currentAvailability: 仍能吃到 | 偶尔能吃到 | 已经吃不到 | 自己尝试复刻`
    - `emotionalLoad: EmotionalLoad`（复用生命整理模块的 4 级取值）
    - `relatedPeople?: PersonRef[]`（关联关系条目）
    - `relatedJourneyNodes?: JourneyNodeRef[]`（关联成长记忆条目）

### 详细模式与回顾

13. 详细模式：支持单条进食补充录入大致克数 / 主要食材 / 实际花费,系统据此推算更精细的营养结构和 value_density
14. 进食条目支持双向关联：记账条目、情绪条目、关系条目、食物记忆条目
15. 周/月回顾视图至少包含：
    - scene 占比 / composition 分布 / origin 多样性
    - structure 节奏（早餐覆盖率、夜宵频次、饮品次数等）
    - scene × value_density 交叉
    - trigger × body_feedback 交叉
    - "本周几乎没出现的"（蔬果/蛋白/家做）—— **根据 DietaryProfile 调整基线**
    - 咖啡因/酒精/含糖饮品摄入次数

### 模块气质

16. UI 层注意：**5 维**进入过滤器/分组/统计；**两个评估属性**进详情页和交叉视图,不做集合过滤；**立场和意图**进设置页,不出现在每顿录入流程里
17. 模块整体气质：不催促、不打卡、不评判；允许长时间不录入而不被提醒"你已经 X 天没记录了"；硬约束触发只做温和事实标注,不弹警告

## 17. 不在本文档讨论的范围

- 单条进食的录入交互、表单细节
- 食材成分数据库的接入方式（GB 28050 食物成分表/外部 API）
- 大致克数 → 营养素的换算算法
- 食物图片识别 / 自动从外卖订单导入的实现方式
- 专业营养建议（本模块仅提供"自我观察"维度,不替代营养师/医生指导,孕期/哺乳期/疾病期等阶段性需求的具体方案需走外部专业渠道）
- 数据库表结构、API 设计
- 与第三方应用（美团、饿了么、下厨房）的数据互通
