# 生命整理模块 - 告别与托付分类建议

## 1. 30 秒看懂这份文档

这份文档建议给每一条生命整理内容打 **5 个分类标签** + **1 个评估属性**：

### 5 个分类标签（用来归类、过滤、看整理进度）

| 标签 | 它回答什么 | 取值示例 |
|---|---|---|
| **category**（内容类别） | 这是一份什么性质的内容？ | 重要交代 / 留给某人的话 / 人生回顾 / 未完成的事 / 纪念偏好（共 5 个，见 §4） |
| **recipient**（接收者） | 这份内容写给谁？ | 特定的人 / 家人 / 朋友 / 公开 / 仅自己 |
| **urgency**（紧急度） | 有多需要被优先知道？ | 关键信息 / 重要 / 锦上添花 / 可选 |
| **visibility**（可见时机） | 什么时候可以被看到？ | 现在 / 某个时间后 / 我离世后 / 条件触发 |
| **status**（完成状态） | 写完了吗？ | 草稿 / 基本完成 / 已完成 / 会持续更新 / 最终版 |

### 1 个评估属性（跟着每份内容走，用来衡量"这一份"）

| 属性 | 它回答什么 | 取值示例 |
|---|---|---|
| **emotional_load**（情感负荷） | 写这份内容时有多重？ | 很重 / 中等 / 轻微 / 平静 |

> **5 维和情感负荷的区别**：5 维用来**归类**（"我有哪些写给家人的话"）；情感负荷用来**评估**（"写这段话我需要什么心理准备"）。详见 §7。

打满这 5 个标签 + 1 个属性后，原来问不出的问题就能问出来了：

- "我有哪些重要的事还没交代？"
- "哪些话我该趁着来得及说？"
- "我的生命整理完成到什么程度了？"
- "哪些内容对我来说最重、需要更多心力去面对？"

## 2. 为什么不是一份"遗书"

很多人听到"生命整理"第一反应是"写遗书"。但真实的心理需求通常不是一份法律文件，而是：

- 有些重要的话想留下，但平时没有合适位置安放
- 有些信息只存在自己脑子里，万一出事别人完全不知道
- 有些话不一定马上给别人看，但值得被保存
- 想回看自己这一生在意过什么、坚持过什么、遗憾过什么

### 换一个问法

不要问"我死后别人该知道什么"，问"**有哪些事情如果没交代清楚，我会觉得不安？有哪些话如果没说出来，我会觉得可惜？**"

> **这个模块的气质定位**：冷静、有尊严、私密、不煽情。它不是围绕死亡想象展开的页面，而是帮用户在活着的时候把重要的事慢慢安放好的空间。

### 按这套维度分类的好处

- **分清轻重缓急**：关键信息（账号、联系人、文件线索）和情感表达（留给某人的信）性质完全不同，分开管理
- **追踪完成度**：知道"我还有 3 份草稿没写完，2 份重要的还没开始"
- **隐私分层**：有些内容现在就能给人看，有些只在特定条件下交付
- **看见自己的在意**：按 category 分布能看到——我留下的内容更多是"对别人的交代"还是"对人生的回顾"

## 3. 用两份具体内容走一遍

### 例 1：一份写给伴侣的信

**5 个分类标签**：

| 维度 | 取值 | 为什么 |
|---|---|---|
| category | **留给某人的话** | 核心是情感表达，不是信息交代 |
| recipient | **特定的人（伴侣）** | 只有这一个人需要看到 |
| urgency | **重要** | 不是"不知道密码就进不去"那种关键，但情感上很重要 |
| visibility | **我离世后** | 现在不想给，但希望在那之后被看到 |
| status | **已完成** | 已经写好了，短期内不打算大改 |

**评估属性**：

| 属性 | 取值 | 含义 |
|---|---|---|
| emotional_load | **很重** | 写的时候哭了好几次，需要很大的心力才写完 |

> 现在你可以问：我有多少份**visibility=我离世后** + **emotional_load=很重**的内容？这些是需要心理准备才能打开写的。

### 例 2：一份重要账号和文件线索

**5 个分类标签**：

| 维度 | 取值 | 为什么 |
|---|---|---|
| category | **重要交代** | 核心是传递信息，不是情感表达 |
| recipient | **特定的人（指定联系人）** | 需要交给能处理这些事的人 |
| urgency | **关键信息** | 不知道这些就没法处理后续事务 |
| visibility | **我离世后** | 生前不需要给别人看 |
| status | **会持续更新** | 账号密码和文件位置会变，需要定期维护 |

**评估属性**：

| 属性 | 取值 | 含义 |
|---|---|---|
| emotional_load | **轻微** | 写起来不重，但很重要 |

> 两份内容 category、emotional_load 都不同，但都指向"离世后交付"——这就是分类维度在帮助区分不同性质的内容和不同的处理节奏。

## 4. 第 1 维：5 种内容类别

### 为什么是 5 种

按**这份内容的核心目的**来分——是传递信息、表达情感、回顾人生、安放遗憾，还是说明偏好。

| 类别 | 核心目的 | 典型内容 | 看完整性的关键问题 |
|---|---|---|---|
| **重要交代** | 传递必须被知道的信息 | 重要联系人、账号线索、文件位置、物品归属、需要先处理的事 | "如果明天我不在了，有谁能接手这些事？" |
| **留给某人的话** | 表达没说出口的情感 | 感谢、道歉、告别、解释、祝福、坦白 | "我对谁还有没说出来的话？" |
| **人生回顾** | 理解自己这一生 | 最重要的人、最珍贵的经历、最深的遗憾、最想被记住的样子 | "如果回头看，我最想留下什么理解？" |
| **未完成的事** | 安放还没做完的牵挂 | 还没见到的人、想修复的关系、想去的地方、未完成的承诺 | "哪些事如果不做完，我会放不下？" |
| **纪念偏好** | 说明希望被如何纪念 | 纪念方式、不希望出现的安排、物品去向、对葬礼的想法 | "我希望离开之后别人怎么对待我的离开？" |

### 类别的交叉

- 人生回顾里提到了某个人，又专门给这个人写了一封信 → 两份独立内容，category 不同
- 未完成的事里有一件是"想对某人说谢谢" → 如果写成了具体的信，就转为"留给某人的话"
- 纪念偏好里涉及物品归属 → 如果物品清单很长，物品归属拆到"重要交代"里

## 5. 第 2 维：recipient（接收者）

`category` 回答"这是什么性质的内容"，`recipient` 回答"写给谁"。

### 5 种接收者

| 接收者 | 含义 | 适用场景 |
|---|---|---|
| **特定的人** | 指名道姓写给一个人 | 给伴侣的信、给孩子的交代、给朋友的话 |
| **家人** | 写给家人群体 | 给"我的孩子们""我的家人" |
| **朋友** | 写给朋友群体 | 给"那些陪我走过的人" |
| **公开** | 不限定对象，可以被公开 | 人生回顾中愿意公开的部分 |
| **仅自己** | 只给自己看，不交付 | 完全私人的整理、还没准备好给任何人看的内容 |

### recipient 与 visibility 的关系

- recipient 回答"目标是谁"——这是内容层面的属性
- visibility 回答"什么时候交付"——这是隐私和时间层面的属性

同一个人（recipient=特定的人），可以是 visibility=现在 也可以 visibility=我离世后。

## 6. 第 3 维：urgency（紧急度）

`category + recipient` 说了"什么内容、给谁"，`urgency` 说"有多需要被优先知道"。

### 4 级紧急度

| 级别 | 含义 | 典型内容 | 缺失的后果 |
|---|---|---|---|
| **关键信息** | 不知道就无法处理后续事务 | 账号密码、保险信息、文件位置、关键联系人 | 别人完全找不到，事务瘫痪 |
| **重要** | 不是生存级别，但非常重要 | 重要的话、物品归属说明、未完成的承诺 | 会有遗憾，但不至于无法运转 |
| **锦上添花** | 有最好，没有也不影响大局 | 一些温柔的话、人生感悟分享 | 少了温暖，但不造成实际困难 |
| **可选** | 完全看心情，写了就好 | 个人偏好、详细的人生故事 | 完全没有后果 |

### urgency 不能被 category 替代

同样是"留给某人的话"——一封"感谢你陪我走过最难的时期"可能是"重要"，一封"记得你小时候喜欢吃的糖"可能是"锦上添花"。category 一样，urgency 不同，对应的心力分配也不同。

## 7. 生命整理内容的评估属性：emotional_load（情感负荷）

> **重要前提**：emotional_load **不是第 6 个分类维度**，它是穿透所有 5 维的**内容内禀属性**。
> 5 维用来**归类**（按类别看分布、按接收者管理、按紧急度排优先级），emotional_load 用来**评估单份内容**（打开写这份内容需要多大的心力）。

### 5 维 vs emotional_load 的区分

| 5 个分类标签 | emotional_load（评估属性） |
|---|---|
| **组织生命整理**：让你能问"我有哪些关键信息还没交代" | **判断单份内容**：让你能问"写这个我需要什么状态、什么准备" |
| 出现在过滤器、分组、总览统计里 | 跟着每份内容走，在内容详情里显示 |
| 用来回答"写了什么、给谁、什么时候能看" | 用来回答"我准备好面对这一份了吗" |

### 4 级取值

| 等级 | 含义 | 写作建议 |
|---|---|---|
| **很重** | 写的时候会触发强烈的情绪，需要很大心力 | 不要一次写完；选自己状态平稳的时候；写完需要自我照顾 |
| **中等** | 有一定重量，但可以承受 | 可以分几次写完，但不需要特别的情绪准备 |
| **轻微** | 有感觉但不深，顺手就能写 | 随时可以写 |
| **平静** | 完全不触发情绪，就是事实交代 | 适合在碎片时间完成 |

### 它能回答的问题

| 问题 | emotional_load 怎么帮 |
|---|---|
| 我该从哪一份开始写？ | 从 emotional_load=轻微/平静 的开始，建立节奏后再碰重的 |
| 为什么我一直拖着没写？ | 可能是 emotional_load 很高但自己没意识到 |
| 我今天状态适合写什么？ | 状态好 → 碰重的；状态一般 → 写轻微的 |
| 我最怕面对的是什么？ | emotional_load=很重 的列表就是答案 |

### UI 呈现建议

- ✅ 内容详情页**始终显示** emotional_load
- ✅ 总览页可按 emotional_load 分组显示，重的标温和的提醒色
- ✅ 从总览点进一份"很重"的内容时，可以有一个轻过渡/确认步骤（不是阻止，是给一个心理准备）
- ❌ **不要**把 emotional_load 放进主筛选器
- ❌ **不要**对"很重"的内容做推送提醒（"你还有没写完的"——这种提醒在别的地方有用，在这里不合适）

## 8. 第 4 维：visibility（可见时机）

### 5 级可见时机

| 时机 | 含义 | 触发条件 |
|---|---|---|
| **现在** | 已经可以给人看 | 内容完成后即可交付 |
| **某个时间后** | 到了特定时间点才解锁 | 如"孩子成年后""某年某月后" |
| **我离世后** | 生前不公开，离世后交付 | 需要配套的交付机制 |
| **条件触发** | 满足某个条件时才可见 | 如"如果我失能""如果某人先于我离开" |
| **永不交付** | 只写给自己看，永远不交付 | 纯私人的整理 |

### visibility 的严肃性

visibility=我离世后 或 visibility=条件触发 的内容，需要配套的交付机制。这不是技术"能不能做到"的问题，而是用户"信不信这个系统"的问题。详见 §11。

## 9. 第 5 维：status（完成状态）

### 5 种状态

| 状态 | 含义 | 行为 |
|---|---|---|
| **草稿** | 刚开始写，内容不完整 | 可以随时继续编辑 |
| **基本完成** | 主体内容写完了，可能会微调 | 可以标记为"差不多了" |
| **已完成** | 确认不再修改 | 可以锁定（如需修改要主动解锁） |
| **会持续更新** | 内容需要定期维护 | 适合"重要交代"中的账号清单等会变化的内容 |
| **最终版** | 最后一次更新，不再改动 | 锁定，附带时间戳和确认 |

## 10. 边界约定

### 决策规则

> 先问"**这份内容的核心目的是什么**"（category），再问"**给谁、什么时候能看、紧急吗**"（recipient + visibility + urgency）。
>
> 如果一份内容同时适合两个 category，选它**最主要想达成的目的**。

### 生命整理 vs 关系模块中"想说的话"的区分

| | 生命整理（留给某人的话） | 关系模块（想说的话） |
|---|---|---|
| **时间视角** | 面向未来/终点——"如果我不在了我想让你知道" | 面向当下/关系——"我现在想对你说但还没说" |
| **交付意图** | 有明确的交付时机（通常是离世后或特定条件） | 不一定要交付，更多是缓冲区 |
| **内容性质** | 总结性、告别性、交代性 | 即时性、过程性、日常性 |
| **典型句式** | "我想让你知道……""谢谢你曾经……" | "今天我想到……""有一件事我一直想解释……" |

关系模块的"想说的话"如果沉淀下来、有了明确的交付意图，可以升级为生命整理中的"留给某人的话"。反过来，生命整理中的内容也可以降级——"其实我只是想说，不一定要交付"。

### 常见归属疑问

| 内容 | category 归属 | 理由 |
|---|---|---|
| 保险单信息 | 重要交代 | 传递信息，不是表达情感 |
| 写给孩子的长信 | 留给某人的话 | 核心是情感表达 |
| "我这一生最感谢的事" | 人生回顾 | 对自己的理解，不是对某个人的话 |
| 还没去的旅行清单 | 未完成的事 | 挂念但还没做 |
| 希望葬礼放什么歌 | 纪念偏好 | 关于纪念方式的偏好 |
| 物品留给谁 | 重要交代 | 信息传递 |
| "对不起，那时候伤害了你" | 留给某人的话 | 对特定人的情感表达 |
| "如果有来生我想……" | 人生回顾 | 对人生的回看和想象 |

## 11. 信任与交付机制

生命整理模块的特殊之处在于：很多内容的交付条件是"我离世后"。这就带来了信任问题——用户需要相信系统会按他们的意愿执行。

### 需要说清楚的边界

- 哪些内容设为私密，即使本人也需要再次确认才能打开
- 哪些内容不参与 AI 分析、不出现在汇总视图里
- 交付条件（离世后、特定时间、条件触发）如何被验证和触发
- 导出时哪些可以带走，哪些只留在设备里
- 如果内容涉及他人，如何对待对方的隐私
- 失能或离世时，指定的人如何获得访问权限

### 设计原则

- 不解决信任问题，用户就不会写真实内容
- 让用户清楚知道"我交付的东西边界在哪里"
- 交付机制不是技术炫技，是整个产品的诚意底线

## 12. 典型问题 → 怎么用回答

| 你的真实困惑 | 用哪几个维度 / 属性 |
|---|---|
| "我有哪些事还没交代？" | 过滤 urgency=关键信息 + status≠已完成/最终版 |
| "我最该趁现在说的话是什么？" | 过滤 category=留给某人的话 + visibility=现在 + status≠已完成 |
| "我的生命整理完成到哪了？" | 按 status 统计 + 按 category 看覆盖度 |
| "有哪些话我还没准备好说？" | 过滤 emotional_load=很重 + status=草稿 |
| "哪些内容是需要定期维护的？" | 过滤 status=会持续更新（账号清单、联系人等） |
| "如果下个月就要离开，现在该补什么？" | 按 urgency 排序，优先补"关键信息"级别的 |
| "有没有什么话我可以现在就说？" | 过滤 visibility=现在 + status=已完成 |
| "我在躲什么？" | 看 emotional_load 很高但 status 仍是草稿的 |
| "我这一生最在意什么？" | 看 category=人生回顾 的内容 + 所有 category 中反复出现的人名/主题 |
| "有没有我忘了但很重要的人？" | 对照关系模块中的重要人物和 recipient 列表 |

## 13. 实施步骤（给开发者）

1. 把 5 种类别作为 `LegacyCategory` 枚举写出来
2. 把 5 种接收者作为 `LegacyRecipient` 枚举写出来
3. 把 4 级紧急度作为 `LegacyUrgency` 枚举写出来
4. 把 5 级可见时机作为 `LegacyVisibility` 枚举写出来
5. 把 5 种状态作为 `LegacyStatus` 枚举写出来
6. 把 4 级情感负荷作为 `EmotionalLoad` 枚举写出来
7. 在 `LegacyItem` 类型上：
   - `category: LegacyCategory`（必填）
   - `recipient: LegacyRecipient`（必填）
   - `urgency: LegacyUrgency`（必填）
   - `visibility: LegacyVisibility`（必填）
   - `status: LegacyStatus`（必填）
   - `emotionalLoad?: EmotionalLoad`（可选，刚创建时可能用户自己也不确定）
8. recipient=特定的人 时支持关联关系模块中的人物条目
9. visibility=我离世后 / 条件触发 的内容需要标记交付条件的具体描述
10. status=最终版 的内容锁定后需要主动解锁才能修改
11. UI 层：**5 维**进入过滤器/分组/总览；**emotional_load** 进详情页和总览的温和标注
12. 模块整体气质：冷静、克制、不推送催促

## 14. 与其他模块的关系

| 关联模块 | 联系方式 |
|---|---|
| **关系深化** | "留给某人的话"可以关联到关系条目；recipient 可引用关系列表中的人物 |
| **成长记忆** | "人生回顾"和记忆模块的内容有大量重叠——但视角不同：记忆模块关注"过去如何塑造我"，生命整理关注"我想留下什么理解" |
| **观念** | "人生回顾"中可能涉及"我相信什么"——可以关联到观念条目 |
| **原则** | "我希望别人怎么对待我的离开"本质上是最后一条原则 |
| **购物** | "物品归属"可以关联到购物模块中标记了情感价值的物品 |

## 15. 不在本文档讨论的范围

- 具体的交付验证机制（如何确认一个人"已离世"）
- 加密和存储方案
- 内容编辑的交互细节
