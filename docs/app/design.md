## 产品需求

### BetterToLive 产品需求文档（V1）

#### 1. 文档目的

本文档用于定义 BetterToLive V1 的产品方向与需求边界，重点回答以下四个问题：

1. 这个产品是什么
2. 它为什么存在
3. V1 要做什么
4. V1 暂时不做什么

本文档不讨论技术实现细节，不展开数据库设计，也不作为对外宣传材料使用。它服务于当前产品思考与后续实现落地。

#### 2. 产品定义

##### 2.1 产品愿景

BetterToLive 是一个帮助人看见自己、整理人生、靠近日常理想状态的私人空间。

它不是一个单纯的工具集合，也不是一个只用来写情绪日记的产品。它的目标是把记录、整理、回看、描画未来放在同一条主线上，让用户逐渐回答这些问题：

- 我过去是怎么走到这里的
- 我现在到底在过什么样的生活
- 我想去哪里，想成为什么样的人

##### 2.2 产品定位

BetterToLive 的统一定位是：

**一个本地优先、单人使用、以反思记录为主要入口的自我认知系统桌面 app。**

它通过 `过去 / 现在 / 未来` 的框架，帮助用户理解自己的人生轨迹、当下状态和未来方向。

##### 2.3 为什么现在做

很多生活内容虽然被记录了下来，但往往是分散的、断裂的、难以回看的。消费记录、待办事项、随手记下的想法、某天晚上的反思，通常各自存在，却很少被放在同一个生命视角里理解。

BetterToLive 想做的事，是把这些零散的内容重新组织起来，让记录不只是堆积，而是逐渐帮助用户认识自己。

#### 3. 要解决的问题

当前产品需要解决的问题主要有以下几类：

- 生活记录分散，难以形成连续的自我认知
- 人往往只在情绪高点或低点时才思考自己，缺少稳定的回看机制
- 现实行为和理想生活之间缺少连接，难以判断自己是否正在靠近真正想要的状态
- 日常工具很多，但很少有工具能把“我是谁”“我在追求什么”“我实际如何生活”放在一起看

#### 4. 目标用户

V1 只定义一个目标用户：产品创建者本人。

当前目标用户画像如下：

- 愿意长期记录自己的生活和想法
- 愿意回看过去，观察自己的变化
- 关心自我成长、人生方向和生活方式
- 不需要社交表达，更需要一个私密、稳定、可持续使用的空间

虽然 V1 只服务于个人，但产品表达应保留一定通用性，使其未来可以自然扩展到类似人群。

#### 5. 产品原则

BetterToLive V1 采用以下产品原则：

- **本地优先**：默认以本地使用为前提，不依赖在线服务才能成立
- **隐私优先**：记录内容默认为私人内容，不围绕分享或公开传播设计
- **低负担记录**：输入门槛要低，用户不需要每次都进行完整复盘
- **自由切换入口**：不强制规定唯一主流程，用户可以按当下需要进入任意模块
- **不锁定使用模式**：用户可以在不同入口、不同记录方式之间来回切换
- **先收集整理，再做洞察**：V1 先解决记录、归档、回看问题，不以自动分析能力为中心
- **所有模块服务于同一主线**：每个模块都应帮助用户更了解自己，而不是单独存在

#### 6. 核心理解框架：过去 / 现在 / 未来

BetterToLive 不是按工具类别组织自己，而是按用户对人生的理解过程组织自己。

##### 6.1 过去

过去关注的是：

- 发生过什么
- 我曾经怎样想
- 我曾经怎样生活
- 某些变化是如何逐渐发生的

对应的主要内容来源包括：反思、记事、记账、已完成或已发生的清单记录。

##### 6.2 现在

现在关注的是：

- 我此刻正在经历什么
- 我把时间、精力和金钱投向了什么
- 我现在的生活状态是否接近我真正想要的样子

对应的主要内容来源包括：近期反思、近期事件、近期消费、当前待办或待买事项。

##### 6.3 未来

未来关注的是：

- 我想成为谁
- 我想过怎样的生活
- 什么对我真正重要
- 接下来我想朝哪些方向靠近

这里的“未来”不是预测，而是对理想自我、理想生活和行动方向的定义。

#### 7. 使用方式与整体体验

V1 的整体体验应体现为一个可以自由进入和切换的私人空间，而不是固定流程工具。

核心约定如下：

- 首页展示全部核心入口
- 用户进入 app 后，不需要先完成某个步骤，才能使用其他功能
- 用户可以先写反思，也可以先记账、记事、查看购物模块或打开未来蓝图
- 用户在选择某种使用模式后，仍然可以随时切换到其他模块

这意味着首页是一个总览入口和切换中枢，而不是一个要求用户按顺序操作的任务面板。

#### 8. V1 信息架构

V1 的结构分为 `1 个总览首页 + 5 个核心功能模块`。

##### 8.1 总览首页

总览首页不是独立业务模块，而是承接全局体验的入口层。它负责：

- 展示全部核心功能入口
- 帮助用户快速决定此刻想做什么
- 承接“过去 / 现在 / 未来”的整体视角
- 提供模块之间的切换能力

##### 8.2 五个核心功能模块

1. 反思
2. 记事
3. 记账
4. 购物 / 生活物品
5. 未来蓝图

#### 9. V1 模块需求

##### 9.1 反思

**定位**

反思是 V1 最核心的主入口，用来承接情绪、想法、困惑、总结和阶段性复盘。

**V1 必备能力**

- 创建一条反思记录
- 支持自由书写
- 能看到记录时间
- 能被后续回看

**它解决的问题**

- 帮助用户表达当下状态
- 把模糊感受转成可被保留的文字
- 为后续理解自己留下连续轨迹

##### 9.2 记事

**定位**

记事用于记录事件、经历、片段和念头，帮助形成个人时间线。

**V1 必备能力**

- 创建一条事件记录
- 记录发生时间或记录时间
- 用简洁方式保存内容
- 后续可以按时间回看

**它解决的问题**

- 让生活不只剩下情绪和消费记录
- 帮助用户记住“发生了什么”
- 为回顾人生阶段提供客观片段

##### 9.3 记账

**定位**

记账不是为了做复杂财务管理，而是为了看见现实生活如何被消费行为塑造。

**V1 必备能力**

- 记录收入或支出
- 记录金额
- 记录类别
- 记录时间

**它解决的问题**

- 让金钱流向成为理解当下生活的重要线索
- 帮助用户观察消费习惯和价值取向
- 让“我实际上如何生活”变得更可见

##### 9.4 购物 / 生活物品

**定位**

购物模块用于承接生活物品分类、阶段清单、采购判断和系统化回看。它在 V1 更接近一个**只读内容工作台**，而不是以录入待买项为核心的操作工具。

**V1 必备能力**

- 能按 `system / space / stage / necessity / lifecycle` 理解生活物品
- 能查看阶段清单、spotlight 和价格判断参考
- 能从“生活系统是否完整”的视角回看当前生活基础设施
- 能承接少量物品样本与采购决策内容，但不要求形成高频录入流

**它解决的问题**

- 把原本分散的生活物品、缺口和购买判断放回同一套生活系统里理解
- 帮助用户看见“我现在的生活基础设施稳不稳”
- 为后续购物决策提供系统化参考，而不是只留下零散欲望清单

##### 9.5 未来蓝图

**定位**

未来蓝图用于记录理想自我、理想生活、重要价值和阶段方向，是产品中承接“未来”的核心模块。

**V1 必备能力**

- 记录想成为的人
- 记录想过的生活状态
- 记录重要价值和方向
- 记录阶段性的行动路径或靠近方式

**它解决的问题**

- 避免所有记录都只停留在过去和当下
- 为用户提供一个持续校准人生方向的地方
- 帮助用户把“想法”转成更清楚的自我定义

#### 10. 模块关系

V1 的五个核心功能模块不是并列小工具，它们共同服务于三个核心问题：

1. 我过去是怎么走到这里的？
2. 我现在到底在过什么样的生活？
3. 我想去哪里，想成为什么样的人？

各模块之间的关系可以概括为：

- `反思` 负责表达主观感受和内在思考
- `记事` 负责保留客观经历和事件片段
- `记账` 负责呈现现实资源流向
- `购物 / 生活物品` 负责呈现生活基础设施、日常需求和采购判断
- `未来蓝图` 负责定义理想方向与想靠近的生活

这些内容在长期使用中应逐渐形成一个更完整的个人画像。

#### 11. 术语与共享对象约定

本次文档不引入代码 API 变更，但需要统一产品层术语。

##### 11.1 术语约定

- `记录`：泛指用户输入到系统中的一切内容
- `回看`：对过去记录进行浏览、翻阅和复盘
- `未来蓝图`：对理想自我和理想生活的定义，不代表预测结果
- `模式切换`：用户可在首页和各模块之间自由进入、自由切换，不受固定流程限制

##### 11.2 共享对象建议

为了让不同模块未来可以被放回统一视角中理解，V1 建议各类记录默认具备以下基础属性：

- 时间
- 内容
- 标签或分类
- 状态（如适用）

后续实现时，应尽量让不同模块的记录能够被放回同一条时间线或同一组人生主题中理解，但 V1 不要求在产品文档中细化到数据库 schema。

#### 12. MVP 范围

V1 采用“全部轻量纳入”的策略。

这意味着：

- 五个核心功能模块都会出现
- 每个模块都只要求形成最基础的记录与回看闭环
- 重点是建立统一产品心智，而不是先把某一个模块做深

V1 成立的标准，不是功能数量多，而是用户已经可以借由这些模块逐渐回答“我是谁、我正在怎样生活、我想成为什么样的人”。

#### 13. 后续版本方向

以下内容进入后续版本路线，而不作为 V1 必做项：

- `性格测试`：作为辅助画像模块，用于补充自我认知，而不是主线入口
- `自动洞察`：例如趋势提炼、主题归纳、变化总结
- `周期报告`：例如周报、月报、阶段性过去/现在/未来对比
- `教练式提问`：通过连续问题帮助用户进一步思考
- `同步/导出`：例如多端同步、内容导出、阶段总结分享

#### 14. 非目标

V1 明确不做以下方向：

- 不做社交产品
- 不做多人协作产品
- 不做公开社区
- 不把 V1 定义为 AI 对话产品
- 不把性格测试作为第一主线
- 不优先解决云同步、账号体系、分享机制等能力

#### 15. 验收标准

当以下条件都成立时，可以认为这份 V1 产品定义是清晰的：

- 实施者读完文档后，不需要再判断它到底是日记 app、工具箱还是人生系统
- 实施者能明确总览首页只是入口层，五个核心功能模块才是业务主体
- 实施者能明确首页不应强制单一路径，而应支持自由进入与切换
- 实施者能明确 V1 的重点是记录与整理优先，而不是 AI 洞察优先
- 实施者能明确性格测试不属于首版核心范围
- 实施者能理解每个模块为什么存在，以及它们如何共同服务于“更了解自己”这条主线

#### 16. 默认假设

- 文档默认服务于产品创建者本人和后续实现者
- 文风采用冷静、真诚、偏产品定义的写法
- 平台默认是当前仓库对应的 Tauri 桌面 app
- 默认使用场景是纯本地、私人使用
- 除非后续单独提出，本版本不补充云同步、账号体系、分享机制等额外约束

#### 4. 共享对象建议

V1 虽然不细化数据库设计，但页面结构需要依赖一组统一对象观念。

##### 4.1 记录类对象

所有记录类内容建议默认包含以下字段：

- `id`
- `type`
- `createdAt`
- `occurredAt`
- `content`
- `tags`
- `status`

##### 4.2 模块对象补充

- `反思记录`：可带 `mood`、`summary`
- `事件记录`：可带 `title`
- `账目记录`：可带 `amount`、`direction`、`category`
- `购物模块内容`：可带 `stage`、`priceExpectation`、`system`
- `未来蓝图项`：可带 `theme`、`targetState`、`nextStep`

#### 5. 关键页面职责

##### 5.1 首页 / 总览

**目标**

- 帮用户在最短时间进入今天最想做的事
- 同时让用户感到自己正在看的是一个完整人生系统，而不是工具集合

**必须承接的内容**

- 顶部欢迎语与当天时间
- 一组可直接点击的主入口
- `形成 / 过去 / 现在 / 未来` 摘要
- 最近记录列表
- 页面逻辑说明
- 快速记录按钮

**不承担的内容**

- 不做复杂编辑
- 不替代模块页完成深度输入

##### 5.2 反思页

**目标**

- 成为日常最自然的表达入口

**必须承接的内容**

- 新建反思输入区
- 最近反思列表
- 按日期回看

##### 5.3 记事页

**目标**

- 让事件和经历形成连续时间线

**必须承接的内容**

- 事件快速录入
- 时间线浏览
- 单条事件的简要说明

##### 5.4 记账页

**目标**

- 让支出和收入成为理解现实生活的证据

## 布局规范

### 布局规范

本文档记录本应用的全局布局规则。每条规则按「规则 → 为什么 → 例外」三段式描述,方便后续矫正代码时直接对照。

---

#### 1. 滚动行为

##### 规则

- **整页(`html` / `body`)不出现滚动条**——视口被布局完整占满,不存在页面级滚动。
- **内容区域的外层容器不出现滚动条**——主内容区作为一个整体,不滚动。这包括 app-shell 的 `<main>` 内层容器(`mx-auto w-full max-w-[1500px] ...`)和每个模块页的根 `<div>`。
- **每个模块页都按"固定壳"模式实现**——根容器在 `wide` / `compact` 模式下都是 `flex h-full flex-col min-h-0 overflow-hidden`,内部任何会溢出的子区块自带滚动容器。**不允许**模块页根容器只写 `space-y-*` 把内容堆下去等外层兜底滚动(这正是 §1 第二条要避免的)。
- **内容内的子区块可以独立滚动**——列表、卡片栈、详情面板等子模块按需 `overflow-y-auto`,且每个滚动容器都要在其链路上的所有 flex/grid 祖先设置 `min-h-0`。
- **窄屏(`stacked` 模式)允许整页滚动**——此时回退到正常文档流,模块页根容器写成 `space-y-*`、由 `html` / `body` `overflow: auto` 接管。模块组件通过 `isStackedLayout` 之类的 prop 区分两套根容器写法,而不是依赖隐式断点。

##### 为什么

- **桌面端定位是工作台**:多区域并排展示(导航 / 列表 / 详情),用户视线在各区域之间切换,而不是从上往下浏览长文。整页滚动会让顶部信息(导航、工具栏)失焦,破坏空间记忆。
- **所有模块共享一种节奏**:如果一部分模块是"固定壳",一部分模块是"长页面",用户在模块之间切换时会有忽然变成滚动视图的错位感。统一为固定壳,模块切换才是无缝的横向移动。
- **窄屏放弃 shell**:小屏幕里多区域并排无意义,堆叠为单列后,正常滚动反而比拆分多个滚动容器更符合直觉,也避免嵌套滚动的手势冲突。

##### 例外 / 实现要点

- **canonical 模式参考**:[shopping-page.tsx](../../src/features/bettertolive/ui/shopping/shopping-page.tsx) 已经实现了完整的固定壳:根容器 `flex h-full flex-col gap-3 space-y-0 overflow-hidden`,顶部摘要 `shrink-0`,中段网格 `min-h-0 flex-1 overflow-hidden`,内部列表用独立的 `overflow-y-auto` 容器。新模块页或重构旧模块页时,都以这个文件为模板。
- **app-shell 配合**:[app-shell.tsx](../../src/features/bettertolive/ui/shell/app-shell.tsx) 的内容容器在非 `stacked` 模式下必须是 `min-h-0 flex-1 overflow-hidden`,而不是 `overflow-y-auto`。`overflow-y-auto` 等价于把"内容会很长"当默认,与本规则相悖。
- **必须配套设置 `min-height: 0`**:flex / grid 子项默认 `min-height: auto`,会被内容撑开导致滚动条出现在错误的层级。所有作为"滚动容器父级"的 flex 子项都要显式 `min-height: 0`(grid 同理 `min-block-size: 0`)。
- **`<m.div>` / motion 包装**:页面切换动画的包装容器在固定壳下应写成 `h-full`,而不是 `min-h-full`。`min-h-full` 会让内容超出时把外层撑高,触发外层滚动。
- **窄屏断点要同时解除两处**:不仅要切布局,还要把 `html, body { overflow: hidden }` 改回 `overflow: auto`,否则窄屏下内容被裁掉。
- **模态层 / Toast 不受此规则约束**:它们脱离正常文档流,自身管理滚动(如 `Dialog` 内部的长表单)。
- **滚动条样式统一**:子区块出现滚动条时使用统一的细滚动条样式(具体值后续在「滚动条样式」章节补充)。

#### 2. 响应式断点

##### 规则

应用使用三档布局模式,断点定义在 [src/features/bettertolive/ui/shell/app-shell.tsx](../../src/features/bettertolive/ui/shell/app-shell.tsx):

| 模式 | 视口宽度 | 形态 | 间距风格 |
|---|---|---|---|
| `wide` | ≥ 1240px | 侧栏 + 主内容并排,侧栏可展开 | 更紧凑(`px-6 py-3` / `py-2.5`) |
| `compact` | ≥ 960px 且 < 1240px | 侧栏 + 主内容并排,侧栏默认折叠 | 标准(`px-4 py-4`) |
| `stacked` | < 960px | 单列堆叠,整页可滚动 | 标准(`px-4 py-4`) |

##### 为什么

- 1240 / 960 两个边界已经在 `app-shell.tsx` 里写死并被布局模式逻辑依赖,新组件不能各自再设一套断点。
- `wide` 模式刻意把间距收紧(`py-2.5`、`h-8` 输入框)——大屏不需要更松,需要更密的信息密度。
- `stacked` 模式同时切换:布局形态(单列)+ 滚动模式(整页滚动)+ 头部样式(竖排)。不要只切其中一项。

##### 例外 / 实现要点

- 组件内部如果要响应布局模式,优先读父层传下的 `isWideLayout` / `isStackedLayout` 之类的 props,而不是再用 `useWorkspaceLayoutMode()`,避免布局判断分散。
- 不引入新的中间断点。需要更细粒度时,用 props 控制密度(参考 `isWideLayout` 模式)。

#### 3. 间距尺度

##### 规则

只在以下尺度内选择,不引入项目内未出现的值:

| 用途 | 主选 | 备选 |
|---|---|---|
| `gap` 行内紧凑(图标+文字、徽章组) | `gap-2` | `gap-1`(更紧) |
| `gap` 卡片内部分组 | `gap-3` | — |
| `gap` 区块之间 | `gap-4` | — |
| 内边距 容器/卡片 | `p-5` | `p-4` |
| 内边距 横向(header / 主内容) | `px-4` | `px-6`(`wide` 模式) |
| 内边距 纵向(header / 主内容) | `py-4` | `py-3`(密集)/ `py-2.5`(`wide` 模式) |

##### 为什么

实际统计:`gap-2`(62 次)、`gap-3`(35)、`gap-4`(35);`px-4`(53)、`py-4`(34)、`p-5`(42)、`py-3`(26)、`px-3`(21)。这套尺度已经是事实标准,继续用它意味着新代码自动一致。

##### 例外 / 实现要点

- 不要用 `gap-5`、`p-3`、`py-8` 这类只出现 1–2 次的零散值——它们是历史残留,新代码不要扩大它们的使用面。
- 紧贴 wrap 元素(`flex-wrap`)时常需要 `gap-3` 而非 `gap-2`,避免折行后挤在一起。
- 长文本区域(段落、列表项)用 `py-1` / `py-2` 控制行间,不与卡片内边距尺度混。

#### 4. 圆角

##### 规则

| 场景 | 类 |
|---|---|
| 默认控件、按钮、卡片、容器 | `rounded-lg` |
| 大型容器、面板 | `rounded-xl` 或 `rounded-2xl` |
| 头像、徽章、纯图标按钮、圆形指示点 | `rounded-full` |

`rounded-lg` 锚定 `--radius: 0.625rem`(10px),其余级别按比例衍生(见 [globals.css:70](../../src/styles/globals.css:70))。

##### 为什么

统计:`rounded-lg` 64 次、`rounded-full` 34 次、`rounded-xl` 11、`rounded-2xl` 4、`rounded-md` 2。`rounded-lg` 是事实上的"默认圆角",`rounded-md` 几乎没人用,新代码不要再选它。

##### 例外 / 实现要点

- 同一卡片里嵌套元素圆角不要超过父级。
- 需要"近乎方形"的内嵌格子时用 `rounded-sm`,不要自己写 `rounded-[2px]`。

#### 5. 容器宽度

##### 规则

| 用途 | 值 |
|---|---|
| 主内容区最大宽度 | `max-w-[1500px]`,配合 `mx-auto` |
| 侧栏展开宽 | `w-[292px]` |
| 侧栏折叠宽 | `w-[92px]` |
| 顶部搜索框最大宽 | `max-w-[400px]` |

##### 为什么

`1500px` 是已在 4 处使用的主内容容器上限。超出会让多列布局看起来太空,信息密度被稀释。`292 / 92` 是 app-shell 的固定值,改动会牵连 motion `layout` 动画。

##### 例外 / 实现要点

- 新增主内容容器一律 `mx-auto w-full max-w-[1500px]`。不要在内部子组件再设一次 `max-w-[1500px]`(已经被父层限制了)。
- 详情面板、对话框有自己的宽度档(后续追加),不复用主内容的 1500。

#### 6. 层级(z-index)

##### 规则

所有 z-index 走 [src/lib/ui-layers.ts](../../src/lib/ui-layers.ts) 的 `UI_LAYERS` 常量,不写裸 `z-*`:

| 层 | 值 | 用途 |
|---|---|---|
| `header` | 20 | 顶部栏 / 主内容内的固定 header |
| `utilityPanel` | 90 | 右上工具面板浮层 |
| `notifications` | 100 | 通知列表 / Toast |
| `dialogOverlay` | 140 | 对话框遮罩 |
| `dialogContent` | 150 | 对话框内容 |

##### 为什么

层级集中管理后,出现"被遮住""挡住下拉"等问题时,只需调常量,而不用全局搜 `z-50` / `z-99`。

##### 例外 / 实现要点

- 需要新增层级时(例如新增"全屏临时视图"),先在 `UI_LAYERS` 加常量,再使用。
- Tailwind 的 `z-10` / `z-20` 之类的预设别再用——已经用裸数字 `z-[20]` 把档位定死。

#### 7. 颜色与字体令牌

##### 规则

- **颜色一律读 CSS 变量**:`[color:var(--text-primary)]`、`bg-[color:var(--chip-bg)]`、`border-[color:var(--surface-border)]`。变量在 [src/styles/globals.css](../../src/styles/globals.css) 与 workspace 主题文件中定义。
- **字体一律走 `--font-sans`**(Geist Variable),通过 `@layer base` 已注入 `html`,不需要每个组件再 `font-sans`。
- **字号锚点**:正文 `text-sm`,辅助说明 `text-xs`,卡片标题 `text-[1.05rem]` ~ `text-[1.15rem]`,页面主标题 `text-[1.35rem]`(`wide` 模式下收到 `text-[1.2rem]`)。

##### 为什么

变量是主题切换的唯一入口(`useWorkspaceTheme`)。一旦在组件里写死 `text-neutral-700` 这种,主题切换就会失效,而 bug 表象是"某些区域不跟着换主题"——很难定位。

##### 例外 / 实现要点

- 用 shadcn 组件时其内部已用 `bg-background` / `text-foreground` 等语义令牌,不要再额外覆盖颜色。
- 业务页面用 workspace 主题变量(`--text-primary` 等),shadcn 内部用 shadcn 令牌(`--foreground` 等),两套不要混。
- 字距类 `tracking-[0.22em]`(标签小字)、`tracking-tight`(标题)——除此之外不引入新字距。

#### 8. 模块页结构骨架

##### 规则

每个模块页(`overview` / `reflection` / `nutrition` / ... )都按下面这一套骨架实现,只有内部子区块的分栏可以差异化:

```tsx
// canonical 模块页骨架,对照 shopping-page.tsx
export function ModulePage({ isWideLayout, /* data... */ }) {
  return (
    <div
      className={cn(
        "space-y-5",                                       // stacked 模式:正常文档流
        isWideLayout && "flex h-full flex-col gap-3 space-y-0 overflow-hidden",
        // compact 模式介于两者之间;如果模块在 compact 也想固定壳,沿用上面这一行
      )}
    >
      <PageIntro ... />                                    // 永远在最上,占自然高度

      <div
        className={cn(
          "grid gap-4 min-[960px]:grid-cols-3",             // 顶部摘要 SummarySurface 阵列
          isWideLayout && "shrink-0 gap-3",                 // 固定壳里要 shrink-0
        )}
      >
        <SummarySurface ... />
      </div>

      <div
        className={cn(
          "grid gap-4 min-[1240px]:grid-cols-[minmax(0,1.2fr)_minmax(320px,0.88fr)]",
          isWideLayout && "min-h-0 flex-1 gap-3 overflow-hidden",
        )}
      >
        {/* 这里的每一列再各自 overflow-y-auto,详见下条 */}
      </div>
    </div>
  )
}
```

每一列(`Surface` 等卡片栈)在固定壳模式下要包一层独立的滚动容器:

```tsx
<Surface className={cn("p-5", isWideLayout && "flex h-full min-h-0 flex-col")}>
  <SectionHeading ... />                                   // shrink-0
  <div
    className={cn(
      "mt-5 space-y-4",
      isWideLayout && "min-h-0 flex-1 overflow-y-auto pr-1",
    )}
  >
    {entries.map(...)}
  </div>
</Surface>
```

##### 为什么

- **一套骨架,横向无缝**:模块切换时如果一个页面是固定壳、下一个是长滚动页,用户会瞬间失去空间方位感。统一骨架后,模块之间就是横向移动,而不是切换到新形态。
- **"哪里出滚动条"是设计决策**:固定壳让"哪一列长、哪一列要滚"成为可控的设计判断,而不是"内容多就自动撑长页面"的隐式后果。
- **`isWideLayout` 作为单一切换点**:只在根容器和容器边界使用 `isWideLayout` 切换形态,内部细节不重复判断,避免布局逻辑散落。

##### 例外 / 实现要点

- **`PageIntro` 不占行高时**:`PageIntro` 在没有 `searchQuery` 时返回 `null`(参考 [shared.tsx:36](../../src/features/bettertolive/ui/shared/shared.tsx:36)),不会破坏 `flex-1` 中段的高度计算,不需要额外处理。
- **顶部摘要可缺省**:不是所有模块都有摘要阵列。没有摘要时,中段网格直接成为 `flex-1 min-h-0`,不要塞占位卡片。
- **滚动容器只能出现在叶子层**:不要在中段网格上写 `overflow-y-auto`——那会让两列同时滚动,空间感塌掉。只让每一列内部的列表 `overflow-y-auto`。
- **`pr-1` / `pr-0.5` 给滚动条留位**:子区块开启 `overflow-y-auto` 后,加一点右内边距让滚动条不贴边,这点参考 [sidebar-navigation.tsx](../../src/features/bettertolive/ui/shell/sidebar-navigation.tsx) 的 `pr-0.5`。
- **`stacked` 模式不走这套**:窄屏直接 `space-y-5`,所有 `flex h-full overflow-hidden` / `min-h-0 flex-1 overflow-y-auto` 都不加。判断逻辑挂在 `isWideLayout`(或 `!isStackedLayout`)props 上,不在组件内部再调 `useWorkspaceLayoutMode()`。
- **数据非常少的模块**:即使现在内容很短也按固定壳实现——内容会随使用增长,等"开始溢出再改"成本远高于一开始就按规则写。
